# 电商订单系统

> 一句话总结：订单系统 = 正向流程（下单-支付-履约）+ 逆向流程（取消-售后）+ 严格状态机 + TCC 分布式事务，四者环环相扣。

---

## 1. 核心三大流程

### 1.1 正向流程

下单 → 支付 → 履约（商家接单 → 物流 → 发货 → 签收 → 完成）

**下单环节关键点**：
- 商品校验：调用商品服务查在售/规格/库存
- 价格计算：`基础价 + 活动价 - 优惠券 - 积分抵扣`
- 库存锁定：预扣库存 + 15-30 分钟支付有效期，超时自动释放

**支付回调丢失方案**：
- 主动查询 + 重试：每分钟查未支付订单，持续 30 分钟

### 1.2 逆向流程

| 流程 | 触发场景 | 关键动作 |
|---|---|---|
| 用户主动取消 | 待支付状态 | 释放预扣库存 + 改状态为"已取消" |
| 超时自动取消 | 定时任务扫描 | 批量释放库存 + 状态变更 |
| 售后（退货/退款/换货） | 支付后 | 商家审核（24h）→ 寄回 → 验货 → 退款 → 状态更新 |

### 1.3 状态机（8 大状态）

`待支付 → 已支付 → 待发货 → 已发货 → 已完成`，支线 `已取消 / 售后中 / 售后完成`。

**关键流转规则**：
- 待支付 **只能** 转已支付/已取消（不能直接跳已发货）
- 已发货 **只能** 转已完成/售后中（不能回退待发货）

> **小结**：
> - 正向链路必须闭环到「用户收货」
> - 逆向流程两大核心：释放资源 + 状态可追溯
> - 状态机是订单系统的"宪法"，杜绝任意跳转

## 2. 核心功能实现

### 2.1 订单创建：三大难点

**难点 1：库存超卖**
- 日常销售：Redis `DECR` 原子扣减 + 异步同步 MySQL；Redis 故障 → 回退 DB 行锁
- 秒杀活动：MQ 削峰 + 顺序消费；活动前 10 分钟预热库存到本地缓存
- 极致并发：Redisson 分布式锁，按 productId 加锁

**难点 2：重复下单三层防护**

| 层级 | 方案 | 效果 |
|---|---|---|
| 前端 | 按钮置灰 3s + UUID 请求标识 | 拦截 80% 重复点击 |
| 后端 | `userId+productId+timestamp` 防重键 + Redis SETNX | 拦截 95% 重试 |
| 数据库 | `(user_id, product_id, created_at)` 联合唯一索引 | 兜底 100% 拦截 |

**难点 3：多服务协调一致性 → TCC**
- 每个服务三步：Try（预扣）→ Confirm（确认）/ Cancel（回滚）
- 全 Try 成功 → 统一 Confirm；任一失败 → 全量 Cancel

### 2.2 支付功能：状态机 + 本地消息表

**5 步处理支付回调**：
1. Redis 唯一性校验（订单号+流水号）
2. 状态机判断（待支付 → 已支付/已取消）
3. 更新订单状态 + 写本地消息表
4. 发 RocketMQ 通知（库存扣减、积分增加）
5. 5min 定时任务重试未发出的消息

**对账兜底**：每天凌晨对账订单表 vs 支付表，补偿状态不一致。

### 2.3 订单查询优化

- 缓存热点：用户最近 30 天订单 → Redis Key `user:{userId}:orders`
- 精准路由：按 userId 哈希分库
- 复杂查询 → 只读库

### 2.4 订单取消：延迟消息 + 定时校验双保险

```
下单成功 → 发延迟消息(超时时间) → 到期后查状态
   ↓                                    ↓
未支付 → 关闭订单 + 释放库存       已支付 → 忽略消息
                                       ↓
              定时任务每 10min 兜底扫描待支付超时订单
```

> **小结**：
> - 超卖防护分场景适配：日常/秒杀/极致并发
> - 重复下单 = 前端 UX + 后端幂等 + DB 唯一索引 三层
> - 支付一致性 = 状态机 + 本地消息表 + 异步对账

## 3. 系统架构：三层解耦

| 层级 | 组件 | 职责 |
|---|---|---|
| 接入层 | Nginx + 限流熔断 | 流量管控、负载均衡 |
| 应用层 | 微服务（订单/支付/库存/物流/用户） | 业务逻辑 |
| 数据层 | MySQL + Redis + RocketMQ + ES | 多种存储协同 |

### 3.1 接入层关键配置

- **限流**：令牌桶算法，下单 10000 QPS、查询 50000 QPS
- **熔断**：支付服务异常率 > 5% 立即熔断
- **防刷**：同 IP 1 min 内 20 单拉黑

### 3.2 应用层服务拆分

| 服务 | 职责 | 同步调用 | 异步消息 |
|---|---|---|---|
| 订单服务 | 创建/状态/查询 | Dubbo → 库存 | 发"支付完成"事件 |
| 支付服务 | 第三方对接、退款 | — | 发"退款成功"事件 |
| 库存服务 | 锁定/释放/扣减 | — | — |
| 物流服务 | 运单/轨迹 | — | 发"已签收"事件 |

### 3.3 数据层选型

| 存储 | 用途 | 选型理由 |
|---|---|---|
| MySQL | 订单核心数据 | 事务 + 复杂查询 + 分库分表 |
| Redis | 热点订单/幂等键/分布式锁 | 高并发 |
| RocketMQ | 异步事件 | 解耦 |
| Elasticsearch | 复杂多维查询 | 搜索快 |

> **小结**：
> - 三层各司其职：接入层保护、应用层解耦、数据层分工
> - 同步用 Dubbo、异步用 RocketMQ，不混用
> - 数据层不把所有鸡蛋放一个篮子，按特性选存储

## 4. 方案对比表

| 库存控制方案 | 性能 | 一致性 | 业务侵入 | 适用 |
|---|---|---|---|---|
| DB 行锁 | 低 | 强 | 低 | 早期小规模 |
| Redis 预扣 + 异步同步 | 极高 | 最终一致 | 中 | 日常销售 |
| MQ 削峰 + 顺序消费 | 高 | 最终一致 | 中 | 促销 |
| Redisson 分布式锁 | 中 | 强 | 中 | 极致并发 |
| TCC 事务 | 中 | 强 | 高 | 金融级 |

## 5. 复习清单

1. **订单系统三大核心逻辑？** 正向流程、逆向流程、状态流转规则。
2. **订单 8 大状态及关键约束？** 待支付/已支付/待发货/已发货/已完成/已取消/售后中/售后完成；待支付不能直跳已发货。
3. **防超卖分场景方案？** 日常用 Redis DECR、秒杀用 MQ 削峰、极致并发用 Redisson 锁。
4. **重复下单三层防护？** 前端 UUID + 后端 Redis 防重键 + DB 联合唯一索引。
5. **TCC 模式三步？** Try（预扣）→ Confirm（确认）/ Cancel（回滚）；全 Try 成功才 Confirm。
6. **支付回调丢失怎么办？** 主动查询 + 重试（1min 间隔，持续 30min）。
7. **支付状态一致性方案？** 状态机 + 本地消息表 + 异步对账。
8. **订单取消双保险？** 延迟消息 + 定时校验（每 10min 兜底扫描）。
9. **Redis 幂等键设计？** Key = `userId+productId+timestamp`，5min 过期。
10. **同步 vs 异步通信选择？** 实时性高用 Dubbo 同步（如下单扣库存），非实时用 RocketMQ 异步（如支付完成通知）。
11. **为什么用 Elasticsearch？** 复杂多维订单查询（时间/金额/状态），比 DB 快。
12. **售后流程关键点？** 商家 24h 审核 → 寄回 → 验货 → 退款 → 状态更新；退款接口失败要重试 + 日志 + 人工兜底。
