# 微服务高可用保障

> 一句话总结：微服务高可用 = **四大原则**（容错、隔离、快速响应、规范化变更）+ **五大工程手段**（限流 / 熔断 / 降级 / 超时 / 异步解耦）+ **全链路故事线**（发现 → 规划 → 落地 → 效果 → 改进）。

---

## 1. 高可用的四大核心原则

| 原则 | 核心思想 | 工程动作 |
|---|---|---|
| 容错设计 | 接受失败是常态 | 多副本、降级兜底、最坏打算 |
| 故障域隔离 | 缩小爆炸半径 | 微服务拆分、核心/非核心拆开 |
| 快速响应 | 缩短 MTTR | 监控告警 + 自动化故障处理 |
| 规范化变更 | 变更 = 80% 故障源 | 严格 Review + 回滚预案 + 发布 Checklist |

> **小结**：
> - 四个原则不是并列，是**递进关系**
> - 容错 → 隔离 → 响应 → 变更，从设计到流程全覆盖
> - 面试时**先讲原则再讲方案**，显得有体系

---

## 2. 原则 1：容错设计 —— 优雅地「凑合」

**核心理念**：无论发生什么，**整体仍能提供服务**，哪怕有损。

> 像自行车掉了一个轮子，优秀骑手会骑独轮车，而不是直接瘫倒。

**故障源**：

| 故障源 | 容错思路 |
|---|---|
| 服务自身 bug | 限流、熔断、超时、降级 |
| 依赖服务 | 高可用部署方案 + 兜底容错 |
| 软件基础设施 | Redis Cluster / 多可用区 MQ |
| 硬件 / 第三方 | 多机房、容灾（出现频率低） |

**兜底示例**：Redis 集群整体崩溃 → 限流保护 DB → 核心功能可用。

> **小结**：
> - 容错 = 接受不完美，**但必须有兜底**
> - 容错的成本 = 多写一份「降级逻辑」+ 多备一份资源
> - 面试关键词：「限流保护 DB」「返回兜底文案」

---

## 3. 原则 2：故障域隔离 —— 控制爆炸半径

**三维度评估影响范围**：

1. 业务损失大小
2. 受影响用户数量
3. 波及的关联组件数量

**隔离策略**：

| 维度 | 手段 |
|---|---|
| 业务边界 | 微服务拆分、领域划分 |
| 模块边界 | 核心 / 非核心拆开 |
| 基础设施 | 数据库 / 缓存 / MQ 拆分 |

**现实难题**：
- 服务间强依赖：业务复杂度决定无法完全解耦
- 共享基础设施：成本问题导致隔离不彻底

> **小结**：
> - 隔离的核心 = 「**别让一颗老鼠屎坏了一锅汤**」
> - 共享 DB / Redis 集群 = 隔离的天敌
> - 核心业务**必须独立基础设施**

---

## 4. 原则 3：快速响应 —— 检测与恢复的速度之战

| 子能力 | 关键点 |
|---|---|
| 快速发现 | 全方位 Observability + 精准分级告警 |
| 快速恢复 | 24h On-Call + 自动化故障处理 + 完善 Runbook |

**没有告警的观测 = 没有灵魂的观测**。

**MTTR（Mean Time To Recover）** 提升的真正瓶颈是**组织能力**：
- 值班人员不可能熟悉所有服务
- 必须依赖**自动化**和**可恢复 Runbook**

> **小结**：
> - 监控要全，告警要**精准分级**
> - 不能依赖个人能力，必须**自动化**
> - 三个 9 之后想冲四个 9 = 拼自动化水平

---

## 5. 原则 4：规范化变更 —— 敬畏每一次上线

> 统计：线上 80%+ 故障由**变更**引起。

**关键动作**：
- 任何变更必须 Review
- 必须有回滚预案
- 上线 Checklist（功能、配置、回滚、监控）
- 重要时期「封网」（双 11 不允许发布）

> **小结**：
> - 变更 = 最大风险源
> - 流程 = 把人为错误降到最低
> - 「封网」是工程界最朴素也最有效的一招

---

## 6. 面试故事线五步法

| 步骤 | 关键词 | 面试表达 |
|---|---|---|
| 1. 发现问题 | 过去时、痛点 | 列举真实故障 + 三个根源（监控 / 治理 / 流程） |
| 2. 规划方案 | 蓝图、阶段化 | 五步路线图，体现系统性 |
| 3. 落地实施 | 进行时、细节 | 真实遇到的阻力 + 技术细节 |
| 4. 取得效果 | 量化、对比 | 可用性从 2 个 9 → 3 个 9，附数据 |
| 5. 持续改进 | 未来时、视野 | 历史遗留问题 + 未来规划（可以大胆讲） |

**规划方案五步**：

```text
1. 补齐短板 ─► 全方位监控告警
2. 构建防线 ─► 服务治理（限流/熔断/降级/超时/隔离）
3. 加固地基 ─► 第三方依赖高可用（Redis Cluster、MHA 等）
4. 核心隔离 ─► 核心/非核心依赖拆分
5. 规范流程 ─► 上线 Checklist + Code Review + 回滚预案
```

> **小结**：
> - 故事线 = 面试时**主动引导节奏**的武器
> - 「发现问题→取得效果」前后对比，凸显个人价值
> - 真实阻力（资源不足、执行不彻底）**反而增加可信度**

## 7. 服务治理五件套

| 手段 | 作用 | 触发场景 |
|---|---|---|
| 限流 | 拒绝超额请求，保护系统 | 秒杀、爬虫、突发流量 |
| 熔断 | 下游异常时快速失败 | 依赖服务超时率高 |
| 降级 | 关闭非核心，返回兜底 | 大促 / 故障期 |
| 超时控制 | 防止线程被长时间占用 | 所有远程调用 |
| 线程池隔离 | 故障不传染 | 核心 / 非核心业务分池 |

```java
// Sentinel 限流示例
@SentinelResource(value = "createOrder", blockHandler = "blockHandler")
public Order createOrder(OrderDTO dto) {
    // 业务逻辑
}

public Order blockHandler(OrderDTO dto, BlockException e) {
    return new Order("系统繁忙，请稍后再试");
}
```

> **小结**：
> - 限流 / 熔断 / 降级常组合使用
> - 超时 = **最容易忽略的救命手段**（不设超时 = 一旦下游慢就拖死上游）
> - 线程池隔离 = **不把鸡蛋放一个篮子**

## 8. 亮点方案 1：异步化 / 解耦

**核心思想**：将流程拆为「关键路径（同步）」+「非关键路径（异步）」。

```text
原流程（同步）：
扣库存 → 创建订单 → 调用支付 → 发邮件 → 加积分

优化后（异步）：
扣库存 → 创建订单 → 调用支付  （核心同步）
                ↓ MQ
        发邮件 + 加积分  （异步消费）
```

**口诀**：**多做多错，少做少错，不做不错**。

> **小结**：
> - 主流程越短，可用性越高
> - 异步靠 MQ 的**重试 + 死信**保证最终一致
> - 和产品经理一起梳理流程 = 改造能否落地的关键

---

## 9. 亮点方案 2：自动化故障处理

**为什么必要**：人不可能 24h 紧绷，也不可能熟悉所有细节。

**典型方案**：

| 方案 | 说明 |
|---|---|
| 自动扩容 | CPU + 内存双指标触发，新增节点 |
| 自动修复数据 | 定时任务对比数据，不一致则触发修复 |
| 自动补发消息 | 扫描卡住流程，触发消息补发 |
| 故障自愈 | 检测异常 → 预设脚本 → 恢复 |

```text
集群监控 ─► 持续 N 分钟 CPU > 90%
          ─► 触发扩容
          ─► 新节点注册到 LB
          ─► 负载降至安全水位
```

> **小结**：
> - 三个 9 → 四个 9 的分水岭 = **自动化程度**
> - 自动扩容的常见指标：CPU + 内存 + QPS + RT
> - 设计触发条件时**强调「所有节点都达到」**，避免负载不均误扩容

## 10. 限流常见算法对比

| 算法 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| 固定窗口 | 单位时间计数 | 简单 | 突刺问题 |
| 滑动窗口 | 细分时间片加权 | 平滑 | 实现稍复杂 |
| 漏桶 | 恒定速率流出 | 流量整形 | 不能应对突发 |
| 令牌桶 | 恒定速率放令牌 | 支持突发 | 实现略复杂 |

```java
// 令牌桶（Guava RateLimiter）
RateLimiter limiter = RateLimiter.create(100); // 100 QPS
if (limiter.tryAcquire(100, TimeUnit.MILLISECONDS)) {
    // 放行
} else {
    // 拒绝 / 降级
}
```

> **小结**：
> - 令牌桶 = 实际生产最常用（支持突发）
> - 漏桶 = 适合流量整形
> - 滑动窗口 = 适合精细限流（Sentinel 默认）
