# 高可用评论系统

> 一句话总结：评论系统 = 读多写少（20:1）+ 脉冲式流量 + MySQL+Redis 混合存储 + 多级缓存 + 分布式微服务架构，扛百万级实时互动。

---

## 1. 灵魂三问：先懂业务再选技术

### 1.1 三类角色诉求

| 角色 | 核心诉求 | 技术优先级 |
|---|---|---|
| 用户 | 顺畅发、痛快看（情绪出口+决策工具） | 体验 > 完美 |
| 产品 | 真实反馈收集（比问卷真实 10 倍） | UGC 内容护城河 |
| 技术 | 流量放大器（地铁站散场式爆发） | 峰值配资源 |

### 1.2 流量特性拆解

- **读写比 20:1**：10 万人发评论 → 200 万人刷评论
- **脉冲式爆发**：明星官宣 1 小时 = 平时 10 天流量
- **用户容忍度**：点赞 +1 慢 3 秒无人在意；评论加载失败 3 次 → 80% 用户划走

> **小结**：
> - 灵魂三问：用户要什么、产品要什么、技术扛什么
> - 读多写少 → 缓存优先
> - 脉冲流量 → 按峰值配资源、按低谷用资源

## 2. 存储方案选型

### 2.1 三种存储对比

| 方案 | 适用场景 | 优点 | 缺点 |
|---|---|---|---|
| 纯 MySQL | 事务强、嵌套 ≤ 3 层 | ACID、索引成熟 | 单表千万慢、分库分表复杂 |
| 纯 MongoDB | 非结构化、嵌套深 | 原生 JSON、写入快 30% | 嵌套查询慢、事务弱 |
| **MySQL + Redis** | **多数业务** | **MySQL 保可靠 + Redis 扛读** | **双写一致性** |

**推荐 MySQL+Redis** 原因：
- MySQL = 数据保险柜（事务 + 嵌套查询）
- Redis = ZSet 列表 + Hash 详情 → 读快几十倍
- 双写方案：先写 MySQL → 删 Redis 缓存（Cache-Aside）

### 2.2 多级缓存策略

| 层级 | 容量 | TTL | 场景 |
|---|---|---|---|
| Caffeine 本地缓存 | 10 万条 | 1h | 爆款前 100 条评论（5min 内查 ≥1000 次） |
| Redis 集群 | 16 分片 | — | 次热点 |
| MySQL | 全量 | — | 兜底 |

**一致性策略**：
- MySQL → 删 Redis（不更新）
- 本地缓存：定时校验（10min）+ 主动失效
- 优先保证最终一致，不追求实时同步

> **小结**：
> - 多数业务 MySQL+Redis 已足够，不必上 MongoDB
> - 多级缓存层层减压，本地 > Redis > MySQL
> - 抓大放小：最终一致 > 实时一致

## 3. 分库分表策略

| 策略 | 路由公式 | 适用 | 问题 |
|---|---|---|---|
| 按内容 ID 哈希 | `contentId % 32` | 热点分散 | 爆款集中单表 |
| 按时间 + ID 复合 | `(contentId%8) + (month%4)` | 时间衰减（新闻/短视频） | 跨月查询聚合 |
| 热点表单独拆分 | 监控触发迁移 | 极端热点（明星/事件） | 运维复杂 |

**无感知迁移 4 阶段**：
1. 全量同步：旧表 → 新分表
2. 双写：应用同时写旧表和新表
3. 校验切换：差异率 < 0.1% 切读流量
4. 停写旧表：观察 1 周稳定后停旧表

> **小结**：
> - 分表核心难题不是选策略，而是无感知迁移
> - 双写 + 校验 + 观察 = 零停机切换
> - 热点表要预留（10-20 张）应对突发

## 4. 架构演进三阶段

| 阶段 | 架构 | 适用 | 问题 |
|---|---|---|---|
| 1.0 青铜 | 单应用 + 单 MySQL + Caffeine | 日活 < 10 万 | 单点故障、一锅端 |
| 2.0 白银 | 垂直拆分（评论/互动分离） | 日活 10-80 万 | 跨服务延迟、代码冗余 |
| **3.0 黄金** | **分布式分层防御** | **百万级** | **复杂度高** |

### 4.1 3.0 黄金架构：4 层协同

**接入层**：
- LVS（4 层 TCP 分发）→ Nginx（7 层路由）→ 自研网关（鉴权/限流/监控）
- 网关鉴权拦 90% 恶意请求；限流 1min/5 条

**应用层 4 个微服务**：

| 服务 | 核心诉求 | 关键设计 |
|---|---|---|
| 评论发布 | 数据不丢 | 双写 + 日志 + Kafka 事件 |
| 评论查询 | 查询快 | 三级缓存 + 缓存预热 + 自动降级 |
| 互动（点赞/回复） | 扛高并发 | 本地计数器 + 10s 批量同步 |
| 内容审核 | 准确高效 | 敏感词库 + 百度 AI + Kafka 通知 |

**数据层**：
- Redis Cluster（16 分片 × 3 主 3 从 = 48 节点 × 8G 内存）
  - ZSet 存评论列表、Hash 存详情、String 存点赞数
  - 嵌套回复用"父 ID + 层级路径"（如 `1001.2003.3005`）
- MySQL 分库分表：`content_id%8 × user_id%4 = 32 张表` + 20 张热点表
- 冷数据：3 个月前 → TiDB，Flink 实时同步
- Kafka 削峰：10 万写 QPS → 2 万/秒匀速

**完整评论链路（延迟 < 500ms）**：
```
LVS → Nginx → 网关 → 发布服务 → 审核服务 → MySQL + Kafka
                                          ↓
                          查询服务消费 Kafka → 同步 Redis
                                          ↓
                          用户查询：Caffeine → Redis → MySQL
```

> **小结**：
> - 架构随业务演进，没有一蹴而就
> - 接入层 LVS+Nginx+网关三层过滤
> - 4 微服务各司其职：发布/查询/互动/审核
> - 热点表 + 冷热分层是数据层关键

## 5. 故障恢复：三级防线

| 故障类型 | 方案 | 效果 |
|---|---|---|
| 缓存雪崩 | Caffeine 兜底 + 过期时间 ±10% 随机 | Redis 挂了用户无感 |
| MySQL 故障 | 1 主 2 从 MGR 集群 + 30s 自动切换 | 比传统快 10 倍 |
| Redis 故障 | 3 主 3 从 + 哨兵 5s 升主 | 单主故障 5s 恢复 |
| 机房故障 | 北京+上海双活，异地同步 | 断电 5min 接管 |

**一键降级三级预案**：

| 等级 | 触发条件 | 动作 |
|---|---|---|
| 一级 | QPS 70% 阈值 | 限频 1min/1 条 |
| 二级 | QPS 90% 阈值 | 关闭图片上传 |
| 三级 | QPS 超阈值 | 仅返回前 100 条 + 提示 |

> **小结**：
> - 高可用 ≠ 不出故障，而是故障时用户无感
> - 多副本 + 快速切换是基础设施保障
> - 降级按压力等级预设，一键触发

## 7. 复习清单

1. **评论系统读写比？** 20:1，10 万人发 → 200 万人看。
2. **为什么 MySQL+Redis 优于纯 MongoDB？** 多数业务要事务+查询性能，MongoDB 事务弱、嵌套查询慢。
3. **MySQL+Redis 双写一致性方案？** 先写 MySQL → 删 Redis 缓存（Cache-Aside）。
4. **多级缓存三层？** Caffeine 本地（10 万/1h）→ Redis 集群（16 分片）→ MySQL 全量。
5. **分表无感知迁移 4 阶段？** 全量同步 → 双写 → 校验切换 → 停写旧表。
6. **1.0/2.0/3.0 架构演进？** 单体 → 垂直拆分 → 分布式分层防御。
7. **3.0 接入层组成？** LVS（4 层）→ Nginx（7 层）→ 自研网关（鉴权/限流/监控）。
8. **4 个微服务职责？** 发布（不丢）、查询（快）、互动（扛并发）、审核（准确）。
9. **嵌套回复存储设计？** "父 ID + 层级路径"（如 `1001.2003.3005`），HMGET 批量拉取。
10. **Redis 集群规模？** 16 分片 × 3 主 3 从 = 48 节点 × 8G 内存。
11. **MySQL 分库分表策略？** `content_id%8 × user_id%4 = 32 张表` + 20 张热点表。
12. **故障恢复核心思想？** 不求"零故障"，但求"故障时用户没感觉"。
