# 系统高可用架构

> 一句话总结：高可用 = **冗余 + 自动故障转移**；架构每一层都要做，单点用仲裁者（Keepalived / ZooKeeper / 哨兵）兜底，辅以隔离、限流、熔断、降级、监控告警。

---

## 1. 高可用的核心思想

| 概念 | 说明 |
|---|---|
| 冗余 | 多副本部署，避免单点 |
| 自动故障转移 | 故障时无需人工切换 |
| 仲裁者 | 第三方判定 + 协调（Keepalived、ZK、哨兵） |

**为什么是冗余 + 自动转移？**
- 假设单机可用性 90%，两机集群 = `1 - 0.1×0.1 = 99%`
- 多副本 + 自动切换 = **近实时**故障转移 = 高可用

### 可用性等级（几个九）

| 级别 | 可用率 | 年宕机 | 行业典型 |
|---|---|---|---|
| 不可用 | 90% | 36.5 天 | 几乎不可接受 |
| 基本可用 | 99% | 87.6 小时 | 每天宕 14 分钟已严重影响业务 |
| 较高可用 | 99.9% | 8.76 小时 | 较成熟业务系统 |
| 高可用 | 99.99% | 52.56 分钟 | **大厂一般要求** |
| 极高可用 | 99.999% | 5.26 分钟 | 列车调度、支付清算等 |

> **小结**：
> - 高可用不是「不出问题」，是「出问题不感知」
> - 单机 → 集群 → 仲裁者 → 自动化，是高可用演进的标配路径
> - 几个九是结果指标，**冗余 + 自动转移**才是工程手段

## 2. 互联网分层架构

```
┌──────────────────────────────────────────┐
│ 接入层：F5 / LVS                          │
│ 反向代理层：Nginx                          │
│ 网关层：流控 / 风控 / 协议转换              │
│ 站点层 + 基础 service：微服务（Dubbo）      │
│ 存储层：MySQL / Oracle                     │
│ 中间件：ZK / Redis / ES / MQ              │
└──────────────────────────────────────────┘
```

> 高可用 = **每一层都做高可用**，任意一层单点都会拖垮整体。

> **小结**：
> - 自顶向下：LVS → Nginx → 网关 → 服务 → 存储 → 中间件
> - 每一层的关键问题：单点怎么破？数据怎么同步？故障怎么转移？

---

## 3. 接入层 & 反向代理层（LVS + Keepalived）

| 组件 | 角色 |
|---|---|
| LVS | 流量入口，主备模式 |
| Keepalived | 仲裁者，通过心跳检测实现 IP 漂移 |
| VIP | 虚拟 IP，仅在 Master 生效 |

```text
Client ──► VIP (115.204.94.139)
              │
        ┌─────┴─────┐
      Master       Backup  (LVS)
        │            │
   keepalived  ◄──心跳──►  keepalived
        │            │
     Nginx 集群    （检测到故障自动接 VIP）
```

**Keepalived 检测机制**：ICMP / TCP 端口扫描；也能检测 Nginx 后端健康。

> **小结**：
> - 主备 LVS + Keepalived = 接入层标配
> - 故障时 VIP 漂移到备机，业务无感知
> - Keepalived 还能扩展用于 MySQL 主从切换

## 4. 微服务层（以 Dubbo 为例）

```text
Provider ──注册──► Registry (ZK/Nacos)
                       ▲
                       │ 订阅/拉取
                       │
Consumer ──负载均衡──► Provider 列表（剔除不可用）
```

| 关键机制 | 说明 |
|---|---|
| 注册中心 | 服务发现 + 健康检查（心跳） |
| 负载均衡 | Consumer 端从多 Provider 中选一个 |
| 自动剔除 | Provider 不可用时从列表移除 |
| 故障转移 | 失败重试其他 Provider |

> Registry = 微服务世界的「仲裁者」，对应 Keepalived 的角色。

> **小结**：
> - Provider 多副本 = 冗余
> - 注册中心心跳 = 自动故障检测
> - Consumer 重试 + 负载均衡 = 自动故障转移

## 5. 中间件层

### 5.1 ZooKeeper（强一致协调服务）

| 角色 | 职责 |
|---|---|
| Leader | 事务处理 + 写广播 + 集群调度 |
| Follower | 处理读、转发写、参与投票 |
| Observer（3.0+） | 只读扩展节点，不参与投票 |

**Leader 选举**：Follower 收不到 Leader 心跳 → 发起投票 → 用 ZAB 协议选新 Leader。

| 协议 | 用途 |
|---|---|
| ZAB | ZK 专用，崩溃恢复 + 原子广播 |
| Paxos | 通用一致性 |
| Raft | 易理解的分布式共识 |

### 5.2 Redis

| 模式 | 架构 | 优点 | 缺点 |
|---|---|---|---|
| 主从 + 哨兵 | 1 主 N 从 + 哨兵集群 | 读写分离、哨兵仲裁 | 单 Master 写压力 + 单机存储瓶颈 + 同步风暴 |
| Cluster 分片 | 16384 槽 + 多 Master | 写压力分散 + 水平扩展 | 架构复杂 |

**Cluster 槽路由**：

```text
Client ──► Proxy ──► CRC16(key) % 16384 ──► 目标 Master
                                       （Master 又挂了 → 哨兵从 Replica 选新 Master）
```

### 5.3 Elasticsearch

| 概念 | 作用 |
|---|---|
| Primary Shard（P） | 主分片，负责写 |
| Replica Shard（R） | 副本，提供读 + 故障接替 |
| Master Node | 集群元数据管理，通过 Bully 算法选举 |

**写流程**：任一节点接写 → 路由到主分片所在节点 → 主分片写完 → 同步副本分片。

### 5.4 Kafka

- Topic 下分多个 **Partition**，分布式存储
- 每个 Partition 有 Leader + Follower（Follower 冷备）
- Leader 挂了 → 从 Follower 选举新 Leader（**不像 ES 多活**）

> **小结**：
> - ZK = 强一致协调，靠 ZAB 协议做选举
> - Redis 主从有写瓶颈，Cluster 分片解决
> - ES 副本分片多活读写，Kafka Follower 冷备
> - 中间件大同小异：**Leader + 副本 + 选举**

## 6. 存储层（MySQL）

| 模式 | 高可用方案 |
|---|---|
| 主从 | Keepalived + VIP 漂移 |
| 分库分表（多主） | 每个主配多个从 = 集群 |

**主从架构**：

```text
Master (写) ──binlog──► Slave (读)
   ▲
   │  Keepalived 心跳
   ▼
Slave（候选主库）
```

> 工程上 MySQL 的连接地址用 **VIP**，不直接写 Master IP，自动接故障转移。

> **小结**：
> - MySQL 高可用思想与 Redis 中间件同源
> - 分库分表之后还要做主从，目的 = **高可用**，不是性能
> - VIP 漂移是工程标配

## 7. 架构之外的高可用措施

| 措施 | 场景 |
|---|---|
| 隔离 | 秒杀独立部署，避免影响日常交易 |
| 限流 | 入口 / 各层防过载 |
| 熔断 | 下游故障时快速失败 |
| 降级 | 非核心功能暂时关闭 / 返回兜底文案 |
| 监控告警 | CPU / 线程数 / 业务指标分级告警 |
| 全链路压测 | 验证容量 |
| 混沌工程 | 主动注入故障演练 |
| 优雅停机 / 平滑发布 | 发布期可用性 |
| 异地多活 / 容灾 | 机房级故障应对 |
| 限权 + 审计 | 防删库跑路、防越权 |

**降级示例**：
- 第三方资金方异常 → 返回「资金方升级中」兜底文案
- 直播卡顿 → 自动降码率
- 双 11 高峰 → 临时关闭注册 / 登录

> **小结**：
> - 架构高可用只是第一步，**生产上还有大量突发情况**
> - 隔离 + 限流 + 熔断 + 降级 = 业务层高可用四件套
> - 监控告警 + 全链路压测 = 事前防御
> - 一键回滚 + 发布记录 = 事后止血

## 8. 关键组件高可用手段速查

| 层 | 组件 | 高可用手段 |
|---|---|---|
| 接入 | LVS | Keepalived 主备 + VIP 漂移 |
| 反向代理 | Nginx | Keepalived + 后端健康检查 |
| 微服务 | Dubbo | 注册中心 + 负载均衡 + 失败重试 |
| 协调 | ZK | ZAB 协议 + Leader 选举 |
| 缓存 | Redis | 主从 + 哨兵 / Cluster 分片 + Raft |
| 搜索 | ES | 多分片 + 副本 + Bully 选举 |
| 消息 | Kafka | Partition + Follower 选举 |
| 存储 | MySQL | 主从 + Keepalived / MHA |

> **小结**：
> - 高可用只有两种思路：**冗余 + 仲裁者**
> - 不同组件协议不同（ZAB/Raft/Bully），但套路一致

## 9. 复习清单

1. **高可用的两大核心思想？** 冗余 + 自动故障转移。
2. **单点 90% 可用性为什么两机就能到 99%？** 1 - 0.1×0.1 = 0.99。
3. **几个九代表什么？** 年可用率指标，3 个 9 = 99.9%。
4. **LVS 高可用怎么做？** Keepalived 主备 + VIP 漂移。
5. **VIP 漂移是什么？** Master 宕机后，VIP 自动漂移到 Backup，业务无感知。
6. **微服务怎么实现自动故障转移？** 注册中心心跳剔除 + Consumer 端负载均衡重试。
7. **ZK Leader 选举用什么协议？** ZAB 协议。
8. **Redis 主从的三个问题？** 单 Master 写瓶颈、单 Master 存储瓶颈、同步风暴。
9. **Redis Cluster 槽数量？** 16384。
10. **ES 主分片和副本分片区别？** 主分片写、副本分片同步 + 提供读 + 故障接替。
11. **Kafka 与 ES 数据分片的区别？** Kafka Follower 冷备，ES 副本分片可读。
12. **MySQL 主从用 VIP 而不是 Master IP 的原因？** VIP 漂移自动接故障转移，Master 切换业务无感知。
