# 熔断策略设计：解决"抖动"的智慧

> 一句话总结：熔断是"舍车保帅"——通过牺牲一小部分请求保全整个服务的稳定；解决"抖动"问题的核心是"软着陆"，推荐用客户端与服务端联动的熔断方案（服务端发信号，客户端从可用列表移除节点）。

---

## 0. 背景与意义

熔断、限流、降级"三兄弟"中，熔断是保障系统可用性的关键一环。

> **熔断的本质**：当一个下游服务因负载过高或程序 Bug 变得不稳定时，主动切断调用直接返回错误，直到该服务恢复。

**关键认知**：直接返回错误反而提高了系统可用性，因为"给了服务端一个喘息和恢复的机会"。

> **小结**：
> - 熔断是"保险丝"，啪的一声切断请求
> - 拒绝新流量 → 处理积压 → CPU 负载下降 → 服务恢复
> - 核心问题：如何判断"不行了"？如何判断"缓过来了"？

---

## 1. 健康状态的判定

### 1.1 常用指标

| 指标 | 适用 | 优势 |
|---|---|---|
| 响应时间 | 对延迟敏感的业务 | 直接反映用户体验 |
| 错误率 | 对错误敏感的业务 | 易于统计和告警 |

### 1.2 阈值设定方法

| 业务情况 | 阈值设定 |
|---|---|
| 产品经理明确 SLA（如 1s 内） | 设 1s 或放宽到 1.2s |
| 无明确业务要求 | 基于历史监控数据（P99=1s → 阈值 1.2~1.5s） |

**原则**：阈值应**明显高于**正常响应时间。

### 1.3 时间窗口：必须持续超过阈值

> 绝不是单次超阈值就立即熔断。

**理由**：
1. 避免网络偶发性抖动或 GC 误判
2. 防止状态"抖动"（Flapping）

**经验值**：30 秒到 1 分钟，根据线上实际表现调优。

> **小结**：
> - 指标 + 阈值 + 时间窗口 = 触发熔断的三大要素
> - 时间窗口过短 → 频繁切换；过长 → 失去保护
> - 持续时间是工程经验与权衡的结果

## 2. 服务恢复：解决"抖动"问题

### 2.1 抖动的根源

> 大多数微服务框架的默认实现：固定时间窗口后**直接 100% 放开流量**

**致命循环**：
- 服务因 1000 QPS 熔断 → 1 分钟恢复期 → 流量未消失 → 100% 放开 → 再次打垮 → 再次熔断

### 2.2 解决方案：软着陆

**半开（Half-Open）状态探测**：

| 阶段 | 流量比例 |
|---|---|
| 熔断期 | 0% |
| 半开期初探 | 10% |
| 探测成功 | 20% → 50% → 100% |
| 探测失败 | 重置计时器，继续隔离 |

### 2.3 更优雅的方案：客户端联动

> 服务端逐步放开流量仍有缺陷——服务端依然接收了 100% 流量，消耗资源。

**核心思想**：让客户端来智能控制流量，从源头控制。

> **小结**：
> - 抖动的根源是"硬着陆"：100% 流量瞬间涌入
> - 软着陆：半开状态逐步试探
> - 客户端联动是更优雅的方案

## 3. 面试应答模板

### 3.1 基础阐述

> "针对不同的服务特性，我制定了不同的熔断策略。对于核心交易服务，主要采用响应时间作为熔断指标。SLA 要求 1s 内响应，将熔断阈值设定在 1.2s。P99 响应时间超过 1.2s **持续 30 秒**后，触发熔断。"

### 3.2 关键追问应答

**Q：阈值 1.2s 怎么定的？**
> 基于业务 SLA 留 20% buffer，或基于历史监控数据。

**Q：为什么是持续 30 秒？**
> 基于经验设定的初始值，过长过短都有利弊，体现权衡思考。

**Q：服务怎么恢复？**
> 等待固定时间（如 1 分钟）后通过半开状态逐步放开流量防"抖动"。

> **小结**：
> - 关键词是"持续超过阈值"
> - 把熔断作为高可用方法论的一环
> - 每个追问都有对应的回答维度

## 4. 亮点方案一：基于依赖的熔断（缓存雪崩防护）

### 4.1 场景

高并发接口严重依赖 Redis 缓存。

### 4.2 策略

> 一旦检测到 Redis 集群不可用，**立即触发熔断**。

**原因**：
- Redis 崩溃 → 所有请求穿透到 MySQL
- 高并发下数据库瞬间被打垮
- 通过熔断牺牲接口可用性，保护数据库资产

### 4.3 恢复机制

启动独立的后台线程（或 Goroutine），**持续低频地 PING Redis**：
- PING 通 → 退出熔断状态，重新接收请求
- PING 不通 → 继续熔断

### 4.4 引导话题的钩子

- **缓存问题**：自然引向 Redis 失效、缓存击穿、穿透
- **高可用**：用熔断保护 MySQL，引出限流
- **恢复策略**：抛出"抖动"问题，引入客户端联动方案

> **小结**：
> - 关键词："关键依赖崩溃"
> - 牺牲非核心保核心，体现"舍车保帅"
> - 恢复策略避免 100% 放开是关键

## 5. 亮点方案二：客户端与服务端联动熔断

### 5.1 整体流程

```
┌────────────────────────────────────────────┐
│ 1. 服务端发出信号                            │
│    触发熔断时返回特定错误码                    │
│ 2. 客户端智能响应                            │
│    RPC 框架识别熔断错误码                     │
│ 3. 流量自动切换                              │
│    将该节点从可用列表暂时移除                  │
│ 4. 客户端主动探测                            │
│    启动计时器（如 1 分钟）                    │
│ 5. 逐步恢复或继续隔离                        │
│    探测成功 → 逐步加回                       │
│    探测失败 → 重置计时器                     │
└────────────────────────────────────────────┘
```

### 5.2 详细步骤

| 步骤 | 行为 |
|---|---|
| 1 | 服务端触发熔断时，返回明确的"已熔断"错误码 |
| 2 | 客户端 RPC 框架识别此错误码 |
| 3 | 立即将出问题的服务端节点从可用列表暂时移除 |
| 4 | 后续新请求不再选择此节点，实现流量无感知切换 |
| 5 | 客户端启动计时器（如 1 分钟） |
| 6 | 计时到点后进入"半开"状态，发探测请求 |
| 7 | 探测成功 → 逐步加回可用列表 |
| 8 | 探测失败 → 重置计时器，继续隔离 |

### 5.3 方案类比

> 像一个**智能的交通调度系统**，客户端不再是盲目发请求，而是和服务端"心有灵犀"。

### 5.4 兜底策略

> 兜底：**所有节点都触发熔断**时，客户端可用列表会变空，必须依赖**监控告警系统**通知人工介入。

> **小结**：
> - 这是综合了负载均衡与熔断的亮点方案
> - 让客户端从源头控制流量，服务端更"轻盈"
> - 体现了架构师的体系化思考能力

## 6. 熔断核心要素对比表

| 要素 | 关键问题 | 经验值 |
|---|---|---|
| 指标选择 | 用响应时间还是错误率？ | 业务敏感度 |
| 阈值设定 | 设多少合理？ | SLA + 20% buffer |
| 时间窗口 | 持续多久触发？ | 30 秒~1 分钟 |
| 恢复策略 | 100% 放开还是逐步放开？ | 软着陆（半开探测） |
| 恢复探测 | 客户端还是服务端主导？ | 客户端联动更优雅 |

## 7. 熔断方案对比表

| 方案 | 触发指标 | 恢复机制 | 复杂度 | 适用场景 |
|---|---|---|---|---|
| 基础熔断 | 错误率/响应时间 | 固定窗口后 100% 放开 | 低 | 简单业务 |
| 半开探测 | 响应时间 | 逐步放开（10%→100%） | 中 | 抖动敏感业务 |
| 关键依赖熔断 | Redis 不可用 | 后台 PING 探测 | 中 | 强依赖缓存 |
| 客户端联动 | 服务端信号 | 客户端从可用列表移除 | 高 | 高可用架构 |
