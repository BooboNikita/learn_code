# 百万 QPS 限流器

> 一句话总结：限流器 = 部署在 API 网关 + 令牌桶算法 + Redis Cluster 分片（一致性哈希）+ Lua 脚本原子操作 + 主从复制高可用，扛 1M QPS 的企业级方案。

---

## 1. 限流器定义

控制流量机制：限制用户/服务请求速率，避免系统过载。
例如：每用户 100 req/min，超出返回 HTTP 429 + `X-RateLimit-Remaining` + `X-RateLimit-Reset`。

> **小结**：
> - 限流 = 保护系统的"水龙头"
> - 拒绝响应需携带剩余次数和重置时间
> - 不是反爬，是 SLA 保障

## 2. 需求与目标规模

### 2.1 功能需求

- 按用户 ID / IP / API Key 识别客户端
- 可配置规则限制（如每用户每分钟 100 次）
- 超限返回 HTTP 429 + 速率限制头

### 2.2 非功能需求（1M QPS 目标）

| 指标 | 要求 |
|---|---|
| 低延迟 | 单次检查 < 10ms |
| 高可用 | 节点间状态可接受最终一致 |
| 高并发 | 1 亿 DAU + 100 万 QPS |

> **小结**：
> - 1M QPS 是设计起点
> - < 10ms 延迟是硬指标
> - 分布式限流可接受最终一致

## 3. 底层设计

### 3.1 核心实体

| 实体 | 作用 | 关键属性 |
|---|---|---|
| Rule | 限流策略 | 时间窗口、请求数、客户端、端点 |
| Client | 被限流对象 | UserID / IP / API Key + 限流状态 |
| Request | 评估请求 | 客户端身份 + 端点 + 时间戳 |

### 3.2 核心接口

```java
isRequestAllowed(clientId, ruleId) -> {
    passes: boolean,      // 是否允许
    remaining: number,    // 剩余次数
    resetTime: timestamp  // 重置时间
}
```

> **小结**：
> - Rule + Client + Request = 限流三元素
> - 接口需返回元数据用于填充 HTTP 头
> - 设计简洁，业务服务易集成

## 4. 部署位置：三种方案对比

| 方案 | 优点 | 缺点 | 推荐 |
|---|---|---|---|
| 进程内限流 | 无网络开销、极快 | 状态不共享、5 实例各 20% 总放行 100% | ❌ |
| 专用服务限流 | 状态集中、动态规则 | 每次请求 +1 网络往返、新单点 | ❌ |
| **API 网关层** | 集中控制、保护上游、零额外调用 | 上下文有限、需 Redis 状态 | ✓ |

**选择 API 网关原因**：
- 应用服务不需关心限流
- 业界最成熟的模式
- 避免为每请求加 RPC 开销

> **小结**：
> - 进程内 = 状态不共享，限流失效
> - 专用服务 = 引入新单点 + 延迟
> - API 网关 = 集中控制 + 零额外调用（推荐）

## 5. 客户端识别

### 5.1 三种标识

| 标识 | 适用 | 注意 |
|---|---|---|
| 用户 ID | 已认证 API | JWT 解析 |
| IP 地址 | 公共 API | NAT/企业防火墙后用户共享 |
| API Key | 开发者 API | X-API-Key 头 |

### 5.2 多层规则叠加

- 每用户限制：小明 1000 req/h
- 单 IP 限制：100 req/min
- 全局限制：50000 req/s
- 端点限制：搜索 10 req/min，个人资料 100 req/min
- **执行最严格的那个**

> **小结**：
> - 不同场景选不同标识，常组合使用
> - 多层规则叠加，执行最严格
> - 高级会员可享受更高限额

## 6. 限流算法：四大算法对比

| 算法 | 原理 | 优点 | 缺点 | 推荐 |
|---|---|---|---|---|
| 固定窗口 | 1min 计数，窗口重置 | 简单 | **临界问题**（前后 0.5s 各 5 = 1s 10 个） | ❌ |
| 滑动窗口 | 拆小子窗口求和 | 解决临界 | 仍非完全平滑 | 中 |
| 漏桶 | 固定速率流出 | 流量均匀 | 无法动态调整、突发处理有限 | ❌ |
| **令牌桶** | 桶中令牌以稳定速率补充 | **既稳定又允许突发** | 参数调优 | **✓** |

**令牌桶工作原理**：
- 桶容量 = 突发容量（如 100）
- 补充速率 = 稳定速率（如 10/min）
- 请求消耗 1 令牌，无令牌则拒绝
- 冷启动：闲置客户端重新激活时拥有满桶

> **小结**：
> - 固定窗口 = 简单但有临界问题
> - 漏桶 = 平滑但缺灵活性
> - 令牌桶 = 稳定 + 突发平衡（推荐）

## 7. 分布式实现：Redis + Lua

### 7.1 选 Redis 的原因

- 高性能：亚毫秒级响应
- 原子操作：Lua 脚本保证并发安全
- 高可用：主从复制 + 哨兵/集群
- 自动过期：EXPIRE 清理不活跃用户

### 7.2 数据结构

```redis
HSET user:bucket tokens <count>
HSET user:bucket last_refill <timestamp>
```

### 7.3 流程

1. 网关 `HMGET` 获取令牌桶状态
2. 内存中计算时间差 × 补充速率 + 新令牌（不能超桶容量）
3. 判断令牌 ≥ 1：是 → -1 放行，否 → 拒绝
4. `HSET` 写回新状态

### 7.4 并发问题与 Lua 脚本

**竞态条件**：两个网关实例同时读到 1 令牌，都放行 → 超放 1 个。

**解决方案**：Lua 脚本封装"读-计算-更新"逻辑，Redis 保证原子执行。

```lua
-- 伪代码
local bucket = HMGET(user:bucket, tokens, last_refill)
local tokens = calculate_refill(bucket, current_time)
if tokens >= 1 then
    tokens = tokens - 1
    HSET user:bucket tokens last_refill
    return {1, tokens}
else
    return {0, 0}
end
```

> **小结**：
> - Redis = 令牌桶状态的"唯一事实来源"
> - Lua 脚本 = 原子性保障
> - 避免"读-改-写"竞态

## 8. 扩展到百万 QPS

### 8.1 Redis 集群分片

- 单 Redis ≈ 10 万 QPS → 10 分片 = 100 万 QPS
- 一致性哈希：`hash(userId) % 10`
- 同客户端请求 → 同一分片 → 状态完整
- **生产推荐 Redis Cluster**：内置 16384 哈希槽，客户端库自动路由

### 8.2 高可用

| 策略 | 描述 |
|---|---|
| 主从复制 | 每分片 1 主多从 |
| 哨兵/Cluster | 自动故障转移 |
| 跨机房 | 异地多活 |

**故障开放 vs 故障关闭**：

| 策略 | 效果 | 适用 |
|---|---|---|
| Fail-Open | 放行所有请求 | 保 API 可用，但可能压垮后端 |
| Fail-Closed | 拒绝所有请求 | 保后端，但 API 不可用 |

**推荐 Fail-Closed**：热点事件时保护后端是首要任务，短暂拒绝部分请求远好过全平台崩溃。

### 8.3 最小化延迟

- **连接池**：复用 TCP 连接，省握手开销
- **地理就近部署**：API 网关与 Redis 同 Region
- **避免本地缓存**：本地缓存与 Redis 不一致会误判（限流可接受短暂不准）

### 8.4 热点用户处理

- **滥用流量**：自动拉黑（1min 触发 10 次限流 → 拉黑 IP/API Key）
- **合法高流量**：高等级 API 套餐 + 独立集群 + 客户端限流退避

### 8.5 动态规则配置

| 模式 | 方式 | 实时性 | 复杂度 |
|---|---|---|---|
| 拉（Pull） | 网关定时拉取（30s） | 中 | 低 |
| 推（Push） | 配置中心长连接推送 | 高 | 高 |

> **小结**：
> - 一致性哈希分片：同客户端 → 同分片
> - Fail-Closed 优先保护后端
> - 连接池 + 地理就近降低延迟
> - 滥用自动拉黑 + 热点独立集群
