# 秒杀系统设计

> 一句话总结：秒杀 = 多层流量拦截（CDN/网关/排队）+ Redis 预扣库存 + MySQL 乐观锁 + TCC/事务消息兜底，四层架构共同扛住 100 万 QPS 瞬时流量。

---

## 1. 需求拆解：业务 + 技术双视角

### 1.1 用户视角 6 个环节

资格校验 → 查库存 → 锁定库存 → 创建订单 → 支付 → 短信通知。
- 只有资格通过才查库存
- 库存不足则不创建订单
- 拆细流程才能定拦截点

### 1.2 技术视角 4 类指标

| 指标类型 | 关键参数 |
|---|---|
| 并发能力 | 峰值 QPS 100 万，核心 TPS ≥ 5 万 |
| 稳定性 | 99% 请求 200ms 内响应，RTO ≤ 5min |
| 数据一致性 | 库存 100% 准确（零超卖），状态延迟 < 10s |
| 安全 | 防脚本 ≥ 99%，黑名单 100% 拦截 |

> **小结**：
> - 秒杀 = 100 倍日常流量 + 极短响应窗口
> - 拆需求比定方案重要：模糊需求 → 不可落地
> - 4 类指标缺一不可：性能 / 稳定 / 一致 / 安全

---

## 2. 整体架构：四层骨架

| 层级 | 核心职责 | 关键技术 |
|---|---|---|
| 接入层 | 拦截无效流量、静态加速 | CDN + APISIX 网关 |
| 流量削峰层 | 缓冲、排队、削峰填谷 | RocketMQ + Redis List |
| 业务逻辑层 | 资格/库存/订单核心流程 | Spring Cloud Alibaba + Caffeine |
| 数据层 | 高读写 + 冷热分层 | Redis Cluster + MySQL 分库分表 |

### 2.1 接入层：CDN + APISIX

- **CDN 加速**：商品图、活动页、倒计时静态资源全量推 CDN，华北/华东地域路由，**活动前 1 小时预热**
- **APISIX 精细化限流**：令牌桶插件按 IP/用户/登录态多维控制（单 IP 5 req/s、单用户 10 req/min）

### 2.2 流量削峰层：RocketMQ + 排队队列

- 选 RocketMQ 而非 Kafka：支持**事务消息**，下单/扣库存同步
- 排队机制用 Redis List：用户进入 → LPUSH 入队 → 返回位置 → 可视化等待
- 排队可降低 30% 重复刷新
- 应急：单分区积压 > 10 万 → 加 2 个临时消费组，暂停非核心消费

### 2.3 业务逻辑层：微服务拆分

- 三个独立服务：资格校验、库存扣减、订单生成
- **Caffeine 本地缓存**：用户等级/资格状态 Key = `userId:productId:qualification`，TTL 30min
- **热点隔离**：爆款商品独立集群（20 台 → 50 台），避免热点拖垮整体

### 2.4 数据层：Redis Cluster + Sharding-JDBC

- Redis：3 主 3 从、16 分片，单分片 5 万 QPS，总 80 万 QPS
- MySQL：按商品 ID 哈希分 8 库 × 16 表 = 128 表，单表 < 50 万
- 读写分离：主库写 / 从库读，主从延迟 1-3s 需提示用户

> **小结**：
> - 四层各司其职：上游拦截 > 下游硬扛
> - RocketMQ 事务消息是秒杀分布式一致的关键
> - 热点隔离与读写分离是数据层降本增效核心

## 3. 关键细节：三个致命坑

### 3.1 防超卖：Redis Lua + MySQL 乐观锁

**Lua 脚本**（Redis 侧原子扣减）：

```lua
-- KEYS[1] = "seckill:stock:{productId}"
local available = redis.call("HGET", KEYS[1], "available")
if not available or tonumber(available) < tonumber(ARGV[1]) then
    return 0  -- 库存不足
end
redis.call("HINCRBY", KEYS[1], "available", -tonumber(ARGV[1]))
return 1
```

**MySQL 乐观锁**（version 字段）：

```sql
UPDATE seckill_stock 
SET available_stock = available_stock - 1, version = version + 1
WHERE product_id = #{productId}
  AND available_stock >= 1
  AND version = #{version};
```

影响行数 = 1 成功；= 0 则回滚 Redis 库存。

### 3.2 缓存三大问题

| 问题 | 表现 | 解决方案 |
|---|---|---|
| 穿透 | 请求不存在的商品 ID | 布隆过滤器 + 缓存空值（1min 过期） |
| 击穿 | 热点 key 过期瞬间压垮 DB | Redisson 分布式锁 + 热点 key 永不过期 |
| 雪崩 | 大量 key 同时过期 | 过期时间 ±10min 随机化 + Caffeine 兜底 |

**Redisson 分布式锁示例**：

```java
RLock lock = redissonClient.getLock("lock:product:" + productId);
try {
    if (lock.tryLock(5, 30, TimeUnit.SECONDS)) {
        Product p = productMapper.selectById(productId);
        redisTemplate.opsForValue().set("product:" + productId, p, 1, TimeUnit.HOURS);
        return p;
    }
} finally {
    if (lock.isHeldByCurrentThread()) lock.unlock();
}
```

### 3.3 分布式一致性：TCC + 事务消息 + 对账

| 方案 | 适用场景 | 关键点 |
|---|---|---|
| 轻量 TCC | 秒杀短事务 | Try 预扣 / Confirm 确认 / Cancel 回滚 |
| RocketMQ 事务消息 | 高峰断连 | 失败重试 3 次 → 死信队列 + 人工 |
| 实时对账 | 兜底 | 每分钟对比 Redis vs MySQL，差值 > 10 以 MySQL 为准修复 Redis |

> **小结**：
> - Redis Lua 脚本是防超卖第一道防线
> - 缓存问题三剑客：穿透/击穿/雪崩，方案各异
> - 一致性方案分层：TCC 主流程 + 事务消息容错 + 对账兜底

## 4. 性能优化三个方向

| 优化方向 | 接入层规则 | 应用层规则 | 效果 |
|---|---|---|---|
| 无效请求过滤 | 单 IP > 5 req/s、UA 异常 | 未登录/黑名单/已秒杀用户 | 接入层挡 30% + 应用层挡 50% |
| 异步处理 | — | RocketMQ 普通消息 | 发短信/通知/积分异步 |
| 弹性扩容 | — | K8s HPA | 10 实例 ↔ 2 实例，节省 40% 成本 |

> **小结**：
> - 90% 秒杀请求是无效的，前置过滤是降本关键
> - 异步化是"用户不关心实时性"操作的银弹
> - HPA 自动扩缩容替代手动运维

## 5. 方案对比表

| 方案 | 性能 | 一致性 | 复杂度 | 适用 |
|---|---|---|---|---|
| Redis 预扣 + 异步落库 | 极高 | 最终一致 | 低 | 日常销售 |
| MQ 削峰 + 顺序消费 | 高 | 最终一致 | 中 | 促销秒杀 |
| Redisson 分布式锁 | 中 | 强一致 | 中 | 极致并发 |
| TCC 事务 | 中 | 强一致 | 高 | 金融级 |

## 6. 复习清单

1. **秒杀系统核心挑战是什么？** 瞬时流量是日常 10-100 倍，需在响应速度与系统稳定间平衡。
2. **四层架构及各层职责？** 接入层（CDN/网关拦截）、流量削峰层（MQ/排队）、业务层（微服务）、数据层（Redis/MySQL）。
3. **CDN 在秒杀中的作用？** 静态资源全量加速，活动前 1 小时预热到全国节点，避免请求穿透到源站。
4. **为什么选 RocketMQ 而非 Kafka？** 支持事务消息，保证下单与扣库存同步；Kafka 适合日志流。
5. **Redis 预扣库存的关键？** Lua 脚本原子执行读-改-写，避免并发超卖。
6. **MySQL 乐观锁原理？** version 字段对比，更新同时 version+1，影响行数 = 0 则回滚。
7. **缓存穿透/击穿/雪崩区别与方案？** 穿透（不存在的 key）→ 布隆过滤器；击穿（热点 key 过期）→ 分布式锁；雪崩（大量 key 同时过期）→ 过期时间随机化。
8. **三种分布式一致性方案？** 轻量 TCC（短事务）+ RocketMQ 事务消息（断连容错）+ 实时对账（兜底）。
9. **Sharding-JDBC 分库分表策略？** 按商品 ID 哈希分 8 库 × 16 表 = 128 表，单表 < 50 万条。
10. **MQ 积压应急方案？** 单分区积压 > 10 万 → 加 2 个临时消费组 + 暂停非核心消费。
11. **热点隔离如何做？** 爆款商品独立集群（20 → 50 台），与普通商品流量隔离。
12. **秒杀架构设计的四个核心原则？** 流量拦截前置、数据一致是底线、监控容灾不偷懒、持续优化不停步。
