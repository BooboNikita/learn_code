# 缓存一致性方案：八种缓存模式深度对比

> 一句话总结：缓存模式本质是性能与一致性的博弈；推荐大多数业务用"延迟双删"保一致性，写多读少用 Write Back 提性能，金额等强一致场景直接走分布式锁。

---

## 0. 背景与意义

缓存模式是后端架构面试的必考题，但 90% 的候选人存在两个致命短板：

1. **知识面不够全**：只知道最基础的一两种模式
2. **理解不够深**：照本宣科，不知道很多缓存模式本身就是一致性问题的"制造者"

> **小结**：
> - 缓存模式解决"读多写少"场景下的性能与一致性博弈
> - 缓存穿透、击穿、雪崩的"三座大山"与缓存模式紧密相关
> - 任何一种缓存模式都有缺陷，不存在银弹

---

## 1. 八种缓存模式总览

| # | 模式 | 一句话核心 | 一致性等级 |
|---|---|---|---|
| 1 | Cache Aside | 应用代码同时管库和缓存 | 弱一致（理论极端不一致，概率极低） |
| 2 | 同步更新 | 业务代码显式同时写库和缓存 | 弱一致（并发会乱序） |
| 3 | Read Through | 缓存组件代理数据库读 | 同 Cache Aside |
| 4 | Write Through | 缓存组件代理数据库写 | 弱一致（并发写） |
| 5 | Write Back | 只写缓存，异步刷库 | 集中式缓存可逻辑闭环 |
| 6 | Refresh Ahead | 监听 Binlog 异步刷新 | 最终一致 |
| 7 | Singleflight | 合并并发请求，防击穿 | 流量控制模式 |
| 8 | 删除缓存 / 延迟双删 | 更新库后删缓存，二次删 | 工程高性价比 |

> **小结**：
> - 模式 1-6 是读写模式
> - 模式 7（Singleflight）是流量控制模式
> - 模式 8（删除/延迟双删）是最常见的工程实践

---

## 2. Cache Aside（最广泛应用）

### 2.1 核心流程

**写操作**：更新数据库 → 删除缓存（注意是"删除"而非"更新"）

**读操作**：查缓存 → 命中返回 / 未命中查库 → 写入缓存 → 返回

### 2.2 写操作顺序：为什么必须"先更新 DB，后删除 Cache"？

| 顺序 | 风险 | 结论 |
|---|---|---|
| 先删 Cache 后更新 DB | 极易脏数据，且难自愈 | 禁止 |
| 先更新 DB 后删 Cache | 理论极端不一致，概率极低 | 推荐 |

**反例时序**（库存 100→99）：
- 线程 A：删 Cache（库存 100）
- 线程 B：查 Cache（Miss）→ 查 DB（读到 100）→ 写 Cache（100）
- 线程 A：更新 DB（99）
- **结果**：DB=99，Cache=100（脏数据长期存在）

> **小结**：
> - 删除而非更新：更新可能涉及复杂计算，且可能浪费（没人读）
> - "理论极端不一致"需要读线程"卡"在中间 + 写线程跑完全流程，概率极低
> - 实际工程中可忽略

---

## 3. 同步更新模式

业务代码把缓存视为与 DB 平起平坐的数据源，**没有任何封装**。

**核心口诀**：盯住"线程 2"，总是姗姗来迟（后开始），却捷足先登（先结束）。

**反例时序**（库存 100→10→20）：
- 线程 1：更新 DB（10）
- 线程 2：更新 DB（20）→ 更新 Cache（20）
- 线程 1：更新 Cache（10）
- **结果**：DB=20，Cache=10（不一致）

> **小结**：
> - 优先写库：库是 Source of Truth
> - **无论先写库还是先写缓存，同步更新都无法保证强一致**
> - 同步更新不是"设计模式"，是"自然写法"

---

## 4. Read Through（读穿透）

### 4.1 核心思想

业务方只管找缓存要数据，缓存里没有就**缓存组件自己**去数据库加载。

**写操作**：与同步更新一致，没有特殊魔法。

### 4.2 两种异步化变种

| 变种 | 行为 | 适用场景 |
|---|---|---|
| 异步回写 | 查到数据立即返回，后台异步写缓存 | 缓存写入慢（大对象/复杂序列化） |
| 全异步加载 | 缓存未命中直接返回默认值/错误码 | 极致 RT 要求，能容忍降级数据 |

> **小结**：
> - 变种二的代价是业务方当次调用拿不到真数据
> - 变种一的收益相对有限
> - 数据一致性表现与 Cache Aside 半斤八两

---

## 5. Write Through（写穿透）

业务方只负责写入缓存，由**缓存组件代替**业务方去更新数据库。

**细节**：
- 没有强制先刷库还是先刷缓存，但实际优先刷库保证数据安全
- 缓存中无此数据时，是否顺便更新？一般预判会被读到就顺手刷新

### 5.1 异步写权衡

| 变种 | 行为 | 风险 |
|---|---|---|
| 异步写库 | 写缓存后立即返回成功，后台异步写库 | **丢数据**（缓存宕机） |
| 同步写库 + 异步刷缓存 | 写库成功立即返回，后台刷缓存 | 适合写库快、刷缓存慢的场景 |

> **小结**：
> - 写穿透也没能解决并发写的一致性问题
> - 异步写变种有丢数据风险
> - 风险与收益并存

---

## 6. Write Back（回写）

### 6.1 核心思想

**只更新缓存，直接返回**。数据库更新被推迟到缓存过期或被逐出时。

### 6.2 反直觉的结论

> "Write Back 虽然容易丢数据，但在数据一致性方面，其实**比 Cache Aside 表现更好**。"

**分情况讨论**：

| 场景 | 一致性表现 |
|---|---|
| 本地缓存（多节点） | 不一致 |
| 集中式缓存（如 Redis）+ 不考虑宕机 | 可做到逻辑闭环 |

### 6.3 一致性闭环分析

1. **写的一致性**：所有写操作只更新缓存，业务读到的永远是刚写入的（自洽）
2. **读的一致性隐患**：读请求从 DB 加载旧数据回填，可能覆盖新值
3. **解决方案**：用 `SETNX` 代替 `SET`，缓存已有值时放弃回填

> **小结**：
> - 适用场景：对数据丢失有一定容忍度，或缓存层极高可用
> - 不适用于金融等核心场景
> - 使用时观察面试官反应，保守可改为"极大地缓解了不一致问题"

---

## 7. Refresh Ahead（预刷新）

监听数据库 Binlog（如 Canal 组件）来异步刷新缓存。

**核心问题**：与 Write Back 一致——读请求在缓存未命中时查库（旧值），可能晚于写请求+Binlog 刷新，从而把新值覆盖掉。

**解决方案**：读请求回填时用 `SETNX`，规避大部分并发不一致。

> **小结**：
> - CDC 普及后非常流行
> - 把缓存更新逻辑从业务代码中剥离
> - Binlog 消费与 DB 写入之间存在时间窗口

---

## 8. Singleflight

**严格说不是读写模式，是流量控制模式**。

### 8.1 核心机制

当缓存未命中、同一时刻一万个请求查询同一 Key 时，Singleflight 保证**只有一个线程**真正去 DB 加载，其他 9999 个在原地阻塞等待。

### 8.2 优缺点

| 优点 | 缺点 |
|---|---|
| 极大减轻数据库并发压力 | 并发量不高时显得鸡肋 |
| 防止缓存击穿导致雪崩 | 增加一定复杂度 |

> **小结**：
> - 特别适合热点数据（Hot Key）场景
> - 是缓存保护的"最后一道防线"

---

## 9. 删除缓存与延迟双删

### 9.1 基础删除缓存

更新数据时先更新 DB，然后**直接删除**缓存。

**反例时序**（balance 100→200）：
- 读线程：Cache 空 → 查 DB（100）→ 准备回填
- 写线程：更新 DB（200）→ 删 Cache
- 读线程：回填旧值 100 到 Cache
- **结果**：DB=200，Cache=100

### 9.2 延迟双删

**流程**：先删缓存（可选）→ 写 DB → 删缓存 → 休眠 N 毫秒 → 再次删缓存

**为什么需要第二次删除？** 防备读线程把脏数据回写进 Cache。

**延迟时间如何设定？** 覆盖主从同步延迟 + 几百毫秒。

> **小结**：
> - 延迟双删能解决 99.99% 的问题
> - 代价 1：降低缓存命中率
> - 代价 2：增加系统复杂度
> - 理论极端情况（读线程在第二次删除后才回写）依然存在，但概率"比中彩票还低"

## 10. 模式选型与落地实践

### 10.1 选型决策树

| 业务类型 | 推荐模式 | 理由 |
|---|---|---|
| 金额等强一致 | 不走缓存 / 分布式锁 | 一致性优先 |
| 大多数常规业务 | **延迟双删** | 性价比最高 |
| 写多读少 / 统计类 | Write Back | 容忍少量丢失 |
| 热点数据防击穿 | Singleflight | 保护 DB |

### 10.2 装饰器模式落地

```go
// 1. 定义标准接口
type Cache interface {
    Get(key string) any
    Set(key string, val any)
}

// 2. Read Through 装饰器
type ReadThroughCache struct {
    c  Cache               // 基础缓存实例
    fn func(key string) any // 数据加载函数
}

// 3. 植入 Read Through 逻辑
func (r *ReadThroughCache) Get(key string) any {
    val := r.c.Get(key)
    if val == nil {
        val = r.fn(key)  // 回源
        r.c.Set(key, val)
    }
    return val
}
```

> **小结**：
> - 设计统一缓存 SDK 避免业务团队乱用
> - 装饰器模式无侵入地实现 Read Through / Singleflight
> - 业务开发者只需初始化装饰器，传入数据加载逻辑

## 11. 八种模式对比表

| 维度 | Cache Aside | 同步更新 | Read Through | Write Through | Write Back | Refresh Ahead | Singleflight | 延迟双删 |
|---|---|---|---|---|---|---|---|---|
| 一致性 | 弱 | 弱 | 弱 | 弱 | 中（逻辑闭环） | 最终 | N/A | 强（工程级） |
| 性能 | 高 | 中 | 高 | 中 | 极高 | 高 | 中 | 中 |
| 复杂度 | 低 | 低 | 中 | 中 | 中 | 中 | 中 | 中 |
| 丢数据风险 | 无 | 无 | 无 | 无 | **有** | 无 | 无 | 无 |
| 命中率 | 高 | 高 | 高 | 高 | 中 | 高 | 高 | **降低** |
| 适用场景 | 通用 | 简单场景 | 通用 | 写少读多 | 写多读少 | CDC 成熟 | 热点防击穿 | 常规业务 |
