# PPO、DPO、GRPO：大模型对齐三大算法

> 一句话总结：SFT 让模型"会说"，RL 让模型"说好"；PPO 是昂贵的"私教班"，DPO 是高效的"改错本"，GRPO 是 DeepSeek 的"小组竞赛制"。

## 一、为什么需要强化学习对齐？

SFT（监督微调）的本质是让模型模仿标注数据。但 SFT 模型只知道"什么是对的"，不知道"什么是更好的"——对同一个问题，标准解题步骤、直接蹦答案、写代码计算都正确，但用户体验差异巨大。

**强化学习对齐要解决的核心问题**：在都正确的回答中，教会模型哪种更受偏好。

### RLHF 三阶段全景

```
预训练 → SFT 监督微调 → RLHF 强化学习对齐
                            ├─ PPO（经典方案）
                            ├─ DPO（简化方案）
                            └─ GRPO（DeepSeek 方案）
```

## 二、PPO：近端策略优化

> 全称：Proximal Policy Optimization
> 来源：OpenAI 2017 年提出，ChatGPT 早期对齐方案 [$TRAE_REF](https://blog.csdn.net/qq_41797451/article/details/156715266)

### 2.1 核心思想

PPO 是经典的在线强化学习算法，通过**奖励模型**为模型输出打分，再用 RL 优化策略。本质是"先评分，再优化"。

### 2.2 训练流程（四模型架构）

PPO 需要同时维护 **4 个模型**，资源开销巨大：

| 模型 | 作用 | 是否训练 |
|------|------|----------|
| **Actor（策略模型）** | 生成回答，被优化的主角 | 训练 |
| **Reference（参考模型）** | SFT 模型的冻结副本，防止跑偏 | 冻结 |
| **Reward Model（奖励模型）** | 模仿人类偏好，给回答打分 | 预训练 |
| **Critic（价值模型）** | 逐 token 评估，提供即时反馈 | 训练 |

训练循环：
1. Actor 对 prompt 生成回答
2. Reward Model 给回答打总分
3. Critic 逐 token 评估，给出即时反馈（优势函数）
4. 计算 KL 散度惩罚（与 Reference 的偏离程度）
5. 总奖励 = 奖励分 - KL 惩罚，用 PPO Clip 损失更新 Actor

### 2.3 关键机制

- **PPO Clip（裁剪）**：限制策略更新幅度，`ratio = π_new / π_old`，超过 `[1-ε, 1+ε]` 范围就裁剪梯度，防止训练崩塌
- **KL 惩罚**：防止模型为了追求高奖励而偏离基础能力（避免 reward hacking）
- **Critic 价值估计**：提供细粒度的逐 token 反馈，让模型知道"哪一步出错了"

### 2.4 优缺点

| 维度 | 评价 |
|------|------|
| 效果上限 | 最高，适合追求极致效果 |
| 训练复杂度 | 高，四模型同时训练 |
| 显存需求 | 约 4 倍模型大小 |
| 超参敏感度 | 高（学习率、KL 系数 β、Clip ε 等） |
| 训练稳定性 | 较差，容易发散 |
| 适用场景 | 不计成本追求最优效果；需要复杂奖励信号 |

## 三、DPO：直接偏好优化

> 全称：Direct Preference Optimization
> 来源：Stanford 2023 年论文，绕过 RL 的颠覆性方案 [$TRAE_REF](https://blog.csdn.net/guoguozgw/article/details/148328728)

### 3.1 核心思想

DPO 的关键洞察：**为什么需要单独训练奖励模型再做 RL？直接用偏好数据不就行了？**

DPO 通过数学变换（Bradley-Terry 模型），将 RLHF 的奖励最大化问题转化为**简单的监督学习问题**，完全绕过奖励模型和 RL 训练。

### 3.2 训练流程（两模型架构）

| 模型 | 作用 | 是否训练 |
|------|------|----------|
| **Policy Model** | 直接在偏好数据上训练 | 训练 |
| **Reference Model** | SFT 模型的冻结副本 | 冻结 |

数据格式：三元组 `(prompt, chosen, rejected)`

DPO 损失函数核心：
- **提高** chosen 回答的生成概率
- **降低** rejected 回答的生成概率
- β 参数控制与 Reference Model 的偏离程度

### 3.3 与 PPO 的本质区别

| 维度 | PPO | DPO |
|------|-----|-----|
| 模型数量 | 4 个（Actor + Ref + RM + Critic） | 2 个（Policy + Ref） |
| 训练方式 | 在线 RL（生成 → 评分 → 更新） | 离线监督学习（直接拟合偏好数据） |
| 训练稳定性 | 差 | 好（等同普通微调） |
| 计算成本 | 高 | 低（降低 40-60%） |
| 数据依赖 | 偏好数据 → 训练 RM → RL 交互 | 直接使用偏好数据对 |
| 奖励信号 | 显式奖励模型 | 隐式编码在损失函数中 |

### 3.4 优缺点

| 维度 | 评价 |
|------|------|
| 训练简单性 | 极高，类似普通微调 |
| 训练速度 | 比 PPO 快 2-5 倍 |
| 显存需求 | 约 2 倍模型大小 |
| 数据质量要求 | 极高，需要大量高质量 chosen/rejected 对 |
| 效果上限 | 接近 PPO，但在复杂奖励场景不如 PPO |
| 适用场景 | 资源有限；有高质量偏好数据；文本对齐 |

## 四、GRPO：组相对策略优化

> 全称：Group Relative Policy Optimization
> 来源：DeepSeek 团队，首次在 DeepSeekMath 论文中提出，DeepSeek-R1 大规模验证 [$TRAE_REF](https://www.ymshici.com/tech/2223.html)

### 4.1 核心思想

GRPO 的关键创新：**不需要 Critic 价值模型，用"组内相对排名"代替"绝对价值估计"**。

对同一问题采样 G 个回答，计算每个回答的奖励，用组内归一化后的相对奖励作为优势函数。不关心绝对分数，只关心"谁比谁好"。

### 4.2 训练流程

```
对每个问题 q：
  1. 策略模型生成 G 个回答 {o1, o2, ..., oG}
  2. 奖励模型（或规则）对每个回答打分 {r1, r2, ..., rG}
  3. 组内归一化：ri_tilde = (ri - mean(r)) / std(r)
  4. 用归一化后的相对奖励作为每个 token 的优势函数
  5. PPO-like 损失更新策略模型（无 Critic）
```

### 4.3 关键设计

- **取消 Critic 模型**：用组内统计量（均值、标准差）代替价值估计，节省一半以上显存
- **保留 KL 惩罚**：继续用 Reference Model 做 KL 约束，防止 reward hacking
- **多候选采样**：客观任务（数学/代码）可用规则评分（对/错），主观任务仍需 RM
- **Beta 分布奖励转换**：对 0/1 奖励做 Beta 分布建模，避免梯度消失

### 4.4 优缺点

| 维度 | 评价 |
|------|------|
| 模型数量 | 2-3 个（Actor + Ref + 可选 RM） |
| Critic 需求 | 无，用组内归一化代替 |
| 训练稳定性 | 较好，保留 PPO 的 ratio + KL 机制 |
| 采样成本 | 中等，每问题需生成多个回答（G=4~16） |
| 效果上限 | 高，DeepSeek-R1 验证了纯 RL 推理的可行性 |
| 适用场景 | 数学/代码等客观任务；推理能力强化；资源有限但要 RL |

## 五、三者对比总结

### 5.1 全局对比

| 维度 | PPO | DPO | GRPO |
|------|-----|-----|------|
| **提出时间** | 2017（RL 经典） | 2023（Stanford） | 2024（DeepSeek） |
| **模型数量** | 4 个 | 2 个 | 2~3 个 |
| **是否需要 RM** | 需要（显式训练） | 不需要（隐式） | 可选（客观任务可用规则） |
| **是否需要 Critic** | 需要 | 不需要 | 不需要 |
| **训练范式** | 在线 RL | 离线监督学习 | 在线 RL（简化版） |
| **训练稳定性** | 差 | 好 | 较好 |
| **显存需求** | 最高（~4x） | 低（~2x） | 中等（~2-3x） |
| **数据要求** | 偏好数据 + RM 训练 | 高质量偏好对 | prompt + 评分规则/RM |
| **效果上限** | 最高 | 接近 PPO | 高（R1 验证） |

### 5.2 选型决策

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| 追求极致效果，资源充足 | PPO | 效果上限最高，Critic 提供细粒度反馈 |
| 中小团队，有偏好数据 | DPO | 实现简单，成本低，效果好 |
| 数学/代码推理任务 | GRPO | 客观任务天然适合规则评分，无需标注 |
| 混合策略（推荐） | DPO → GRPO → PPO | 先快对齐，再针对性强化，最后精细调优 |

### 5.3 一句话速记

- **PPO = 私教班**：四个老师（Actor/Ref/RM/Critic）全力培训，效果最好，费用最高
- **DPO = 改错本**：直接看"好答案 vs 坏答案"对比学习，省了中间环节，数据质量是关键
- **GRPO = 小组赛**：同一题做 8 份卷子，组内排名定胜负，自然产生训练信号

## 六、技术演进脉络

```
2020-2022  PPO 时代
  └─ ChatGPT 用 PPO 做 RLHF，证明 RL 对齐有效，但成本极高

2023      DPO 时代
  └─ Stanford 绕过 RM + RL，一步到位做偏好优化，降低门槛

2024-2025 GRPO 时代
  └─ DeepSeek 砍掉 Critic，组内相对竞争，纯 RL 跑通推理
  └─ 后续改进：DAPO、GSPO、SAPO 等

趋势：
  └─ 更高效（省模型、省显存）
  └─ 更简单（减少工程复杂度）
  └─ 更通用（多模态对齐、在线学习）
```
