# 亿级视频排行榜

> 一句话总结：亿级排行榜 = 哈希表+堆 维护 TopK + 多副本快照容灾 + 一致性哈希分片 + 双指针法（上升沿+下降沿）精确维护滑动窗口，Redis 缓存 1min TTL 抗海量读。

---

## 1. 需求与规模估算

### 1.1 功能需求

- 客户端查询指定时间周期内 Top K 视频（K ≤ 1000）
- 时间窗口：1h / 1d / 1m / 全时段总榜
- 不支持任意起止时间段查询

### 1.2 非功能需求

| 指标 | 要求 |
|---|---|
| 数据延迟 | < 1min |
| 精确性 | 必须精确，不允许近似 |
| 吞吐量 | 700K TPS（70 万/秒） |
| 查询延迟 | 10-100ms |

### 1.3 规模估算

```
# 写入吞吐
70B views/day ÷ 100k seconds/day = 700K TPS

# 视频总量
1M videos/day × 365 × 10 years ≈ 3.6B videos

# 朴素存储
4B × (8B ID + 8B count) = 64GB（可全内存）
```

> **小结**：
> - 70 万 TPS 是设计起点
> - 64GB 可全内存处理 → 多机分布式可行
> - 几十毫秒响应要求 → 必须预计算

## 2. 底层设计

### 2.1 核心实体

- **Video**：被排行统计的主体
- **View**：用户的观看行为事件
- **Time Window**：统计时间范围（1h/1d/1m）

### 2.2 API

```http
GET /views/top?window={WINDOW}&k={K}

Response: {
  "videos": [
    { "videoId": "...", "views": "..." }
  ]
}
```

> **小结**：
> - 实体非常清晰：视频、观看、时间窗口
> - API 简单：一个 GET 即可
> - 重点在架构演进，不在接口设计

## 3. 上层架构：四步演进

### 3.1 基础方案（单机版）

**核心数据结构**：
- 哈希表（HashMap）：`VideoID → Count`
- 最小堆（Min-Heap）：容量 K=1000，维护 Top K

**处理流程**：
1. 从 Kafka 消费 View 事件
2. 哈希表计数值 +1（原子）
3. 与堆顶比较：> 堆顶 → 移除堆顶，插入新元素 + 调整堆
4. 客户端查询：直接读堆

**致命问题**：
- 单机吞吐远低于 70 万 TPS
- 单点故障：宕机即数据全丢

### 3.2 解决单点故障：三种方案

| 方案 | 描述 | 问题 |
|---|---|---|
| 数据库计数 | 哈希表+堆持久化到 DB | 70 万 TPS 写 DB 不可行、并发更新难、索引开销大 |
| 多副本策略 | N 个副本独立计算 | 资源浪费 ×N、恢复需"追赶"全部历史 |
| **带快照的副本**（推荐） | 定期快照到对象存储 | 快照+追赶仍有开销 |

**带快照方案**：
- 副本定期快照到 OSS/S3
- 新实例启动：加载最新快照 → 从 Kafka 快照时间点开始消费
- 缩短恢复时间

### 3.3 扩展写入吞吐量：分片

| 方案 | 路由 | 优点 | 问题 |
|---|---|---|---|
| 按 ID 固定分区 | `hash(video_id) % P` | 简单 | 扩缩容需大量数据迁移 |
| **弹性分区**（推荐） | 一致性哈希 | 动态扩缩容 | 需 ZooKeeper/etcd 协调 |

**聚合服务**（Top-K Service）：
- 查询所有 P 个分片的 Top-K 堆
- 内存合并排序
- 返回全局 Top-K 给客户端

### 3.4 处理时间窗口：双指针法

**微桶方案**（不可行）：
- `Map<[VideoID, MinuteTimestamp], Count>`
- 问题：堆数据陈旧、计算成本高（30天=43200 桶）、内存爆炸

**双指针法**（推荐）：

每个时间窗口维护独立哈希表 + Top-K 堆，启动两组消费者：

| 指针 | 作用 | 消费内容 | 操作 |
|---|---|---|---|
| 上升沿（Leading Edge） | 处理实时数据 | 当前 Kafka 流 | 各窗口计数值 +1 |
| 下降沿（Trailing Edge） | 处理过期数据 | N 窗口之前的流 | 对应窗口计数值 -1 |

**示例（1小时榜）**：

```
00:05 视频A被观看 → 上升沿：{A:1}
00:20 视频B被观看 → 上升沿：{A:1, B:1}
00:40 视频B被观看 → 上升沿：{A:1, B:2}
01:05 下降沿消费 00:05 记录 → {A:0, B:2}
01:20 下降沿消费 00:20 记录 → {B:1}
01:40 下降沿消费 00:40 记录 → {} (空)
```

**双指针法挑战**：
- Kafka 需保留 1 个月数据（成本）
- 总消费负载 = 1 上升 + 3 下降 = 4 倍读取
- 堆容量设为 2K（K=1000），防频繁进出

### 3.5 海量读请求：缓存

- 缓存 TTL 1min（确保结果不超 1min 延迟）
- 请求 → 缓存 → 各分片堆 → 合并 → 回写缓存

> **小结**：
> - 基础方案 = 哈希表 + 堆
> - 容灾 = 副本 + 快照
> - 扩展 = 一致性哈希分片
> - 滑动窗口 = 双指针法
> - 读性能 = 1min TTL 缓存

## 4. 方案对比表

| 容灾方案 | 资源 | 恢复时间 | 复杂度 |
|---|---|---|---|
| 数据库计数 | 低 | 快（重启） | 中 |
| 多副本 | 高（×N） | 极慢（追赶全量） | 低 |
| **副本+快照** | **高** | **中（加载快照+追赶）** | **中** |

| 分片方案 | 扩缩容 | 复杂度 | 推荐 |
|---|---|---|---|
| 固定分区 | 难 | 低 | ❌ |
| 一致性哈希 | 易 | 中 | ✓ |

| 滑动窗口方案 | 计算成本 | 内存 | 推荐 |
|---|---|---|---|
| 微桶 | 极高 | 爆炸 | ❌ |
| 刷新堆 | 高 | 中 | 中 |
| **双指针法** | **中** | **中** | **✓** |

## 5. 复习清单

1. **亿级视频排行榜的吞吐估算？** 70B views/day ÷ 100k s/day = 700K TPS。
2. **视频总量与存储估算？** 3.6B 视频，朴素存储 64GB（可全内存）。
3. **基础方案数据结构？** 哈希表（计数）+ 最小堆（Top K 容量）。
4. **基础方案致命问题？** 单机吞吐不足 70 万 TPS + 单点故障。
5. **三种容灾方案对比？** 数据库计数（写慢）、多副本（资源×N）、副本+快照（推荐）。
6. **副本+快照方案流程？** 定期快照到对象存储 → 新实例加载快照 → 从快照时间点消费 Kafka 追赶。
7. **固定分区 vs 一致性哈希？** 固定分区扩缩容需大量数据迁移；一致性哈希动态扩缩容。
8. **Top-K 聚合服务作用？** 查询所有分片 Top-K 堆 → 内存合并排序 → 返回全局 Top-K。
9. **微桶方案为什么不可行？** 堆数据陈旧、计算成本高（43200 桶）、内存爆炸。
10. **双指针法原理？** 上升沿 +1、下降沿 -1，每个窗口独立维护精确计数。
11. **双指针法挑战？** Kafka 存 1 个月数据成本高、4 倍消费负载、堆频繁进出（设 2K 容量）。
12. **读性能优化？** 1min TTL 缓存，请求 → 缓存 → 各分片堆 → 合并 → 回写。
