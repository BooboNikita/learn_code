# MQ 消息不丢失

> 一句话总结：要实现 MQ 消息"100% 不丢失"，必须建立**生产者端 + Broker 端 + 消费端**的端到端保障闭环：生产端用 `acks=all` + `min.insync.replicas ≥ 2` + 禁用 `unclean` 选举；Broker 端靠多副本 ISR；消费端坚持"先处理业务、后提交 offset"；跨服务一致性用本地消息表或事务消息。

---

## 1. 消息可能丢失的三个阶段

一条消息从产生到结束，经历三个关键阶段：

| 阶段 | 过程 | 丢失风险 |
|---|---|---|
| **生产阶段** | 业务代码创建消息 → 网络发送 → Broker 回复 ACK | 网络抖动、Broker 宕机、ACK 丢失 |
| **存储阶段** | Broker 接收消息 → 持久化到磁盘 | 操作系统 Page Cache 未刷盘、机房断电 |
| **消费阶段** | 消费者拉取消息 → 本地执行业务 | 异步消费时 worker 未处理完就 ACK |

> **小结**：
> - 三个阶段都要分别设计保障方案，不能寄希望于单一环节。
> - Kafka 的 `acks` 参数只解决生产阶段问题。
> - 端到端保障闭环是面试亮点。

---

## 2. Kafka 消息存储机制基础

理解消息丢失场景必须先理解 Kafka 存储模型：

| 概念 | 含义 |
|---|---|
| **Topic** | 逻辑上的消息分类（业务维度） |
| **Partition** | Topic 的物理分片，独立有序队列 |
| **Replica** | 每个分区的副本，分布在不同 Broker |
| **Leader / Follower** | 主分区处理读写，从分区同步数据 |
| **ISR** | 与 Leader 保持同步的副本集合 |

> **小结**：
> - Topic 是逻辑概念，Partition 才是物理存储单元。
> - 副本机制是 Kafka 高可用的基石。
> - ISR 副本数决定了 `acks=all` 的实际可靠性。

---

## 3. 生产阶段：acks 参数三级可靠性

`acks` 是 Kafka 生产者发送消息时最关键的可靠性参数：

| acks 值 | 含义 | 性能 | 可靠性 | 丢失场景 |
|---|---|---|---|---|
| `acks=0` | 发送即忘，不管 ACK | **最高** | 最低 | 网络抖动、Broker 宕机必丢 |
| `acks=1`（默认） | 写入 Leader 即成功 | 中 | 中 | Leader 写入后未同步 Follower 就宕机 |
| `acks=all`（或 -1） | 写入所有 ISR 副本才成功 | 最低 | **最高** | 几乎不丢（但仍有 Page Cache 风险） |

**关键追问**：`acks=all` 真的能保证消息不丢吗？
- 答案是**不一定**——还涉及 Broker 存储阶段的 Page Cache 刷盘问题。

> **小结**：
> - `acks=0` 是性能优先，`acks=1` 是默认平衡，`acks=all` 是可靠性优先。
> - 高性能业务不一定能用 `acks=all`，需结合业务容忍度。
> - 面试时要分场景说明三档选择。

---

## 4. 存储阶段：Page Cache 刷盘风险

Kafka 与数据库一样，为性能考虑先写入操作系统的 **Page Cache**，再由操作系统异步刷盘。

**风险场景**：即使 `acks=all`、所有副本都确认收到，消息可能还在内存中。**机房断电**时所有 Broker 同时宕机且无法恢复，内存数据全部丢失。

**Kafka 提供的刷盘参数**（但生产环境很少主动调）：

| 参数 | 作用 |
|---|---|
| `log.flush.interval.messages` | 达到多少条消息时刷盘 |
| `log.flush.interval.ms` | 距离上次刷盘超过多少毫秒强制刷盘 |
| `log.flush.scheduler.interval.ms` | 周期性检查刷盘 |

**权衡**：刷盘间隔越短，性能越差，可靠性越好。**生产实践**中更愿意依赖 Kafka 的副本机制（多 Broker 分布在不同机器）来保证可靠性。

> **小结**：
> - Page Cache 刷盘失败是 Kafka 唯一无法完全避免的丢失场景。
> - 多副本 ISR 机制是更现实的可靠性保障方案。
> - 单个 Broker 100% 不丢理论上做不到，只能通过参数缓解。

## 5. 消费阶段：异步消费的"幽灵丢失"

异步消费架构下消息丢失的典型场景：

```text
消费者线程：拉取消息 → 放入内存队列 → 立即 ACK
                                    ↓
                  worker 线程从内存队列取消息执行业务
                                    ↓
                   业务未处理完 → 应用宕机 → 消息永久丢失
```

**问题根因**：消费者线程把消息放入内存队列后立刻 ACK，实际业务还未处理。

> **小结**：
> - 异步消费追求性能时最容易出现"幽灵丢失"。
> - 解决思路是"先处理业务，后提交 offset"。
> - 批量提交是性能与可靠性的平衡方案。

## 6. 消息丢失的检测机制

**生产端序列号检测法**（轻量级自研方案）：

```text
生产者发送消息时，注入「Producer名称-分区号-递增序号」格式的 ID：
  第一条：ProducerA-Partition0-1
  第二条：ProducerA-Partition0-2
  ...

消费者侧维护「对每个 Producer 的每个分区的期望序列号」：
  当前收到 ...-2，期望下一条 ...-3
  实际收到 ...-4 → 检测到第 3 条丢失，立即告警
```

**分布式环境下的注意事项**：

| 注意点 | 原因 |
|---|---|
| **分区维度有序** | 全局有序难保证，MQ 只能保证分区内有序 |
| **多生产者协调** | 多实例向同一分区发消息时，全局序号协调复杂 |
| **更实际的方案** | 每个生产者维护自己的序列号，附带唯一标识（IP/实例 ID） |

> **小结**：
> - 有 SkyWalking/Jaeger 等链路追踪系统时优先使用。
> - 序列号检测是轻量级兜底方案，体现产品化设计思维。
> - 序列号检测是面试亮点，能拉开与普通候选人的差距。

## 7. 端到端保障方案（核心）

### 7.1 Kafka 端参数配置

```properties
# 极限配置：生产端三大保险
acks = all                    # 写入所有 ISR 副本才返回成功
min.insync.replicas = 2       # ISR 最少需要 2 个副本（防止降级）
unclean.leader.election.enable = false  # 禁止从非 ISR 选 Leader
```

**参数详解**：

| 参数 | 作用 |
|---|---|
| `acks=all` | 确保消息写入 Leader + 所有 ISR Follower |
| `min.insync.replicas ≥ 2` | 防止 ISR 副本数不足时仍允许写入（避免降级到 `acks=1` 都不如） |
| `unclean.leader.election.enable=false` | 防止从落后太多的非 ISR 副本选 Leader 导致数据丢失 |

### 7.2 生产端代码健壮性

**同步发送**：

```java
try {
    RecordMetadata metadata = producer.send(record).get();
    System.out.println("消息发送成功，分区：" + metadata.partition() 
                       + ", 偏移量：" + metadata.offset());
} catch (Throwable e) {
    System.out.println("消息发送失败，准备重试或记录日志！");
    // 实现重试逻辑或将消息持久化到本地磁盘/数据库
}
```

**异步发送**（回调函数必须检查结果）：

```java
producer.send(record, (metadata, exception) -> {
    if (exception != null) {
        System.out.println("消息发送失败，进行处理！");
        // 实现重试或补偿逻辑
    } else {
        System.out.println("消息异步发送成功，分区：" + metadata.partition() 
                           + ", 偏移量：" + metadata.offset());
    }
});
```

### 7.3 消费端批量提交

**核心原则**：先处理业务，后提交 offset。

**批量提交策略**：
- 消费者线程一次性拉取一批消息（如 100 条）
- 分发给 worker 线程池
- **等待 100 条全部处理完**后，一次性提交整批
- 提交完成后再拉取下一批

> **小结**：
> - 三大参数 + 代码健壮性 + 批量提交，构成消息不丢的核心保障。
> - 异常处理（重试/补偿/落库）必须落到代码，不能只靠参数。
> - 批量提交是异步消费场景下避免"幽灵丢失"的关键。

## 8. 亮点方案：本地消息表实现分布式事务

**业务场景**：用户注册成功后给用户加积分——「注册」和「发消息」是两个独立操作，需保证要么都成功要么都失败。

**本地消息表方案**：

```text
1. 开启本地事务
2. 业务操作：在用户库插入用户记录
3. 记录消息：在同一事务中向「本地消息表」插入「待发送」状态记录
4. 提交本地事务（用户 + 待发送消息 原子写入）
5. 事务提交后，尝试发送消息到 Kafka
   ├─ 成功 → 更新消息表状态为「已发送」或删除
   └─ 失败 → 定时任务扫描「待发送」超时的消息进行重试
```

**追问应对**：数据库事务提交了，但服务器宕机了，消息还没发送出去怎么办？
- 不担心——**异步补偿任务**会定期扫描消息表中待发送的消息进行补发。

**方案对比**：

| 维度 | 本地消息表 | RocketMQ 事务消息 |
|---|---|---|
| **通用性** | 通用，适用任何 MQ | 需 MQ 原生支持 |
| **侵入性** | 与业务绑定，高耦合 | 低耦合 |
| **性能影响** | 占业务库资源 | 无 |
| **适用规模** | 业务量不大的中小项目 | 任意规模 |
| **复杂度** | 低（自建轮询） | 中（需理解两阶段提交） |

> **小结**：
> - 本地消息表是"业务 + 消息"原子性的经典方案。
> - 与 RocketMQ 事务消息二选一：MQ 支持优先用事务消息，否则用本地消息表。
> - 失败补偿机制（异步扫描重试）是方案的精髓。

## 9. 端到端保障体系总览

| 阶段 | 风险 | 保障手段 |
|---|---|---|
| **生产阶段** | ACK 失败 / Broker 宕机 | `acks=all` + 同步/异步重试 + 异常处理回调 |
| **存储阶段** | Page Cache 未刷盘 | ISR 多副本 + `min.insync.replicas ≥ 2` + 禁 `unclean` 选举 |
| **消费阶段** | 异步消费提前 ACK | 先处理业务 + 批量提交 offset |
| **跨服务一致性** | 业务与发消息非原子 | 本地消息表 / MQ 事务消息 |

> **小结**：
> - 端到端保障是"参数 + 代码 + 事务"的组合拳。
> - 单点优化无法保证 100% 不丢，必须全链路协同。
> - 面试呈现思考过程：分析问题 → 三阶段拆解 → 闭环方案。

## 10. 复习清单

1. **消息可能丢失的三个阶段？** 生产阶段、存储阶段、消费阶段。
2. **Kafka acks 三档分别是什么？** 0（发送即忘）、1（写入 Leader）、all（写入所有 ISR）。
3. **acks=all 为什么仍可能丢消息？** Page Cache 未刷盘，机房断电时内存数据全部丢失。
4. **min.insync.replicas 的作用？** 设定 ISR 最少副本数，防止写入可靠性降级。
5. **unclean.leader.election.enable=false 的作用？** 禁止从非 ISR 副本选举 Leader，避免数据丢失。
6. **异步消费的"幽灵丢失"场景？** 消费者线程把消息放入内存队列后立即 ACK，worker 线程未处理完应用就宕机。
7. **消息丢失检测的轻量级方案？** 生产端为每条消息注入「Producer-分区-递增序号」ID，消费者检查序号连续性。
8. **本地消息表方案的核心思想？** 将消息发送封装进本地数据库事务，异步补偿任务扫描重试。
9. **本地消息表与 MQ 事务消息的对比？** 本地消息表通用但侵入高，事务消息低耦合但需 MQ 原生支持。
10. **本地消息表宕机场景如何应对？** 异步补偿任务定期扫描"待发送"且超时的消息进行补发。
11. **消费端批量提交的核心原则？** 一次性拉取 N 条消息，等 N 条全部处理完后再一次性提交 offset。
12. **为什么必须建立端到端保障闭环？** 单一参数/环节无法保证 100% 不丢，需生产者 + Broker + 消费者 + 跨服务一致性整体协同。
