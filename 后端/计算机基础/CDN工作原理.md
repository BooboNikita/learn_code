# CDN 工作原理

> 一句话总结：**CDN = "避远就近"——把源站内容缓存到全球各地边缘节点，通过智能 DNS 解析把用户请求路由到最近的节点，从而把"跨大洲"的网络耗时压到"同城"级别**。

---

## 0. 为什么需要 CDN

如果源站只部署在伦敦，纽约用户访问就要跨大西洋走物理光纤，RTT 动辄 100ms+，丢包率也高，体验极差。

CDN 的核心思路：**"避远就近"**。

- 在全球各地部署**边缘服务器（Edge Server）**。
- 把源站内容**提前缓存**到各边缘节点。
- 用户请求过来时，**智能 DNS 解析**把它引导到"地理位置最近 + 链路最优"的节点。

> **小结**：
> - CDN 不是单一服务器，而是**分布式缓存集群**。
> - 它解决的核心矛盾 = 物理距离带来的**高 RTT + 高丢包率**。
> - 适用对象：**静态资源**（图片、CSS、JS、视频）+ 部分**动态内容**（边缘计算）。

---

## 1. CDN 工作全流程（11 步拆解）

以"纽约用户 Bob 访问 www.myshop.com"为例，浏览器输入 URL 后会发生什么：

| 步骤 | 角色 | 动作 | 关键点 |
|---|---|---|---|
| 1 | 浏览器 | 在**本地 DNS 缓存**里查域名 | 命中则直接返回 IP，跳过后续 |
| 2 | 浏览器 → DNS 解析器 | 本地无缓存，向 **DNS 解析器**（通常在 ISP 内）发起递归查询 | |
| 3 | DNS 解析器 | 递归查询，最终问到**权威名称服务器** | |
| 4 | 权威名称服务器 | **不使用 CDN**：直接返回源站 IP；**使用 CDN**：返回 `www.myshop.cdn.com`（CDN CNAME 别名） | 这是 CDN 切入的关键一步 |
| 5 | DNS 解析器 | 继续向权威名称服务器解析 `www.myshop.cdn.com` | |
| 6 | 权威名称服务器 | 返回 `www.myshop.lb.com`（**CDN 负载均衡器**域名） | |
| 7 | DNS 解析器 → CDN 负载均衡器 | CDN 负载均衡器综合**用户 IP / 用户 ISP / 请求内容 / 服务器负载**等挑出"最合适的边缘节点" | 核心调度逻辑 |
| 8 | CDN 负载均衡器 | 返回该边缘节点的 IP 给 DNS 解析器 | |
| 9 | DNS 解析器 | 把 IP 返回给浏览器 | |
| 10 | 浏览器 → 边缘节点 | 浏览器向该边缘节点发起 HTTP 请求，边缘节点**返回缓存内容** | 静态页面/图片/视频 + 边缘计算结果 |
| 11 | 边缘节点（缓存未命中） | 依次回源：边缘 → 区域 → 中央 → **源站** | 分层回源，**靠近用户优先** |

> **小结**：
> - **智能 DNS** = CDN 的大脑：把"请求方"和"最近的资源"匹配起来。
> - **多层回源** = CDN 的容错：边缘没命中就逐层向上找，而不是每次都打源站。

## 2. CDN 缓存的两类内容

| 类型 | 内容 | 例子 |
|---|---|---|
| **静态内容** | 不会因用户/时间变化的内容 | 静态 HTML、图片、CSS、JS、视频文件 |
| **动态内容** | 通过**边缘计算**（Edge Computing）现场生成的结果 | A/B 测试分桶、个性化推荐、地区限流 |

边缘节点没命中缓存时**回源**（回源 = 回源站拿数据）：

```
用户请求
   ↓
边缘 CDN 节点（最近）
   ↓ 缓存未命中
区域 CDN 节点
   ↓ 缓存未命中
中央 CDN 节点
   ↓ 缓存未命中
源站（Origin Server）
```

> **小结**：
> - **静态内容**走 CDN 收益最大（直接缓存复用）。
> - **动态内容**靠边缘计算实现"算力下沉"，但调试和缓存策略复杂。
> - **回源层级** = 缓存命中率低时性能退化的速度；层级越深，回源链路越长。

## 3. CDN 防盗链：4 种保护机制

视频被盗版是 CDN 上最常见的风险之一。常见 4 种防护手段：

| 机制 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **视频加密 + 水印** | AES 等加密视频；动态水印可追溯 | 强保护，水印能定位泄露源 | 加解密消耗性能 |
| **签名 / Token 验证** | 视频链接带时间戳 + 加密签名，过期失效 | 实现简单，对中间人攻击有效 | 签名泄漏后失效 |
| **限制缓存时长** | 设置短 TTL 或主动清除 | 降低被长期盗用风险 | 命中率下降，回源增加 |
| **防盗链（Referer 检查）** | 服务端校验 HTTP `Referer`，只允许白名单域名 | 实施成本极低 | 容易被伪造 `Referer` 绕过 |

> **小结**：
> - 没有银弹：**4 种机制常组合使用**（如签名 + Referer + 短 TTL + 动态水印）。
> - 视频类业务防盗链是**攻防对抗**，单一方案一定会被绕过。

## 4. CDN 核心机制对比

| 维度 | 不使用 CDN | 使用 CDN |
|---|---|---|
| **请求路径** | 用户 → 源站 | 用户 → 最近边缘节点 →（必要时）源站 |
| **RTT** | 跨大洲动辄 100ms+ | 同城 < 20ms |
| **源站压力** | 全部流量打源站 | 缓存命中后零压力 |
| **容灾能力** | 源站宕机 = 全部不可用 | 源站宕机，已缓存内容仍可服务 |
| **成本** | 源站带宽 + 单点机房 | 边缘节点订阅费，但能省源站带宽 |
| **适用内容** | 动态业务逻辑为主 | 静态资源、大文件分发 |

## 5. 复习清单

1. **CDN 的核心思想？** "避远就近"——把内容提前缓存到全球各地边缘节点，让用户访问最近的那一台。
2. **CDN 是如何"找到"最近节点的？** 通过智能 DNS 解析，由 CDN 内部的负载均衡器根据用户 IP、ISP、请求内容和服务器负载综合决定。
3. **CDN 工作流的关键转折点？** 权威名称服务器原本返回源站 IP，使用 CDN 后改为返回 `*.cdn.com` 的 CNAME 别名，把请求"收编"到 CDN 体系。
4. **CDN 缓存的两类内容？** 静态内容（图片/视频/HTML）和动态内容（边缘计算结果）。
5. **CDN 边缘节点未命中怎么办？** 依次回源：边缘 → 区域 → 中央 → 源站。
6. **CDN 防盗链的 4 种手段？** 视频加密 + 水印 / 签名 Token / 限制缓存时长 / Referer 防盗链。
7. **为什么需要 Referer 防盗链？** 防止别人网站直接引用你的资源（盗用流量），校验 HTTP 请求来源域名。
8. **CDN 的核心价值？** 降低 RTT、降低源站压力、提升容灾能力、降低整体带宽成本。
