# 操作系统启动

> 一句话总结：**操作系统启动 = CPU 加电 → 执行 ROM 里的 BIOS/UEFI 自检 + 找启动设备 → 通过 MBR/EFI 引导（一阶段 + 二阶段 boot loader）→ 把操作系统内核加载到内存 → 操作系统自身初始化 → 创建用户态进程**。

---

## 0. 问题的提出

我们写的程序是由**操作系统**加载运行的——这是《程序是怎么一步步运行起来的》里讲过的。

**那么问题来了**：

> 操作系统本身也是一个程序，它**被谁加载**到内存运行？

这是一个"鸡生蛋"的问题。要回答它，必须先理解**内存断电即丢数据**这个特性。

> **小结**：
> - 内存是易失性存储（volatile），断电后**数据全丢**。
> - 所以 CPU 加电那一刻，**内存里一片空白**，不能从中取指。
> - 必须有"CPU 加电后第一段就能执行的程序"——它必须存在**非易失性存储**（ROM、Flash 等）里。

---

## 1. 第一段程序：BIOS / UEFI

计算机加电时，CPU 从一个**硬件上固定好的地址**开始取指。在 IA-32 架构上，这个地址是 `0xFFFF0`。

这个地址指向 **ROM**（现在一般是 Flash）里保存的一小段程序：

| 旧方案 | 新方案 | 全称 |
|---|---|---|
| **BIOS** | **UEFI** | Basic I/O System / Unified Extensible Firmware Interface |

**BIOS / UEFI 干了 3 件事**：

1. **自检（POST, Power-On Self-Test）**：检查 CPU、内存、硬盘、键盘等硬件是否正常。
2. **找到启动设备**：识别哪个磁盘里装着操作系统（一般是硬盘）。
3. **加载启动设备的固定区域**：
   - **BIOS 方案**：把磁盘**前 512 字节**（MBR, Master Boot Record）copy 到内存 `0x7C00` 并跳转执行。
   - **UEFI 方案**：直接读取 EFI 分区里的 `.efi` 引导程序。

> **小结**：
> - BIOS/UEFI 是"开机第一段程序"，**固化在 ROM 里**，不需要被加载。
> - 它的工作就是**自检 + 找到启动设备 + 把控制权交给启动设备**。

---

## 2. MBR 与一阶段 boot loader

以经典 BIOS + MBR 方案为例。

### 2.1 MBR（Master Boot Record）

磁盘**前 512 字节**就是 MBR，结构：

```
MBR（512 字节）
├── 引导程序（boot loader 代码）  ← 446 字节
├── 磁盘分区表                  ← 64 字节
└── 魔数 0x55AA                ← 2 字节（标识这是 MBR）
```

### 2.2 一阶段 boot loader

这段 446 字节的程序**非常简陋**，只能做一件事：

> 把磁盘中**一段连续的数据** copy 到内存并跳转执行。

所以严格来说，它并不是直接"加载操作系统"，而是加载一个**更大的、功能更强的程序**——**二阶段 boot loader**。

> **小结**：
> - 一阶段 boot loader = MBR 里的 446 字节代码，能力极弱。
> - 它存在的意义 = **搭桥**——把控制权从 MBR 交到"二阶段 boot loader"。

---

## 3. 二阶段 boot loader

### 3.1 为什么需要二阶段？

- MBR 只有 512 字节，装不下复杂的文件解析器。
- 操作系统内核是**多文件 + 有目录结构**的复杂程序。
- 一阶段无法识别文件系统结构，所以必须有个"中间人"。

### 3.2 二阶段 boot loader 干啥？

- **解析文件系统**（FAT、ext4 等），找到内核镜像。
- **加载操作系统内核**到内存。
- **传递启动参数**（如 root= 设备、根分区等）。
- **支持多系统选择**——如果你装了多系统，grub 会列出菜单让你选。

典型代表：**GRUB2**（Linux）、**bootmgfw.efi**（Windows）。

### 3.3 Linux 上常见流程

```
BIOS/UEFI
   ↓
一阶段 boot loader（MBR 446B）
   ↓
GRUB stage 1.5（位于 MBR 后的空白扇区，能识别文件系统）
   ↓
GRUB stage 2（位于 /boot/grub，可读 ext4）
   ↓
vmlinuz（Linux 内核镜像）被加载到内存
   ↓
内核自解压 + 初始化
   ↓
启动第一个用户态进程（PID 1：systemd / init）
   ↓
启动完毕，可以登录
```

> **小结**：
> - 一阶段"砖"很弱，二阶段"工具箱"很全。
> - 二阶段 boot loader 才是**真正能加载操作系统**的程序。
> - UEFI 时代省了一阶段，直接由 UEFI 读取 EFI 分区里的 .efi 引导器。

---

## 4. 操作系统自启动

内核被加载到内存并开始执行后，做的事：

1. **硬件初始化**：中断、内存管理、调度器、文件系统。
2. **挂载根文件系统**（`/`）。
3. **启动第一个用户态进程**（PID 1）：
   - Linux：systemd / init
   - Windows：`smss.exe`
4. **PID 1 启动各种系统服务**（网络、登录管理、cron 等）。
5. 出现登录提示符，**操作系统启动完成**。

> **小结**：
> - 内核启动 = **硬件初始化 + 挂载根 FS + 拉起 PID 1**。
> - PID 1 是所有用户态进程的祖先，永远不能死。

---

## 5. BIOS vs UEFI 对比

| 维度 | BIOS | UEFI |
|---|---|---|
| **诞生年代** | 1970s | 2000s 后 |
| **运行模式** | 16 位实模式 | 32/64 位保护模式 |
| **磁盘分区表** | MBR（最大 2TB，4 个主分区） | GPT（最大 9.4ZB，128 个分区） |
| **引导方式** | MBR → 一阶段 → 二阶段 | 直接读 EFI 分区的 .efi 文件 |
| **启动速度** | 较慢 | 快（支持 Fast Boot） |
| **图形界面** | 无 | 有（鼠标、图形） |
| **安全启动** | 无 | Secure Boot（防恶意引导） |

> **小结**：
> - UEFI 是 BIOS 的现代替代品，**更快、更大、更安全**。
> - 现代电脑基本都使用 UEFI，老式机器才用 BIOS + MBR。

## 6. 复习清单

1. **内存的特性？** 易失性，断电后数据全丢。
2. **CPU 加电后从哪里取指？** 硬件上固定好的地址（IA-32 是 `0xFFFF0`），指向 ROM/Flash 里的 BIOS/UEFI。
3. **BIOS 的三件大事？** 自检（POST）、找启动设备、把 MBR 加载到内存执行。
4. **MBR 是什么？** 磁盘前 512 字节，含 446 字节引导代码 + 64 字节分区表 + 2 字节魔数 0x55AA。
5. **为什么需要一阶段 + 二阶段 boot loader？** 一阶段能力太弱（446 字节），无法解析文件系统；二阶段才能加载真正的内核。
6. **二阶段 boot loader 干啥？** 解析文件系统、找到内核镜像、加载到内存、传递启动参数。
7. **操作系统自启动后做的三件事？** 硬件初始化、挂载根文件系统、拉起 PID 1。
8. **PID 1 为什么特殊？** 它是所有用户态进程的"祖先"，由内核直接创建，永远不能死。
9. **UEFI 相对 BIOS 的优势？** 支持 GPT 大磁盘、启动更快、图形界面、支持 Secure Boot。
