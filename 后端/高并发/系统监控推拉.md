# 系统监控推拉：数据采集的两种哲学

> 一句话总结：拉模式（Pull）控制灵活、简化处理但有请求开销和系统负担；推模式（Push）实时性强、事件驱动但有数据过载和管理复杂性；生产环境常通过服务发现（K8s/Zookeeper）解决端点变更问题。

---

## 0. 背景与意义

系统度量数据的收集方式通常分为：

| 方式 | 中文 | 核心角色 |
|---|---|---|
| 拉动式 | Pull | 数据收集方主动发起请求 |
| 推动式 | Push | 数据源主动向收集系统发送 |

两种方式在数据采集和传输中具有不同特点和适用场景。

> **小结**：
> - 监控数据采集的两大流派：Pull vs Push
> - Pull 控制灵活；Push 实时性强
> - 实际系统常结合服务发现解决端点变更

---

## 1. 拉模式（Pull）

### 1.1 工作机制

> 数据收集系统定期向目标设备或系统发送请求，以获取数据。

**关键特征**：
- 收集方主动发起请求
- 目标系统被动响应
- 收集器（Metrics Collector）需知道要从哪些服务端点获取数据

### 1.2 简单方式 vs 服务发现

| 方式 | 优点 | 缺点 |
|---|---|---|
| 文件保存端点 | 实现简单 | 大规模难维护，服务器添加/移除易遗漏 |
| 服务发现（K8s/Zookeeper） | 可靠、可扩展、可维护 | 引入额外组件 |

**服务发现机制**：
- 服务注册其可用性
- 端点列表变化时收集器收到通知
- 包含何时何地收集度量的配置规则

### 1.3 拉模式采集流程（HTTP）

| 步骤 | 操作 |
|---|---|
| 1 | 指标收集器从服务发现获取服务端点配置元数据 |
| 2 | 收集器通过预定义的 HTTP 端点（如 /metrics）获取指标数据 |
| 3 | 收集器注册变更事件通知或定期轮询端点变化 |

> 要公开 /metrics 端点，通常需要在服务中添加客户端库。

### 1.4 优缺点

| 优点 | 描述 |
|---|---|
| 控制灵活性 | 可按需请求数据，指定采集时间或条件 |
| 减少不必要传输 | 无请求时无数据传输，避免浪费 |
| 简化处理 | 数据请求时即可处理，通常已被筛选加工 |

| 缺点 | 描述 |
|---|---|
| 延迟问题 | 请求频率低时存在延迟 |
| 请求开销 | 每次请求消耗资源，数据量大时易成瓶颈 |
| 系统负担 | 多设备/系统使用拉模式时服务器压力大 |

> **小结**：
> - 拉模式 = 收集方主动 + 目标被动
> - 服务发现是端点动态变化的关键
> - 适合对实时性要求不高的场景

---

## 2. 推模式（Push）

### 2.1 工作机制

> 数据源主动向数据收集系统发送数据。

**关键特征**：
- 数据生成方在事件发生或数据更新时主动推送
- 接收方被动接收
- 适合监控、实时分析等高实时性场景

### 2.2 优缺点

| 优点 | 描述 |
|---|---|
| 实时性强 | 数据在生成或变化瞬间被推送 |
| 减少请求开销 | 收集系统不需周期性发请求 |
| 事件驱动 | 根据特定事件或条件触发，精确高效 |

| 缺点 | 描述 |
|---|---|
| 数据过载 | 推送过于频繁时接收方难以处理 |
| 管理复杂性 | 数据源需管理发送频率和触发条件 |
| 网络负担 | 高频推送在带宽有限时成问题 |

> **小结**：
> - 推模式 = 数据源主动 + 收集方被动
> - 实时性强但需控制频率
> - 适合监控告警、实时分析

---

## 3. 拉模式 vs 推模式对比

| 维度 | 拉模式（Pull） | 推模式（Push） |
|---|---|---|
| 发起方 | 数据收集方 | 数据源 |
| 实时性 | 中（取决于请求频率） | 高（事件触发） |
| 控制灵活度 | 高 | 中 |
| 网络流量 | 按需（可优化） | 持续（可能过载） |
| 数据过载风险 | 低 | 高 |
| 请求开销 | 有 | 无 |
| 服务发现依赖 | 强 | 弱 |
| 实施复杂度 | 中 | 高 |
| 数据一致性 | 按采集时刻 | 实时 |
| 适用场景 | 通用监控 | 实时分析、告警 |

## 4. 实际应用选择

### 4.1 典型场景

| 场景 | 推荐 | 理由 |
|---|---|---|
| Prometheus 采集 | Pull | 通过服务发现动态拉取 |
| StatsD / InfluxDB 推送 | Push | 应用主动推送时序数据 |
| 业务日志 | Push | 应用主动上报 |
| 主机性能指标 | 两者皆可 | 取决于规模和实时性要求 |

### 4.2 混合模式

生产环境常采用**混合模式**：
- 关键指标 Push（保证实时性）
- 大批量/低优先级指标 Pull（控制流量）
- 服务发现作为统一的元数据来源

> **小结**：
> - 没有绝对的"最优解"，需结合业务
> - 服务发现是现代监控体系的关键组件
> - 混合模式能平衡实时性和流量控制
