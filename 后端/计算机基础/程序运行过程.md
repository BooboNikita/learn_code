# 程序运行过程

> 一句话总结：**"双击图标"背后 = 操作系统解析可执行文件格式 → 分配内存并加载段 → 用 ELF 入口地址初始化 CPU 指令寄存器 → CPU 从第一条机器指令开始执行**。

---

## 0. 什么是"可执行程序"

很多人以为 `.exe` 或 Linux 可执行文件很神秘，其实它就是**一个普通的文件**。

| 文件类型 | 内容 | 给谁看 |
|---|---|---|
| **txt** | 字符 | 给人看 |
| **chrome.exe / a.out** | 机器指令序列 | 给 CPU 执行 |

**区别**只是：可执行文件里的二进制是 **CPU 能直接解码执行的指令**，外加一些辅助信息（如段表、依赖库清单）。

> **小结**：
> - 可执行程序没有"魔法"，本质是磁盘上一个**有序字节流**。
> - "双击"或者 `./a.out` 的瞬间，操作系统接管，CPU 才会开始工作。

---

## 1. 程序运行的完整流程

从"用户在终端敲下命令"到"屏幕输出结果"，操作系统做了 5 件事：

```
[用户]  ./a.out
   ↓
[1. 定位文件] 操作系统用文件系统找到磁盘上可执行文件的位置
   ↓
[2. 解析文件格式] 识别 ELF / PE 等可执行格式
   ↓
[3. 加载到内存]   把代码段、数据段等 copy 到内存
   ↓
[4. 分配运行区域] 划分代码区 / 数据区 / 堆区 / 栈区
   ↓
[5. 启动执行]     用入口地址初始化 CPU 指令寄存器 (RIP)
   ↓
[CPU]  从第一条指令开始执行
```

### 1.1 解析可执行文件

Linux 下用 **ELF**（Executable and Linkable Format）描述可执行文件。它至少记录：

- **代码段**（.text）在文件中的偏移和大小
- **数据段**（.data / .bss）的位置
- **依赖的动态库**清单
- **入口地址**（Entry Point Address）—— 程序第一条指令在哪

可以用 `readelf -h a.out` 看 ELF 头部信息。

### 1.2 加载到内存

操作系统在内存中找到**一段大小合适的空闲区域**，然后把可执行文件中的：

- **代码段** → 复制到内存（只读 + 可执行）
- **数据段** → 复制到内存（可读写）
- **BSS 段** → 初始化为 0

### 1.3 划分内存区域

经典布局（4 个区）：

```
高地址
+--------------------+
|     栈区 (stack)    |  ← 函数调用、局部变量
+--------------------+
|        ↓↓↑↑         |
+--------------------+
|     堆区 (heap)     |  ← malloc/new 动态分配
+--------------------+
|     数据区 (data)   |  ← 全局变量、静态变量
+--------------------+
|     代码区 (text)   |  ← 机器指令
+--------------------+
低地址
```

如果依赖**动态库**（shared library / .so），动态库会**被加载到堆和栈之间的空闲区域**（因为堆、栈增长方向相反，中间是天然空洞）。

### 1.4 启动 CPU 执行

操作系统的最后一步：用 ELF 头部记录的 **Entry Point Address** 初始化 CPU 的指令寄存器（x86-64 是 `RIP`）。CPU 下次取指令时，自然就从程序的第一条机器指令开始执行了。

> **小结**：
> - **加载 = 磁盘到内存的复制 + 内存区域划分**。
> - **执行 = CPU 从 Entry Point 指定的地址开始一条条取指执行**。
> - 堆和栈只是"使用上的约定"，物理内存里没有"天然"的堆或栈区。

## 2. 关键代码示例

**验证 Entry Point 是什么**：

```bash
$ readelf -h a.out
ELF Header:
  Magic:   7f 45 4c 46 ...
  Entry point address:               0x400586   ← 这就是程序第一条指令
  ...
```

**典型的程序内存布局**（4 个区）：

```c
// 代码区
int sum(int a, int b) { return a + b; }   // .text

// 数据区
int global_init = 100;        // .data（初始化的全局）
int global_uninit;            // .bss（未初始化，自动清零）

// 堆区
char *buf = malloc(1024);     // heap

// 栈区
int main() {
    int local = 0;            // stack
    return 0;
}
```

## 3. 几个常被混淆的"细节"

### 3.1 堆 vs 栈

| 维度 | 栈（stack） | 堆（heap） |
|---|---|---|
| **分配方式** | 编译器自动（局部变量、函数调用） | 程序员手动（malloc / free、new / delete） |
| **增长方向** | 从高地址向低地址 | 从低地址向高地址 |
| **分配速度** | 极快（仅移动栈指针） | 较慢（要走内存分配器） |
| **管理责任** | 编译器 / CPU 指令 | 程序员 / 内存分配器 |
| **典型场景** | 局部变量、函数参数、返回地址 | 动态数据结构、大块内存 |

### 3.2 物理内存 vs 虚拟内存

作为程序员写 C 时，可以**假设程序独占 4G 内存**（32 位），即使物理内存只有 512M。原因：

- 操作系统提供**虚拟地址空间**映射机制。
- 物理内存 512M，**虚拟地址空间**可以做到 4G（甚至更大）。
- 这就是后续要讲的**内存分页** + 内存分配器的内容。

> **小结**：
> - 堆和栈**不是物理上划定的区域**，只是"使用方式的约定"。
> - 程序员"独占内存"的幻觉来自**虚拟内存**，物理内存远小于虚拟地址空间。

## 4. 复习清单

1. **可执行程序是什么？** 磁盘上一个保存机器指令序列的普通文件。
2. **Linux 可执行程序格式？** ELF（Executable and Linkable Format），里面有段表、依赖库、入口地址等。
3. **操作系统加载程序的 5 步？** 定位文件 → 解析格式 → 复制段到内存 → 划分区域 → 初始化 CPU 指令寄存器。
4. **Entry Point 是什么？** 可执行文件中记录的程序第一条指令的地址，CPU 从这里开始执行。
5. **典型内存 4 区？** 代码区、数据区、堆区、栈区。
6. **堆和栈的区别？** 栈编译器自动管理、增长向低地址；堆程序员手动管理、增长向高地址。
7. **动态库加载到哪？** 堆和栈之间的空闲区域（因为二者增长方向相反，中间有空洞）。
8. **为什么"程序独占 4G 内存"是幻觉？** 因为有虚拟地址空间映射，物理内存可以远小于虚拟地址空间。
