# 分布式事务详解

> 一句话总结：分布式事务本质是「在多数据源/微服务下保证多步操作同时成功或同时回滚」；理论派走 2PC/3PC/TCC/SAGA，工程派走 MQ 事务消息 + Seata AT 模式。

---

## 1. 事务的 ACID 基础

| 特征 | 含义 | 备注 |
|---|---|---|
| A 原子性 | 全成功或全失败 | 底层 undo log 实现 |
| C 一致性 | 前后不破坏数据完整性 | 实质由 AID 共同保证 |
| I 隔离性 | 事务互不干扰 | 涉及隔离级别（RU/RC/RR/Serializable） |
| D 持久性 | 提交后数据永久生效 | redo log 落盘保证 |

> **小结**：
> - AID 是机制，C 是目的，不要把 C 单独理解为某种能力
> - 隔离级别越高，并发性能越差，MySQL InnoDB 默认 RR
> - 单机事务靠 undo/redo log 实现，分布式事务要解决的就是把这一套搬到多节点

---

## 2. XA 规范：协议层

XA（eXtended Architecture）由 X/Open 提出，定义三角色：

- **TM（Transaction Manager）**：事务协调者
- **RM（Resource Manager）**：资源管理器（一般是数据库）
- **AP（Application）**：业务应用

```text
      AP
     /  \
    TM   RM1, RM2 ...
     \  /
   协调 + 2PC
```

> **小结**：
> - XA 是规范/协议，不是具体实现
> - 数据库要支持 XA 协议才能参与分布式事务（如 MySQL InnoDB 支持）
> - 2PC / 3PC 是 XA 协议的具体落地方式

---

## 3. 2PC（二阶段提交）

两阶段：**投票阶段** + **执行阶段**。

**阶段 1：CanCommit / Prepare**
- TM 向所有 RM 发 `prepare`
- RM 执行事务 SQL（不提交），返回 YES/NO

**阶段 2：DoCommit / Abort**
- 全 YES → 发 `commit`
- 有 NO / 超时 → 发 `rollback`

```text
TM ─prepare─► RM1 ──YES──┐
   ─prepare─► RM2 ──YES──┼──► commit
   ─prepare─► RM3 ──NO ──┘   abort (rollback)
```

**三大缺陷**：

| 缺陷 | 说明 |
|---|---|
| 同步阻塞 | 整个流程中所有 RM 锁资源，其他事务被阻塞 |
| TM 单点 | TM 宕机 → 整个流程卡死 |
| 数据不一致 | 执行阶段 RM 脑裂，部分 commit 部分未 commit |

> **小结**：
> - 2PC 是分布式事务的「教科书开局」
> - 同步阻塞和单点可缓解，数据不一致基本无解
> - 面试常考三大缺陷，要能脱口而出

---

## 4. 3PC（三阶段提交）

在 2PC 基础上加了一个 **CanCommit** 阶段，并引入**参与者超时机制**。

| 阶段 | 2PC | 3PC |
|---|---|---|
| 询问 | 无 | CanCommit（先问能不能执行） |
| 准备 | Prepare | PreCommit |
| 提交 | Commit | DoCommit |
| 超时 | 无 | 参与者超时自动 commit |

**改进点**：
- CanCommit 降低「无脑 prepare」的浪费
- 超时机制缓解了同步阻塞与 TM 单点

**仍然未解决**：脑裂场景下 PreCommit 后超时自动 commit → 仍可能数据不一致。

> **小结**：
> - 3PC 引入超时 = 给参与者「自己拿主意」的能力
> - 同步阻塞时间变短，但**未根治**
> - 现实工程几乎不用 3PC，记住结论即可

## 5. TCC（Try-Confirm-Cancel）

TCC 是应用层的 2PC 变种，把一个接口拆成 3 个：

```java
public interface OrderTccService {
    // Try: 预占资源（冻结库存、订单预创建）
    boolean tryCreate(OrderDTO dto);

    // Confirm: 真正执行业务（扣减冻结库存）
    boolean confirm(OrderDTO dto);

    // Cancel: 释放 Try 阶段预占的资源
    boolean cancel(OrderDTO dto);
}
```

| 步骤 | 类比 2PC | 业务动作（库存场景） |
|---|---|---|
| Try | Prepare | 冻结库存 |
| Confirm | Commit | 真正扣减冻结库存 |
| Cancel | Rollback | 释放冻结库存 |

> **小结**：
> - TCC 把分布式事务从「数据库层」上提到「应用层」
> - 业务侵入大、幂等要求高，但性能好、灵活
> - 适合核心链路的少量关键服务（如资金、库存）

---

## 6. SAGA

将长事务拆为**多个本地短事务**，每个短事务对应一个**补偿操作**。

| 恢复策略 | 行为 | 适用场景 |
|---|---|---|
| 向前恢复 | 失败重试，**必须成功** | 强一致性关键路径 |
| 向后恢复 | 反向调用补偿接口回滚 | 大部分业务场景 |

> **小结**：
> - SAGA 适合**长流程**业务（订单→支付→物流→积分）
> - 适合一致性要求不高、可以接受最终一致的业务
> - 国内生产实际应用较少，多用于面试表达

---

## 7. 基于 MQ 的最终一致性（工程派主流）

| 方案 | 思路 | 复杂度 |
|---|---|---|
| 本地消息表 | 业务表 + 消息表同库写，独立 worker 投递 | 中 |
| MQ 事务消息（如 RocketMQ） | 半消息 + 状态回查 | 低 |

**RocketMQ 事务消息流程**：

```text
1. 发送 half 消息（对消费者不可见）
2. MQ ACK 给生产者
3. 生产者执行本地事务（不 commit）
4. 根据结果发 commit / rollback
5. 异常时 MQ 反查生产者状态
6. commit → 投递到消费者；rollback → 删除
```

> **小结**：
> - MQ 方案靠「消息可达」+「消费幂等」+「重试」达到最终一致
> - RocketMQ 事务消息是当前最「省心」的轻量方案
> - 关键：**幂等** + **死信** + **告警** 三件套

## 8. 框架：Seata 与阿里云 GTS

**核心组件**：

| 组件 | 作用 |
|---|---|
| TC（Transaction Coordinator） | 事务协调器，维护全局事务 |
| TM（Transaction Manager） | 控制事务边界，发起 commit/rollback |
| RM（Resource Manager） | 分支事务，汇报状态、执行指令 |

**四种模式对比**：

| 模式 | 侵入度 | 一致性 | 性能 | 机制 |
|---|---|---|---|---|
| AT | 低（代理数据源） | 最终一致 | 高 | UNDO_LOG 逆向 SQL |
| TCC | 高（3 接口） | 强一致 | 高 | 应用层 try/confirm/cancel |
| XA | 低（数据库原生） | 强一致 | 低 | 数据库 XA 协议 |
| SAGA | 中（补偿接口） | 最终一致 | 中 | 正向 + 补偿链 |

**AT 模式核心流程**：

```text
1. TM 向 TC 注册全局事务，拿到 XID
2. RM 代理 JDBC，生成前置 UNDO_LOG
3. RM 提交本地事务（业务 + UNDO_LOG 同事务）
4. TC 收到 commit → 异步删除 UNDO_LOG
5. TC 收到 rollback → 查 UNDO_LOG 逆向回滚
```

> **小结**：
> - Seata 是国内事实标准，开源免费
> - AT 模式「零业务代码」即可接入（推荐首选）
> - GTS = 阿里云托管版，API 与 Seata GA 兼容

## 9. 方案总对比

| 方案 | 一致性 | 性能 | 业务侵入 | 学习成本 | 实际生产用 |
|---|---|---|---|---|---|
| 2PC | 强一致 | 低 | 低 | 低 | 少（多用于 DB 内部） |
| 3PC | 强一致 | 低 | 低 | 中 | 几乎不用 |
| TCC | 强一致 | 高 | 高 | 高 | 核心链路少量使用 |
| SAGA | 最终一致 | 中 | 中 | 中 | 长事务场景 |
| MQ 事务消息 | 最终一致 | 高 | 低 | 低 | **中小公司首选** |
| Seata AT | 最终一致 | 高 | 极低 | 低 | **国内主流** |
| Seata XA | 强一致 | 低 | 低 | 低 | 数据库原生支持时 |

> **小结**：
> - 面试答案模板：**Seata AT（默认）+ MQ 事务消息（轻量）+ TCC（核心强一致）**
> - 选型口诀：**侵入低 + 性能高 + 能兜底** 的就先用

## 10. 复习清单

1. **2PC 三大缺陷？** 同步阻塞、TM 单点、数据不一致。
2. **3PC 相对 2PC 的核心改进？** 增加 CanCommit + 参与者超时机制。
3. **TCC 三个方法分别是什么？** Try（预占）、Confirm（确认）、Cancel（回滚）。
4. **TCC 与 2PC 的关系？** TCC 是应用层的 2PC 变种。
5. **SAGA 两种恢复策略？** 向前恢复（重试）和向后恢复（补偿回滚）。
6. **RocketMQ 事务消息的核心机制？** Half 消息 + ACK + 状态回查。
7. **Seata 三个核心组件？** TC（协调器）、TM（边界）、RM（分支）。
8. **Seata AT 模式如何回滚？** 通过前置镜像 UNDO_LOG 生成逆向 SQL 回滚。
9. **AT 与 XA 模式的核心区别？** AT 是应用层代理最终一致，XA 是数据库层协议强一致。
10. **生产环境怎么选分布式事务方案？** 优先 Seata AT 或 MQ 事务消息，强一致核心场景用 TCC。
11. **XA 是什么？** 是 X/Open 提出的分布式事务**规范**，2PC/3PC 是其实现。
12. **分布式事务和本地事务最大的区别？** 涉及多数据源/多服务，需要全局协调者（TM/TC）保证一致。
