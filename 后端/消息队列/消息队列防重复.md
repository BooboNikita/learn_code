# 消息队列防重复

> 一句话总结：消息队列的三大语义（At Most Once / At Least Once / Exactly Once）中，**Exactly Once 最难实现**；主流 MQ 自带的幂等机制（Kafka 幂等 Producer、Pulsar BrokerDeduplication）只能做到 Topic/Partition 级别，**真正的防线是消费端业务幂等**。

---

## 1. 消息队列的三大语义

| 语义 | 含义 | 适用场景 |
|---|---|---|
| **At Most Once**（至多一次） | 消息不会被重复消费，但可能丢失 | 日志收集（少量丢失可容忍） |
| **At Least Once**（至少一次） | 消息不会丢失，但可能重复消费 | 大部分通用业务（需消费端幂等） |
| **Exactly Once**（精确一次） | 消息精准消费一次，不丢不重 | 支付、对账等强一致场景（最难，需事务消息） |

> **小结**：
> - 三种语义体现可靠性的不同权衡：性能、复杂度与一致性。
> - Exactly Once 需要事务消息支持，目前主流 MQ 实现成本较高。
> - 业务选型时先确定"我能容忍丢还是重"，再选对应语义。

---

## 2. 消息重复的产生原因

消息从生产到消费，可能在两个节点发生重复：

**节点 1：生产者 → Broker**
- Broker 已保存消息，但**返回 ACK 失败**（如网络抖动）
- 生产者误以为发送失败，触发重试
- Broker 保存了重复消息

**节点 2：Broker → 消费者**
- 消费者已处理消息，但**返回 ACK 失败**
- Broker 未更新偏移量（offset）
- 同一条消息再次推送给消费者

> **小结**：
> - 重复的本质是"ACK 失败 + 重试机制"在网络异常下的副作用。
> - At Least Once 语义天然伴随重复风险。
> - 任何带重试的 MQ 都需要消费端做幂等。

---

## 3. 生产者幂等：Kafka 幂等 Producer

Kafka 从 0.11.0 起引入幂等 Producer，**仅在单分区内有效**。

```java
Properties props = new Properties();
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);
KafkaProducer<String, String> producer = new KafkaProducer<>(props);
```

**核心原理**：生产者引入两个参数，由 Broker 单分区内唯一标识一条消息。

| 参数 | 含义 |
|---|---|
| **PID**（Producer ID） | 唯一标识一个 Producer 实例 |
| **Sequence Number** | 单调递增数值，标识同一 Producer 发到指定分区的消息 |

- Broker 收到消息时检查 `<PID, SequenceNumber>` 是否已存在
- 已存在则丢弃，保证单分区幂等
- **多分区不生效**：各分区之间无法互相访问 `<PID, SequenceNumber>`

> **小结**：
> - Kafka 幂等 Producer 解决了"生产阶段"重复问题。
> - 局限性：单分区有效，多分区需要事务消息。
> - 启用幂等后会自动开启 `acks=all`、`retries=Integer.MAX_VALUE`。

---

## 4. Broker 幂等：Pulsar 防重机制

Pulsar Broker 通过参数 `BrokerDeduplicationEnabled` 开启防重，对重复消息返回 `-1:-1` 响应。

**核心机制**：按 `ProducerName` 维度记录 `highestSequenceId`。

```text
Producer 发送消息 → 带 sequenceId 字段
                  ↓
        Broker 检查：sequenceId > highestSequenceId?
                  ↓
       是 → 保存 + 更新 highestSequenceId
       否 → 丢弃 + 返回 -1:-1
```

**三个极端情况处理**：

| 场景 | 处理方式 |
|---|---|
| **Producer 断开重连** | 本地 sequenceId 仍在，递增后继续发送 |
| **Producer 宕机重启** | 缓存丢失，Broker 查出 highestSequenceId 发回给 Producer |
| **Producer 和 Broker 都宕机** | Broker 从快照恢复 highestSequenceId，但**可能不准确**（最新未及保存） |

> **小结**：
> - Pulsar Broker 幂等和 Kafka Producer 幂等类似，都只能保证 Topic/Partition 级别。
> - 宕机场景存在数据丢失风险，不能完全依赖。
> - 单机或集群故障下，幂等机制仍有窗口期。

---

## 5. 消费端幂等：业务层防重（最终防线）

靠生产者/Broker 防重都只能到 Topic/Partition 级别，**真正的防线是消费端业务幂等**。

**核心思路**：消息体携带**全局唯一 ID**，消费端根据该 ID 做去重。

### 方案 A：MySQL 唯一索引

```sql
-- 消费订单消息时，订单ID 唯一索引天然防重
INSERT INTO order (id, ...) VALUES (?, ...);
```

| 优点 | 缺点 |
|---|---|
| 简单可靠 | MySQL 不能用 Change Buffer |
| 数据库一致性 | **仅插入场景**有效（更新库存无法去重） |

### 方案 B：Redis SETNX

```java
if (jedis.setnx(ID, "1") == 1) {
    // 首次消费：处理业务，返回 ACK
} else {
    // 已消费过：直接返回 ACK
}
```

| 优点 | 缺点 |
|---|---|
| 通用（适合更新/任意操作） | 需保证 SETNX 成功后的业务一定完成（否则漏消费） |
| 高性能 | 需处理 Redis 持久化丢失问题 |

> **小结**：
> - 消费端幂等是消息防重的"最后一道防线"，业务敏感场景必做。
> - 唯一 ID 推荐使用业务字段（如订单号）或雪花算法生成的全局 ID。
> - 选用 MySQL 唯一索引还是 Redis SETNX，取决于操作类型（插入/更新）和性能要求。

---

## 6. 三层防重方案对比表

| 层级 | 机制 | 作用范围 | 可靠性 | 性能影响 |
|---|---|---|---|---|
| **生产者** | Kafka 幂等 Producer（PID + SequenceNumber） | 单分区 | 高（仅单分区） | 低 |
| **Broker** | Pulsar BrokerDeduplication（按 ProducerName 记录 sequenceId） | Topic/Partition | 中（宕机快照不准确） | 中 |
| **消费端** | MySQL 唯一索引 / Redis SETNX | 全局业务 | **高（业务保证）** | 取决于存储 |

> **小结**：
> - 三层叠加使用：生产者幂等减少"源头重复"，Broker 防重兜底，消费端幂等为最终防线。
> - 对重复敏感的业务（支付、对账）**必须**做消费端幂等。
> - 选型决策：若 MQ 原生支持事务消息，优先用事务消息 + 消费幂等组合。

## 7. 复习清单

1. **MQ 三大语义分别是什么？** At Most Once（不重可能丢）、At Least Once（不丢可能重）、Exactly Once（不丢不重）。
2. **Exactly Once 为什么最难实现？** 需要事务消息支持，保证生产、消费、去重原子性。
3. **消息重复的两个产生节点？** 生产者→Broker 阶段 ACK 失败重试；Broker→消费者阶段 ACK 失败重投。
4. **Kafka 幂等 Producer 的核心参数？** PID（Producer ID）+ Sequence Number，在单分区内唯一标识消息。
5. **Kafka 幂等 Producer 的局限性？** 只能在单分区内生效，多分区需要事务消息。
6. **Pulsar Broker 防重的核心机制？** 按 ProducerName 维度记录 highestSequenceId，重复消息返回 -1:-1。
7. **Pulsar 防重的宕机风险？** Producer 和 Broker 都宕机时，highestSequenceId 快照可能不准确。
8. **消费端幂等的核心思路？** 消息体携带全局唯一 ID，消费端根据该 ID 做去重（MySQL 唯一索引 / Redis SETNX）。
9. **MySQL 唯一索引做幂等的缺点？** 不能用 Change Buffer；仅插入场景有效（更新库存无法去重）。
10. **Redis SETNX 做幂等的注意事项？** SETNX 成功与业务完成需绑定（如加过期时间兜底），处理 Redis 持久化丢失。
11. **三层防重应该如何组合？** 生产者幂等 + Broker 防重 + 消费端幂等，但消费端幂等是最终防线。
12. **为什么不能完全依赖 MQ 自带的防重？** Topic/Partition 级别有限；宕机快照不准确；不能覆盖所有业务场景。
