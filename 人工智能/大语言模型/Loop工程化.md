# Loop 工程化

> 一句话总结：**Loop Engineering = 把「亲自给 Agent 写 prompt 的那个人」替换成一个能自己运转的循环系统**——你不再是 prompt 的作者，而是 prompt 生产系统的设计师。这是继 Prompt / Context / Harness 之后的第四个 "engineering"：模型每变强一截，瓶颈就往外移一层，最后移到了「坐在键盘前按回车的你」身上。

---

## 0. 这个新词从哪冒出来的

点火的是 **Peter Steinberger**（开源 Agent 项目 OpenClaw 作者），2026-06-07 发了条 800 万浏览推文：

> 「每月例行提醒：你不该再给 coding agent 打 prompt 了。你该去设计那个给 agent 打 prompt 的 loop。」

紧接着 **Addy Osmani**（Google Cloud AI 总监）写了长文给概念定名：

> 「Loop engineering 就是把『亲自给 agent 写 prompt 的那个你』替换掉。你转而去设计那个代替你做这件事的系统。」

几天后 **Claude Code 创始人 Boris Cherny** 在访谈里补了一刀：

> 「我已经不 prompt Claude 了。是 loop 在运行着 prompt Claude、决定做什么。我的工作是写 loop。」

一个造工具的、一个定方法论的、一个每天泡在一线的产品创始人，同一周说了同一件事——**这种程度的共振，值得认真看一看**。

> **小结**：
> - Loop Engineering 不是营销轮换，是模型变强后瓶颈迁移的必然产物。
> - 三位关键人物背书：Steinberger（造工具）、Osmani（定概念）、Boris Cherny（一线产品）。
> - 核心动作变化：从「写 prompt」升级为「设计 loop」——杠杆变长，对设计者要求也变高。

## 1. AI 工程化的四次瓶颈迁移

把四个 engineering 排成时间线，规律一目了然：**模型每变强一截，瓶颈就往外移一层**。

| 阶段 | 出现时间 | 解决什么 | 核心动作 | 瓶颈卡在哪 |
|---|---|---|---|---|
| **Prompt Engineering** | 2023 | 模型「听懂」你 | 角色 / 思维链 / 少样本 | 「怎么说」 |
| **Context Engineering** | 2025 | 模型「知道」该看什么 | 把对的代码、文档、工具、历史记忆在对的时机塞进窗口 | 「喂什么」 |
| **Harness Engineering** | 2026 初 | 模型「做对」一连串事 | 工具 + 权限 + 沙箱 + 子 Agent + 上下文管理 | 「在什么环境做」 |
| **Loop Engineering** | 2026-06 | 模型「自己」开始干活 | 设计一个能自己运转的循环，代替你按回车 | 「谁来按回车」 |

**递进关系**：四个词是**层层外推**而非相互替代——上一层的解法是下一层的输入。

打个比方：
- Prompt：跟一个聪明但一根筋的实习生**说话的技巧**。
- Context：给实习生**准备一桌整理好的资料**。
- Harness：给实习生**装一套驾驶舱**（工具 / 权限 / 仪表盘）。
- Loop：把实习生**升级成工单系统**——电话怎么分流、哪类问题转给谁、办结标准是什么、办不了的怎么升级，都由系统决定。

> **小结**：
> - 四个 engineering 是一段**瓶颈迁移史**，不是营销轮换。
> - Prompt ⊂ Context ⊂ Harness ⊂ Loop（层层外推）。
> - Loop Engineering 瞄准的是**最后一个人肉环节**——你坐在屏幕前敲下一条 prompt，agent 才动一下。

## 2. Loop Engineering 到底是什么

### 2.1 一句话定义

> **Loop = 一个递归式的目标：你定义一个目的，AI 持续迭代，直到完成。**

> **Loop Engineering = 把「亲自给 Agent 写 prompt 的那个你」替换掉，让你转而去设计那个代替你做这件事的系统。**

### 2.2 它和传统协作的区别

| 维度 | 过去（回合制） | Loop Engineering（循环制） |
|---|---|---|
| **协作方式** | 你 → prompt → agent → 输出 → 你 | loop → prompt → agent → 输出 → loop |
| **谁在握方向盘** | 你（人全程握着） | loop（系统握着） |
| **节奏** | 你按一次，它动一次 | 心跳驱动，自动循环 |
| **你的位置** | 接线员（每个电话都接） | 工单系统设计者（规则定好，系统自己转） |
| **投入属性** | 消耗品（每条 prompt 一次性） | 资产（好 loop 之后每轮都受益） |
| **难度** | 说一句话 | 设计一套制度（触发/并行/验证/状态/止损） |

### 2.3 一句话记住

> **「我不再是 prompt 的作者，我是 prompt 生产系统的设计师。」**

> **小结**：
> - Loop 的本质 = 「发现任务 → 布置任务 → 检查结果 → 决定下一步」这个闭环的自动化。
> - 工作没变轻松，**杠杆的支点移动了**——从「一次回答变好」到「之后每轮都变好」。
> - 设计 loop 比写 prompt 难得多，等于从「说一句话」升级到「设计一套制度」。

## 3. 一个 Loop 由什么组成（五大件 + 记忆）

把真正跑起来的 loop 拆开看，零件出奇地一致——**五大件，外加一个记东西的地方**。

### 3.1 第一件：自动化（loop 的心跳）

你写了一个完美的工作流脚本，但每次都要你手动启动，它算 loop 吗？**不算**。自动化才让 loop 成为真正的 loop。

- **定时触发**：每天早上 8 点自动跑一轮
- **事件触发**：PR 合并 / CI 失败 / 新 issue 进来时跑一轮

没有心跳，循环不「活」。

### 3.2 第二件：Worktree（让并行不变成打架）

loop 一旦跑起来，经常是几个 agent 同时干活。这时候你会撞上一个**特别具体的麻烦**：两个 agent 同时改同一个文件。

**解法**：**git worktree**——给每个 agent 一个独立的工作目录和独立分支，共享同一份仓库历史，但物理上互不干扰。各干各的，最后各开各的 PR。

```
main
├── worktree-A (agent-A 修改中)
├── worktree-B (agent-B 修改中)
└── worktree-C (agent-C 修改中)
```

### 3.3 第三件：Skill（治好 Agent 的「金鱼记忆」）

Agent 有个天生缺陷：**每个会话都是冷启动**，你项目里的规范、约定、坑，它一概不知。你不解释的地方，它不会空着，会用**一个自信的猜测**填上。

**解法**：**Skill** = 把项目知识写成文件放在仓库里，让 agent 该用的时候自己读。

- 对单次会话：省得你每个会话重新解释一遍。
- 对 loop：**没有 skill，loop 每个周期都从零重新推导你的项目；有了 skill，知识是复利的**。

### 3.4 第四件：Connector（让 loop 摸到真实世界）

一个只能看见文件系统的 loop，撑死了算半个 loop。真实工作流不止于代码：要读 issue 工单、查监控、发消息、开 PR。

**解法**：**Connector**（基于 MCP 协议的连接器）= 把这些外部系统接进来的桥。

接上前后差别：
- 接上前：Agent 只会告诉你「修复方案在这里」。
- 接上后：完整的 loop 会**自己开好 PR、关联好工单，等 CI 变绿之后自己去频道里通知人**。

### 3.5 第五件：Sub-Agent（写的人和查的人必须分开）

五大件里**最有用的结构性设计**，我认为遥遥领先的一条：**把写代码的 agent 和检查代码的 agent 分开**。

理由只有一句话：写代码的那个模型，给自己的作业打分时，**实在太手下留情了**。

- 让 A 出方案
- 让一个**干净上下文**的 B 来挑刺
- B 没有「希望自己是对的」的包袱，挑出来的问题才是真问题

> 同样的「写查分离」思路还能用在**停止条件**上：每轮结束后由一个独立模型判断「做完了没」，不让干活那个 agent 自己说 done。

### 3.6 第六件：记忆（loop 的命根子）

模型在两次运行之间**会忘掉一切**。今天的循环干了什么、哪些做完了、哪些卡住了，明天的循环一概不知道。

**解法朴素到让人意外**：把记忆放在磁盘上，而不是上下文里。一个 markdown 文件、一个任务看板，什么都行，只要它**活在单次对话之外**，记录着「做完了什么、下一步是什么」。

> Osmani 的金句：**「Agent 会忘，但 repo 不会。」**

| 记忆存放位置 | 持久性 | 跨会话 | 推荐 |
|---|---|---|---|
| 上下文窗口 | ❌ 对话结束即失 | ❌ | 只放当前任务 |
| 状态文件（markdown / JSON） | ✅ 永久 | ✅ | **推荐** |
| 向量数据库 | ✅ 永久 | ✅ 检索 | 复杂、过度设计 |
| SQLite / Redis | ✅ 永久 | ✅ | 高频读写场景 |

> **小结**：
> - 一个能转的 loop = **五大件 + 记忆**。
> - 心跳（自动化）让它活着；worktree 让并行不打架；skill 治金鱼记忆；connector 摸真实世界；sub-agent 写查分离。
> - 记忆必须**外化到磁盘**，不放上下文窗口——这是整个 loop 的命根子。

## 4. 拼起来之后，一个真实的 loop 长什么样

Osmani 自己用的 loop（任务是：每天早上自动把项目里值得修的问题找出来、修好、提交审核）：

```
┌─────────────────────────────────────────────────┐
│  Step 1: 心跳触发（早上 8 点）                      │
│  └→ 调用分诊 skill                                │
│      ├→ 读 CI 失败记录 / 未关闭 issue / 最近提交     │
│      └→ 把"哪些值得做"写进状态文件                    │
├─────────────────────────────────────────────────┤
│  Step 2: 对每个值得做的问题                         │
│  └→ 开 worktree，派 sub-agent 起草修复              │
├─────────────────────────────────────────────────┤
│  Step 3: 第二个 sub-agent 登场（写查分离）           │
│  └→ 对照项目 skill 规范和现有测试审一遍              │
├─────────────────────────────────────────────────┤
│  Step 4: 审过了                                    │
│  └→ connector 自动开 PR、更新对应工单              │
├─────────────────────────────────────────────────┤
│  Step 5: loop 搞不定的问题                         │
│  └→ 丢进待办收件箱，等真人来看                       │
├─────────────────────────────────────────────────┤
│  Step 6: 所有经过写回状态文件                        │
│  └→ 明天的循环从今天停下的地方继续                   │
└─────────────────────────────────────────────────┘
```

**整个过程里，你只设计了一次，中间的任何一步，你都没有写过 prompt。**

> **「好 loop 的标志：你只在设计时出现一次，之后只在收件箱前出现。」**

> **小结**：
> - 一个真实 loop = **心跳 + 分诊 + 隔离 + 审查 + 提交 + 升级 + 持久化**七步闭环。
> - 人出现的两个时点：**设计时 + 收件箱里有东西时**——其他时间全交给系统。
> - 最小可跑版本不需要一次配齐，五大件**一件一件长出来**。

## 5. 工具已经追上来了，现在就能搭

### 5.1 五大件对照表（Claude Code vs Codex）

五大件**全部内置**在主流产品里——一年前要自己写 bash 脚本维护，现在打开即用：

| 五大件 | Claude Code | OpenAI Codex |
|---|---|---|
| **自动化（心跳）** | `/loop` 命令（按间隔触发） | `/goal` 命令（按条件触发） |
| **Worktree** | 内置 worktree 机制，多 agent 物理隔离 | 同样的 worktree 支持 |
| **Skill** | `CLAUDE.md` 6 层级声明式指令 | 等价的 repo 级指令文件 |
| **Connector** | MCP 协议连接器 | MCP 协议连接器 |
| **Sub-Agent（写查分离）** | 内置 review sub-agent | 内置 review sub-agent |
| **记忆** | markdown 文件 + 索引常驻 | 同样的 markdown 持久化方案 |

### 5.2 `/loop` vs `/goal`：一个被忽略的关键区别

| 命令 | 节奏 | 停止条件 | 适用场景 |
|---|---|---|---|
| **`/loop` + 间隔** | 每 N 分钟跑一轮（如 10m） | 跑够次数 / 你手动停 | 「等 CI 跑完」类轮询 |
| **`/loop` 不带间隔** | 节奏由模型自己定 | 同上 | 「CI 一般要跑多久」类智能调度 |
| **`/goal` + 条件** | 持续跑 | **条件为真**才停（如「所有测试通过且 lint 干净」） | 有明确「做完」标准的任务 |

> `/goal` 更妙的地方：**每轮结束后由一个独立的模型来判断条件是否达成**——把「写查分离」用在了「什么时候算干完了」这个停止条件上。

### 5.3 实操：30 秒搭出你的第一个 loop

场景：提了 PR 后等 CI，挂了就修，绿了就叫你。**一条命令搞定**：

```bash
# Claude Code 示例
/loop 10m 跑 ./scripts/check_ci.sh
       若失败，按 CLAUDE.md 里的规范修，修完推到当前分支
       若全绿，发 Slack 通知 #my-pr 频道
       完成后输出当天总结
```

**拆开看，麻雀虽小，五脏俱全**：

```python
loop_config = {
    "heartbeat": "10m",                     # 心跳：每 10 分钟自动醒
    "task": "check CI, fix on failure",      # 任务：每轮干什么
    "stop_condition": "all_tests_pass",      # 停止：什么时候算完
    "report": "Slack #my-pr + summary"       # 汇报：完了怎么交差
}
```

> **小结**：
> - loop 部件正在**工具无关化**——Claude Code 和 Codex 五大件一一对应，差的只是商标。
> - **`/goal`（按条件）比 `/loop`（按间隔）更高级**——它用「写查分离」决定停止条件。
> - 最小 loop 只需心跳 + 任务 + 停止条件；worktree / 写查分离的 sub-agent 是后续加的，不是起手就配。

## 6. 三盆冷水：loop 越好用，这三个问题越尖锐

Osmani 本人在博客里也直说：「现在还早，我是持怀疑态度的。」冷水的核心是：**loop 改变了工作，但没有把你从工作中删除**。

### 6.1 第一盆：验证仍然归你

loop 无人值守地运行，听起来很美。但反过来念这句话：

> **一个无人值守运行的 loop，也是一个无人值守犯错的 loop。**

你睡觉时它在干活，也意味着**你睡觉时它在犯错**。就算你配了负责检查的 sub-agent，也别高兴太早：检查 agent 嘴里的「done」，**只是一个声明，不是一个证明**。它说没问题，和真的没问题，中间还隔着**你的眼睛**。

### 6.2 第二盆：理解债（Comprehension Debt）

loop 交付你没写过的代码越快，「仓库里实际存在的东西」和「你脑子里真正理解的东西」之间的鸿沟就越大。这有个专门的名字：**理解债（Comprehension Debt）**。

| 维度 | 技术债 | 理解债 |
|---|---|---|
| **本质** | 代码烂 | 代码可能不烂，但你不知道为什么是对的 |
| **暴露时机** | 写的时候 | 出问题那天 |
| **偿还方式** | 重构 | **读 loop 产出的东西** |
| **顺滑 loop 的影响** | 缓解 | **加速膨胀** |

> 一个顺滑的 loop 不会帮你还这笔债，只会让它涨得更快。

### 6.3 第三盆：认知投降（Cognitive Surrender）

loop 自己转起来之后，你会发现一个**特别舒服的姿势**：不再对产出有自己的观点，它给什么就收什么。

这个状态有个专门的名字：**认知投降（Cognitive Surrender）**。

> Osmani 的话很锋利：**「带着判断力去设计 loop，它是解药；为了逃避思考去设计 loop，它是助燃剂。同一个动作，相反的结果。」**

### 6.4 隐藏第四笔账：Token 成本

loop 是按循环烧 token 的，sub-agent 每多一个，就多一份模型和工具的开销。**务实做法**：把 sub-agent 用在「值得买第二意见」的地方，而不是处处双保险。

> **小结**：
> - 这三盆冷水**不是 loop 的 bug，而是 loop 的代价**，由你来付。
> - 验证 → 理解债 → 认知投降，三个问题**随 loop 越好用反而越尖锐**。
> - Token 成本：sub-agent 是「奢侈品」不是「标配」，买在刀刃上。

---

## 7. 最后：三句话收尾

> **第一**：从 prompt → context → harness → loop，四个词是一场**瓶颈迁移**——模型越强，瓶颈越往外移，最后移到了「亲自按回车的你」身上。

> **第二**：一个 loop = **五大件 + 一份磁盘记忆**——自动化是心跳，worktree 防打架，skill 治金鱼记忆，connector 摸真实世界，sub-agent 写查分离，状态文件让明天接得上今天。

> **第三**：loop 把你的杠杆变长了，但**验证、理解债、认知投降**这三笔账也同时变大了。工具分不出你是在加速还是在逃避，**你自己分得出**。

Osmani 文章结尾的最狠一段：

> 「两个人可以搭一模一样的 loop，得到完全相反的结果。一个用它在自己深刻理解的工作上加速，另一个用它彻底逃避理解工作。loop 分不出区别，**你分得出**。」

## 8. 面试回答套路（3 段式）

1. **澄清问题 + 定性**：「Loop Engineering 是继 Prompt / Context / Harness 之后的第四个 engineering，本质是把『亲自给 Agent 写 prompt 的你』替换成一个能自己运转的循环系统。它的核心动作变化是：从『写 prompt』升级为『设计 loop』——杠杆变长，对设计者要求也变高。」
2. **破题 + 五大件**：点出 loop = **五大件 + 记忆**——自动化（心跳）、worktree（防并行打架）、skill（治金鱼记忆）、connector（接真实世界）、sub-agent（写查分离）。可以用 `/loop 10m 等 CI 变绿` 这条命令举例，说明 30 秒搭一个最小 loop。
3. **坑 + 收尾**：亮三盆冷水——**验证仍归人 / 理解债 / 认知投降**，强调「loop 改变了工作但没把你从工作中删除」。最后点出**「工具无关化」**趋势（Claude Code / Codex 五大件一一对应），证明你理解行业走向。

| 档次 | 答案特征 |
|---|---|
| **60 分** | 「Loop 就是让 agent 自动循环跑任务」——没区分和 Prompt / Context / Harness 的关系 |
| **80 分** | 提到五大件 + 心跳/停止条件，能举 `/loop` 命令例子 |
| **95 分** | 提到**「写查分离用在停止条件上」+「认知投降」+「工具无关化趋势」**——证明你读过原博客并理解行业走向 |

> **小结**：
> - 面试关键词：**瓶颈迁移、五大件、写查分离、认知投降、工具无关化**。
> - 用 3 段式回答：定性 → 破题（五大件 + 命令示例）→ 坑（三盆冷水 + 行业趋势）。

## 9. 复习清单

1. **Loop Engineering 的本质是什么？** 把「亲自给 Agent 写 prompt 的你」替换成一个能自己运转的循环系统。
2. **四个 engineering 的瓶颈迁移顺序？** Prompt（怎么说）→ Context（喂什么）→ Harness（在什么环境做）→ Loop（谁来按回车）。
3. **Loop = ?** 五大件（自动化 / worktree / skill / connector / sub-agent）+ 磁盘记忆。
4. **心跳为什么是 loop 的第一件？** 没有自动化（定时 / 事件触发），loop 只是一个跑过一次的任务，不是 loop。
5. **为什么用 git worktree 做并行？** 给每个 agent 独立工作目录 + 分支，物理隔离，避免改同一个文件打架。
6. **Skill 解决什么？** Agent 每个会话冷启动，skill 把项目知识写成文件让 agent 自己读，对 loop 是**复利的**。
7. **Connector 是什么？** 基于 MCP 协议的连接器，把 issue / 监控 / 消息 / PR 等外部系统接进 loop。
8. **为什么写查要分离？** 写代码的模型给自己作业打分太手软，干净上下文的检查者挑刺才准。
9. **记忆该放哪？** 磁盘上的 markdown / 状态文件，**不放上下文窗口**。金句：「Agent 会忘，repo 不会。」
10. **`/loop` 和 `/goal` 的区别？** `/loop` 按间隔跑，`/goal` 按条件为真才停；`/goal` 的「达成判断」也用写查分离。
11. **理解债和技术债的区别？** 技术债是代码烂，理解债是**代码不烂但你不知道为什么对**。
12. **什么是认知投降？** 用了 loop 之后不再对产出有自己的观点，**它给什么就收什么**——loop 的最隐蔽代价。
13. **Loop Engineering 最大的行业趋势？** **工具无关化**——Claude Code / Codex 五大件一一对应，差的只是商标，loop 设计图纸是真正的资产。
