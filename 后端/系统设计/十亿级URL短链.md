# 十亿级 URL 短链

> 一句话总结：短链系统 = KGS 发号器（62 进制转换）+ NoSQL 宽表（按短链码哈希分区）+ LRU 缓存 20% 热点 + 惰性删除 + 限流防滥用，支撑 300 亿条记录与 2 万读 QPS。

---

## 1. 短链系统定义

将冗长 URL 转换为短 URL，用户访问短链时通过 HTTP 302 重定向到原始长链。

**核心流程**：
- 生成：应用发长 URL → 短链系统返回短 URL
- 重定向：浏览器访问短 URL → 系统返回 302 + Location 头 → 浏览器跳原 URL

### 1.1 业务价值

| 价值 | 说明 |
|---|---|
| 美观便捷 | 社交媒体、短信、二维码刚需 |
| 数据追踪 | 点击次数、来源、地域、设备 |
| 功能扩展 | 有效期、密码保护、设备跳转 |
| 隐藏原始链接 | 推广场景隐藏复杂参数 |

> **小结**：
> - 短链不仅是缩短，更是"中间代理层"
> - 核心是"生成 + 重定向"两个动作
> - 商业价值（追踪） > 字符缩短

## 2. 容量预估

### 2.1 流量估算

| 指标 | 估算值 |
|---|---|
| 写入 QPS | ~200/s（5 亿/月） |
| 读取 QPS | ~20K/s（100:1 读写比） |
| 入口带宽 | 100 KB/s |
| 出口带宽 | 10 MB/s |

### 2.2 存储估算

- 总记录数：5亿/月 × 12月 × 5年 = **300 亿条**
- 单条记录：500 字节
- 总存储：**15 TB**

### 2.3 缓存估算

- 17 亿次/日请求 × 20%（二八原则） = 3.4 亿次
- 所需内存：**170GB**（单台 256GB 机器可承载）

> **小结**：
> - 写 200/s、读 20K/s，悬殊读写比
> - 300 亿记录，15TB 存储，需分片
> - 170GB 缓存需求，单机可行

## 3. API 设计

| API | 路由 | 用途 |
|---|---|---|
| 创建短链 | POST /api/shorten | 返回短链码 + 元数据 |
| 重定向 | GET /{shortened_url} | 302 跳转原 URL |
| 数据分析 | GET /analytics/{shortened_url} | 点击/来源/地域统计 |
| 链接管理 | GET /user/urls | 用户的短链列表 |
| 删除 | DELETE /{shortened_url} | 软/硬删除 |

**防滥用**：每 IP 每分钟最多 10 次创建，匿名用户更严格。

> **小结**：
> - RESTful 接口覆盖完整生命周期
> - userId 来自认证，不通过 Body
> - 必须加限流防短码耗尽

## 4. 数据库设计

### 4.1 选 NoSQL 而非关系型

- 300 亿记录，关系型分库分表复杂
- 数据关系简单，NoSQL 水平扩展能力强
- 推荐：DynamoDB / 阿里云 OTS / Riak

### 4.2 表结构

**URL 表（url_mapping）**：
| 字段 | 类型 | 说明 |
|---|---|---|
| short_code（PK） | String | 短链码，主键 |
| original_url | String | 原始长链 |
| creation_date | Timestamp | 创建时间 |
| expiration_date | Timestamp | 过期时间 |
| user_id | String | 创建者 |

**用户表（users）**：UserID + Name/Email/PasswordHash

**分析表（analytics）**：AnalyticsID + URLHash + ClickTimestamp + ReferringSite + Location

> **小结**：
> - 300 亿记录 + 简单关系 = NoSQL 更合适
> - 宽表模型：短链码为主键，单查询路由
> - 分析表独立，避免影响主表性能

## 5. 短链码生成：核心算法

### 5.1 方案一：哈希编码

- MD5/SHA256 哈希长 URL → Base62/Base64 编码
- 取前 6-8 位作为短链码
- **致命问题**：
  - 6 位 Base64 = 64^6 ≈ 687 亿（够用但临界）
  - 标准 Base64 的 `+`/`/` 在 URL 中需编码（`%2B`/`%2F`）
  - 同 URL 不同编码会产生不同哈希
  - 哈希冲突需重新哈希或加盐

### 5.2 方案二：发号器 + 进制转换（推荐）

**核心思路**：
1. 全局唯一 ID 生成服务（Snowflake / DB 自增）
2. ID 转 62 进制：`10086 (10) → 2Bi (62)`
3. 短链码 = 62 进制字符串

**优势**：
- 绝对唯一：ID 唯一 → 转换后无冲突
- 长度可控：从短到长递增，有效利用码空间
- 性能高效：纯计算

**KGS（Key Generation Service）设计**：
- 预生成大量短码存 Key-DB
- 分配时从未用表移到已用表
- 内存缓存一批供快速响应
- 分配时加锁同步

**存储空间**：
- 62^6 ≈ 568 亿种组合
- 6 字节 × 568亿 = 340GB

> **小结**：
> - 哈希方案：临界 + 冲突 + URL 编码复杂
> - 发号器方案：唯一 + 高效 + 长度可控（推荐）
> - KGS 需预生成 + 缓存 + 锁同步

## 6. 数据分区与复制

| 方案 | 描述 | 问题 |
|---|---|---|
| 基于范围 | 按短链码首字母分区 | 数据倾斜（E 开头特别多） |
| **基于哈希**（推荐） | `hash(short_code) % 256` | 数据均匀 |
| 一致性哈希 | 哈希环分片 | 扩缩容数据迁移少 |

**副本**：每分片主从复制 + 读写分离 + 故障转移。

> **小结**：
> - 哈希分区 > 范围分区（避免倾斜）
> - 一致性哈希支持弹性扩缩容
> - 每分片主从副本保高可用

---

## 7. 缓存策略

- **缓存内容**：短链码 → 原始长 URL
- **工作流程**：
  ```
  请求 → 缓存
    ├─ 命中 → 返回 302
    └─ 未命中 → 查 DB → 写缓存 → 返回 302
  ```
- **容量**：从日流量 20% 起步 ≈ 170GB
- **淘汰策略**：LRU（热点保留，冷链淘汰）
- **缓存集群**：多副本同步写保证一致性

> **小结**：
> - 缓存 20% 热点扛 80% 流量
> - LRU 策略契合短链场景
> - 缓存集群需多副本同步写

---

## 8. 负载均衡

3 个关键节点都需 LB：

| 位置 | 策略 |
|---|---|
| 客户端 → 应用服务器 | 智能 LB（最少连接 / 响应时间加权） |
| 应用 → 数据库 | 健康探测 + 动态权重 |
| 应用 → 缓存 | 同上 |

> 简单轮询不关心服务器实际负载，智能策略更优。

> **小结**：
> - 三层 LB：客户端/DB/缓存
> - 轮询只适合初期，智能 LB 更稳定
> - 健康探测 + 动态权重是关键

---

## 9. 过期链接清理

| 策略 | 描述 |
|---|---|
| 默认有效期 | 2 年 |
| 访问时删除 | 查数据时检查过期，过期则删除 + 返"链接失效" |
| 异步清理 | 凌晨低峰期批量清理"僵尸链接" |
| 密钥复用 | 删除后短码回收到 KGS 未用库 |

> **小结**：
> - 主动扫描全库成本太高
> - 惰性删除 + 异步清理组合
> - 短码回收循环利用

| 短链码生成 | 唯一性 | 冲突 | URL 编码 | 推荐 |

|---|---|---|---|
| 基于范围 | 严重 | 难 | ❌ |
| 基于哈希 | 均匀 | 难 | 中 |

## 11. 复习清单

1. **短链系统核心？** 生成（长→短）+ 重定向（短→长，HTTP 302）。
2. **读写比？** 100:1（读 20K/s，写 200/s）。
3. **5 年总存储？** 300 亿条 × 500 字节 = 15TB。
4. **缓存容量？** 20% 热点 ≈ 170GB（单台 256GB 机器可承载）。
5. **为什么用 NoSQL？** 300 亿记录 + 简单关系，关系型分库分表复杂。
6. **API 防滥用？** 限流（每 IP 每分钟 10 次），匿名更严。
7. **哈希编码方案问题？** 临界长度 + URL 编码复杂 + 同 URL 不同编码冲突。
8. **发号器方案优势？** 绝对唯一 + 长度可控 + 性能高效。
9. **62 进制示例？** `10086(10) = 2Bi(62)`。
10. **KGS 设计要点？** 预生成短码 + 未用/已用两表 + 内存缓存 + 分配加锁。
11. **数据分区方案？** 一致性哈希 > 基于哈希 > 基于范围。
12. **过期清理策略？** 惰性删除（访问时检查）+ 异步清理（凌晨）+ 短码回收。
