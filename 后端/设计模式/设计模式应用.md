# 设计模式实战：六大模式在项目中的应用

> 一句话总结：**设计模式不是炫技**——「**策略**消分支、**责任链**做拦截、**模板**抽骨架、**观察者**解通知、**工厂**封装创建、**单例**保资源」，选型看场景。

---

## 1. 六大模式速查

| 模式 | 类型 | 一句话 | 典型场景 |
|---|---|---|---|
| **策略** | 行为型 | 封装算法族，互相可替换 | 不同文件类型解析、不同支付方式 |
| **责任链** | 行为型 | 多对象依次处理请求 | 下单流程：非空/安全/黑名单/规则校验 |
| **模板方法** | 行为型 | 父类定骨架，子类填细节 | 多个商户 HTTP 调用流程一致，仅代理不同 |
| **观察者** | 行为型 | 一对多状态通知 | 注册成功后发 IM/短信/邮件 |
| **工厂** | 创建型 | 封装对象创建过程 | 配合策略模式返回不同实现 |
| **单例** | 创建型 | 全局唯一实例 | 配置类、连接池、Task Manager |

> **小结**：
> - 行为型（4 个）处理**算法/流程**问题
> - 创建型（2 个）处理**对象怎么来**问题
> - 出现「多种类型 / 多步骤 / 多通知 / 多种创建」时优先考虑

---

## 2. 策略模式：消灭 if...else 分支

### 2.1 业务场景
大数据系统推送不同类型文件，需按类型选解析方式。传统写法是 `if(type=="A")...else if(type=="B")...`，违反**开闭原则**和**单一职责原则**。

### 2.2 核心结构

```java
public interface IFileStrategy {
    FileTypeResolveEnum gainFileType();   // 自报家门
    void resolve(Object objectparam);     // 差异化实现
}

@Component
public class AFileResolve implements IFileStrategy {
    public FileTypeResolveEnum gainFileType() { return FileTypeResolveEnum.File_A_RESOLVE; }
    public void resolve(Object objectparam)   { /* A 类型解析 */ }
}
```

### 2.3 Spring 容器自动收集策略

```java
@Component
public class StrategyUseService implements ApplicationContextAware {
    private Map<FileTypeResolveEnum, IFileStrategy> iFileStrategyMap = new ConcurrentHashMap<>();

    @Override
    public void setApplicationContext(ApplicationContext ctx) {
        ctx.getBeansOfType(IFileStrategy.class).values()
           .forEach(s -> iFileStrategyMap.put(s.gainFileType(), s));
    }

    public void resolveFile(FileTypeResolveEnum type, Object param) {
        IFileStrategy s = iFileStrategyMap.get(type);
        if (s != null) { s.resolve(param); }
    }
}
```

> **小结**：
> - 策略接口要有 **「标记类型 + 业务实现」** 两个方法
> - 用 `ApplicationContextAware` 启动期**自动建索引**
> - 加新策略**只需加一个 `@Component` 类**

## 3. 责任链模式：多步骤校验

### 3.1 业务场景
下订单要做：非空校验 → 安全校验 → 黑名单校验 → 规则拦截。常见反例是**用异常做流程控制**（违反阿里开发手册：禁止用异常做条件判断）。

### 3.2 抽象类骨架

```java
public abstract class AbstractHandler {
    private AbstractHandler nextHandler;

    public void setNextHandler(AbstractHandler next) { this.nextHandler = next; }
    public AbstractHandler getNextHandler() { return nextHandler; }

    public void filter(Request req, Response resp) {
        doFilter(req, resp);                       // 子类业务
        if (getNextHandler() != null) {
            getNextHandler().filter(req, resp);    // 传给下一个
        }
    }
    abstract void doFilter(Request req, Response resp);
}
```

### 3.3 `@Order` 注解控制顺序 + 启动期连链

```java
@Component @Order(1) public class CheckParamFilterObject     extends AbstractHandler { /* ... */ }
@Component @Order(2) public class CheckSecurityFilterObject  extends AbstractHandler { /* ... */ }
@Component @Order(3) public class CheckBlackFilterObject    extends AbstractHandler { /* ... */ }
@Component @Order(4) public class CheckRuleFilterObject      extends AbstractHandler { /* ... */ }

@Component("ChainPatternDemo")
public class ChainPatternDemo {
    @Autowired private List<AbstractHandler> abstractHandleList;
    private AbstractHandler abstractHandler;

    @PostConstruct
    public void initializeChainFilter() {
        for (int i = 0; i < abstractHandleList.size(); i++) {
            if (i == 0) { abstractHandler = abstractHandleList.get(0); }
            else { abstractHandleList.get(i - 1).setNextHandler(abstractHandleList.get(i)); }
        }
    }

    public Response exec(Request req, Response resp) { abstractHandler.filter(req, resp); return resp; }
}
```

> **小结**：
> - 抽象类要有 **next 引用 + set 方法 + 模板方法 filter() + 抽象 doFilter()**
> - 用 `@Order` 注解**声明式控制**节点顺序
> - `@PostConstruct` 中**串链**，业务调用方只调 `exec()`

## 4. 模板方法：固定流程中的可变点

### 4.1 业务场景
A/B 商户调外部 HTTP 接口，流程都是「查商户 → 加签 → 发 HTTP → 验签」，但 A 走代理、B 直连。

### 4.2 骨架 + 可变点下沉

```java
abstract class AbstractMerchantService {
    // 模板方法（final 修饰更佳）
    Resp handlerTempPlate(req) {
        queryMerchantInfo();
        signature();
        httpRequest();
        verifySinature();
    }
    abstract boolean isRequestByProxy();   // 可变点交给子类
}

class CompanyAServiceImpl extends AbstractMerchantService {
    Resp handler(req) { return handlerTempPlate(req); }
    boolean isRequestByProxy() { return true; }   // 走代理
}
class CompanyBServiceImpl extends AbstractMerchantService {
    Resp handler(req) { return handlerTempPlate(req); }
    boolean isRequestByProxy() { return false; }  // 直连
}
```

> **小结**：
> - 核心：**算法骨架在父类，具体步骤在子类**
> - 适用：流程固定、**只有少数步骤不同**
> - 钩子方法（hook）决定可变行为

## 5. 观察者模式：解耦通知

### 5.1 业务场景
用户注册成功后要发 IM/短信/邮件，传统写法 `register()` 里堆叠 3 个发送调用，违反开闭原则且**一个失败会阻塞主流程**。

### 5.2 经典实现：被观察者 + 观察者

```java
public class Observerable {
    private List<Observer> observers = new ArrayList<>();
    private int state;
    public void addServer(Observer o)    { observers.add(o); }
    public void removeServer(Observer o) { observers.remove(o); }
    public void notifyAllObservers(int state) {
        if (state != 1) return;
        for (Observer o : observers) o.doEvent();
    }
}
```

### 5.3 实战：Guava EventBus

```java
public class EventBusCenter {
    private static EventBus eventBus = new EventBus();
    public static void register(Object obj) { eventBus.register(obj); }
    public static void post(Object obj)     { eventBus.post(obj); }
}

public class EventListener {
    @Subscribe
    public void handle(NotifyEvent e) {
        // 发 IM / 短信 / Email
    }
}

// 使用
EventBusCenter.register(new EventListener());
EventBusCenter.post(new NotifyEvent("13372817283", "123@qq.com", "666"));
```

> **小结**：
> - 观察者模式**一对多状态通知**，被观察者状态变 → 通知所有观察者
> - 实战优先用 **Guava EventBus**，注解 `@Subscribe` 即可
> - 注意：**失败不要影响主流程**（加 try-catch 或异步）

## 6. 工厂模式：封装对象创建

### 6.1 经典工厂

```java
interface IFileResolveFactory { void resolve(); }
class AFileResolve implements IFileResolveFactory { public void resolve() { /* A */ } }
class BFileResolve implements IFileResolveFactory { public void resolve() { /* B */ } }
```

### 6.2 实战：工厂 + 策略合体

```java
@Component
public class StrategyUseService implements ApplicationContextAware {
    private Map<FileTypeResolveEnum, IFileStrategy> iFileStrategyMap = new ConcurrentHashMap<>();

    @Override
    public void setApplicationContext(ApplicationContext ctx) {
        ctx.getBeansOfType(IFileStrategy.class).values()
           .forEach(s -> iFileStrategyMap.put(s.gainFileType(), s));
    }
    // 这段 setApplicationContext 本身就是「工厂」——根据枚举返回不同实例
}
```

> **小结**：
> - **工厂模式经常配合策略模式**一起出现
> - 单纯工厂用得少，多数场景是 Spring 容器替你做工厂
> - 核心：**把 `new XXX()` 收拢到一处**

## 7. 单例模式：四类写法对比

| 写法 | 线程安全 | 延迟加载 | 防反射/序列化破坏 | 推荐度 |
|---|---|---|---|---|
| **饿汉式** | ✔ | ✘（类加载即建） | ✘ | ★★ |
| **懒汉式** | ✘（需 synchronized） | ✔ | ✘ | ★ |
| **双重校验锁** | ✔（`volatile` 必加） | ✔ | ✘ | ★★★ |
| **静态内部类** | ✔ | ✔ | ✘ | ★★★★ |
| **枚举** | ✔ | ✘ | **✔**（自动防） | ★★★★★ |

```java
// 双重校验锁
public class DoubleCheckSingleton {
    private volatile static DoubleCheckSingleton instance;
    private DoubleCheckSingleton() {}
    public static DoubleCheckSingleton getInstance() {
        if (instance == null) {
            synchronized (DoubleCheckSingleton.class) {
                if (instance == null) instance = new DoubleCheckSingleton();
            }
        }
        return instance;
    }
}

// 静态内部类
public class InnerClassSingleton {
    private static class Holder { private static final InnerClassSingleton INSTANCE = new InnerClassSingleton(); }
    public static final InnerClassSingleton getInstance() { return Holder.INSTANCE; }
}
```

> **小结**：
> - **`volatile` 必不可少**：防止指令重排导致返回未初始化对象
> - **枚举写法**最优雅：自动防反射、自动防序列化、代码最少
> - 实际项目用 **Spring `@Component`（默认单例）** 即可

## 8. 模式选型决策树

```
需要消除大量 if...else 分支?
├─ 是 → code 有业务含义? → 是 → 策略 + 工厂 / 注解
│                        → 否 → 模板方法 / 责任链
└─ 否 → 多步骤串行校验?  → 是 → 责任链
        → 固定流程中步骤可变? → 是 → 模板方法
        → 一对多通知?    → 是 → 观察者 (EventBus)
        → 全局唯一资源?  → 是 → 单例
        → 封装对象创建?  → 是 → 工厂
```

> **小结**：
> - **没有银弹**：一种模式解决一类问题
> - 实战中**策略 + 工厂 + 单例**最常用
> - 不要为了用模式而用模式，**业务简单就别上**

## 9. 复习清单

1. **六大模式中行为型有几个？** 4 个（策略、责任链、模板、观察者）。
2. **策略模式解决什么问题？** 一组可互换算法，消除类型分支。
3. **策略接口的两个必备方法？** 类型标识 `gainFileType()` + 业务实现 `resolve()`。
4. **责任链模式相比异常的优点？** 异常只能返回异常信息、责任链可返回任意结果且性能高。
5. **责任链启动期如何串链？** `@PostConstruct` 中遍历 `List<AbstractHandler>` 调 `setNextHandler`。
6. **模板方法的核心思想？** 算法骨架在父类，可变步骤下沉到子类（钩子方法）。
7. **观察者模式的核心成员？** 被观察者（Observerable）+ 观察者（Observer）。
8. **Guava EventBus 注解？** `@Subscribe` + `EventBus.register(obj)` + `post(event)`。
9. **单例双重校验锁为什么必须 `volatile`？** 防止指令重排，避免返回未初始化的对象。
10. **单例推荐写法？** 枚举（最优雅，自动防反射与序列化）。
11. **策略模式和工厂模式的常见组合？** 工厂用 Map 持有策略实例，按 type 取。
12. **设计模式选型的第一原则？** 业务简单就不要用模式，避免过度设计。
