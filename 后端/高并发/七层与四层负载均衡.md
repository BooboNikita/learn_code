# 七层与四层负载均衡

> 一句话总结：LVS（4 层）工作在内核态只负责包转发，性能极致；Nginx（7 层）在用户态解析协议做应用层处理；大流量场景用 LVS + Nginx 组合，DNS 轮询做最外层入口。

---

## 0. 背景与意义

> 经典面试题：「请说出在淘宝输入一个关键词到最终展示网页的整个流程」

这个流程涉及 HTTP、TCP、网关、LVS 等一系列知识点，**掌握后能极大地点亮技能树**。

架构演化的每一步都紧贴业务发展：

| 阶段 | 痛点 | 解决方案 |
|---|---|---|
| 单机 Tomcat | 性能瓶颈 + 单点故障 | 部署多台机器 + LB |
| 多机 + LB | 流量直接打 server 不安全 | 引入网关（鉴权/风控/限流） |
| 动静不分离 | 静态资源也打到 Tomcat | Nginx 处理静态资源 |
| Nginx 单点 | 主备机制 + keepalived | 主备切换 |
| Nginx 性能瓶颈 | 七层建立两次 TCP 消耗大 | 加 LVS 四层 |

> **小结**：
> - 架构没有银弹，每一步都对应业务痛点
> - "没有什么是加一层解决不了的"是经典共识
> - 中小流量 Nginx 足够，大流量用 LVS+Nginx

---

## 1. 核心概念：OSI 七层模型定位

| 负载均衡 | 工作层 | 代表 | 关键差异 |
|---|---|---|---|
| 四层 LB | 传输层（TCP/UDP） | LVS | 只转发包，不建立连接 |
| 七层 LB | 应用层（HTTP/HTTPS） | Nginx | 解析协议，可做应用层处理 |

> **小结**：
> - 四层只看到 TCP/UDP 包头，按 IP/端口转发
> - 七层能解析 HTTP URL、Header、Cookie 等
> - 四层性能高，七层功能强

---

## 2. 七层负载均衡（Nginx）

### 2.1 工作机制

1. 与 client 建立 TCP 连接
2. 接收 client 请求
3. 与 upstream server 建立 TCP 连接
4. 转发请求到 upstream
5. 接收 upstream 响应
6. 返回给 client

**性能瓶颈**：要建立 2 个 TCP 连接（上下行各一个），消耗 TCP Socket、收发缓冲区等内存。

### 2.2 Nginx 能力

- 反向代理 + 负载均衡
- 静态资源 proxy cache（内存+磁盘）
- 动静分离（静态资源不打 Tomcat）
- 简单网关能力

> **小结**：
> - 七层 LB 必须建立两次 TCP 连接
> - 抗负载能力受机器 I/O、CPU、内存限制
> - 百万级连接时性能急遽下降

---

## 3. 四层负载均衡（LVS）

### 3.1 核心思想

> "类似路由器那样只转发包但不需要建立连接的负载均衡器"

**优势**：
- 不需要建立连接
- 不需要维护 TCP 连接
- 抗负载能力、性能远超 Nginx
- 对内存/CPU 资源消耗低
- 性能可达 F5 硬件的 60%

### 3.2 工作机制

1. 接收 client 的 SYN 请求
2. 通过负载均衡算法选择最佳服务器
3. 修改目标 IP 为后端服务器 IP
4. 直接转发给该服务器
5. 三次握手由 client 和 server 直接建立
6. LB 设备只做类似路由器的转发动作
7. 回包时可能修改源地址以保证正确返回

> **小结**：
> - LVS 工作在内核态，避免用户态/内核态切换开销
> - 处理速度极快，适合极高并发
> - 缺点：不能做应用层处理（如 URL 路由、Header 改写）

## 4. 架构演化路径

### 4.1 单机 → 多机 + LB

```
Client → LB (Nginx) → Server 集群
```

### 4.2 多机 + LB + 网关

```
Client → LB → Gateway 集群（鉴权/风控/限流）→ Server 集群
```

### 4.3 动静分离

```
Client → LB → 静态资源 → Nginx proxy cache
                  ↓
                动态请求 → Gateway → Server
```

### 4.4 Nginx 主备（keepalived）

```
Client → 主 Nginx ↔ 备 Nginx（心跳包）
                ↓
            Gateway / Server
```

### 4.5 LVS + Nginx 组合（生产级）

```
Client → DNS 轮询 → LVS 多台 → Nginx 集群（主备）→ Gateway → Server
```

### 4.6 CDN 终极形态

```
Client → DNS → CDN（静态资源）→ LVS → Nginx → Gateway → Server
```

> **小结**：
> - 每一步演化都对应业务发展阶段
> - LVS 工作在内核态，性能远超用户态的 Nginx
> - CDN 解决跨地域静态资源分发

## 5. 架构选型决策

### 5.1 按流量规模

| 流量规模 | 推荐架构 |
|---|---|
| 中小型（QPS 千级以下） | 单层 Nginx |
| 中大型（万级 QPS） | LVS + Nginx |
| 超大型（数十 Gbps、上千万并发） | 自研四层 LB（如美团 MGW） |

### 5.2 LVS 的不可替代性

> "Nginx 1.9 之后支持四层负载均衡，貌似 LVS 不必要？"

**Nginx vs LVS 性能差异**：

| 维度 | LVS | Nginx |
|---|---|---|
| 工作位置 | 内核态 | 用户态 |
| 性能 | 极快 | 一般 |
| 稳定性 | 高 | 中 |
| 适用规模 | 千万级连接 | 百万级连接 |

**LVS + Nginx 组合的精髓**：
- LVS 做初级流量分发（极致性能）
- Nginx 做应用层处理（功能丰富）
- 功能和性能的最佳组合

### 5.3 巨量流量的困境

> "像美团这样的巨量流量（数十 Gbps、上千万并发），LVS 也不管用了"

实测虽然使用 LVS 仍出现丢包，美团因此开发了自研四层 LB（MGW）。

> **小结**：
> - Nginx 1.9+ 的四层能力不能完全替代 LVS
> - LVS 在内核态处理数据包，效率远超用户态
> - 巨量流量场景下仍需自研 LB

## 6. 动静分离架构

### 6.1 为什么需要动静分离？

| 资源类型 | Tomcat 处理 | Nginx 处理 |
|---|---|---|
| 静态资源（js/css/img） | 每次从磁盘加载，性能差 | proxy cache 缓存到本地 |
| 动态请求 | 优势 | 需透传到 Tomcat |

### 6.2 特殊场景：内部后台

> 运营中心后台是内部员工使用，鉴权与网关的 API 鉴权不同

- 部署两台运营中心 server
- Nginx 将运营中心请求直接打到这两台 server，**绕过网关**

> **小结**：
> - 动静分离让 Tomcat 专注动态请求
> - 内部系统可绕过网关层
> - 不是所有动态请求都必须经过网关

## 7. 四层 vs 七层 LB 对比表

| 维度 | 四层 LB（LVS） | 七层 LB（Nginx） |
|---|---|---|
| 工作层 | 传输层 | 应用层 |
| 工作位置 | 内核态 | 用户态 |
| 是否建立 TCP | 否（只转发包） | 是（上下游各一） |
| 抗负载能力 | 极强（千万级连接） | 中等（百万级） |
| 性能 | 极高 | 中等 |
| 资源消耗 | 低 | 中 |
| 协议解析 | 仅 IP/端口 | 完整 HTTP/HTTPS |
| 应用层能力 | 无 | URL 路由、Header 改写、缓存等 |
| 适用规模 | 巨量 | 中大型 |

---

## 8. 生产级架构分层

| 层级 | 组件 | 作用 |
|---|---|---|
| DNS | DNS 轮询 | 流量入口分散 |
| L1 | LVS 多台 | 四层流量分发（高性能） |
| L2 | Nginx 集群（主备） | 七层应用层处理 |
| L3 | Gateway 集群 | 鉴权、风控、限流、协议转换 |
| L4 | 业务 Server | 实际业务处理 |
| L5 | CDN | 静态资源分发 |

> **小结**：
> - 分层使每个模块各司其职，功能解耦
> - 方便扩展，TCP/IP 协议栈就是分层的典范
> - 每层只管自己的事，不 care 下层实现
