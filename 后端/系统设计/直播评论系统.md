# 直播评论系统

> 一句话总结：直播评论 = 客户端 SSE 长连接 + 服务端 Pub/Sub 广播 + L7 负载均衡（同一直播路由到同节点）+ DynamoDB 游标分页，支撑百万用户实时弹幕。

---

## 1. 需求与边界

### 1.1 功能需求

| 核心 | 非核心 |
|---|---|
| 发布评论 | 楼中楼回复 |
| 实时查看（端到端 < 200ms） | 点赞/表情回应 |
| 历史评论（无限滚动） | — |

### 1.2 非功能需求（百万级）

- 高扩展：水平扩展扛数百万观众 + 数千 TPS 评论
- 高可用：可接受最终一致性
- 低延迟：端到端 ≤ 200ms
- 安全性：用户认证授权
- 内容完整性：审核/反垃圾

> **小结**：
> - 边界先行：明确"做什么"与"做到什么程度"
> - 百万级是设计选型的分水岭
> - 评论可接受最终一致，但延迟要近实时

---

## 2. 底层设计

### 2.1 核心实体

- **User**：观众/主播
- **Live Video**：评论依附的直播内容
- **Comment**：用户在指定直播的消息文本

### 2.2 API 设计

```http
// 发布评论
POST /comments/:liveVideoId
Header: JWT | SessionToken
Body: { "message": "这个直播太精彩了!" }

// 获取历史评论（游标分页）
GET /comments/:liveVideoId?cursor={last_comment_id}&pageSize=10&sort=desc
```

> **注意**：`userId` 从请求头 JWT 解析，不通过 Body 传，防伪造。

> **小结**：
> - 实体先于接口，渐进式设计
> - 游标分页从接口设计之初就考虑
> - JWT/Session 防身份伪造

## 3. 高层架构：从 MVP 到分布式

### 3.1 MVP：三层架构

```
Client → Service → Database（NoSQL: DynamoDB/OTS）
```

发布评论：客户端调用 POST → 服务端校验 → 写入 DB。

### 3.2 实时广播：轮询 → SSE

**轮询方案**（不可行）：
- 100 万观众每 3s 轮询 = 33 万 QPS，99% 返回空 → 资源浪费

**SSE 方案**（推荐）：
- 单向持久 HTTP 连接，服务器主动 push
- 写入用标准 POST，读取用 SSE 流
- 优点：轻量、支持断线重连、调试简单
- 挑战：部分代理不支持流式、浏览器并发连接数限制

**WebSocket vs SSE 选择**：

| 维度 | WebSocket | SSE |
|---|---|---|
| 通信方向 | 双向 | 单向（服务器推） |
| 适用场景 | 读写均衡（如聊天） | 读多写少（如直播评论） |
| 连接开销 | 高 | 低 |
| 推荐度 | — | ✓ |

> **小结**：
> - 轮询在百万级不可行
> - 读多写少场景 SSE > WebSocket
> - 关注基础设施挑战：代理、浏览器连接数

## 4. SSE 水平扩展：Pub/Sub 演进

### 4.1 核心难题

百万观众分散到多台服务器：评论落到服务器 A，但观众连接在 B/C/D → 如何让 B/C/D 都能收到？

### 4.2 四种方案对比

| 方案 | 描述 | 问题 |
|---|---|---|
| 简单 Pub/Sub | 每台服务器处理全量评论 | 资源浪费 |
| 分区 Pub/Sub | `hash(liveVideoId) % N` 分频道 | 轮询导致订阅多个频道 |
| **分区 Pub/Sub + L7 负载均衡**（推荐） | 同一直播观众路由到同节点 | — |
| 调度器替代 Pub/Sub | 调度服务直接路由到目标服务器 | 映射维护复杂 |

### 4.3 L7 负载均衡实现

- 一致性哈希：`hash(liveVideoId) % N`，同一直播观众始终路由到同一服务器
- 动态查找：ZooKeeper 存储 `liveVideoId → server` 映射
- 效果：每台服务器只订阅少数频道，复杂度大幅降低

### 4.4 Pub/Sub 系统选型

| 系统 | 延迟 | 消息必达 | 适用 |
|---|---|---|---|
| Redis Pub/Sub | 极低 | 弱（发后即忘） | 允许丢消息 |
| Kafka | 稍高 | 强（持久化） | 必达场景 |

> **小结**：
> - 分区 Pub/Sub + L7 负载均衡是面试推荐答案
> - 一致性哈希让"同一直播观众聚集"
> - Redis Pub/Sub vs Kafka：必达性优先选 Kafka

## 5. 历史评论：游标分页

### 5.1 两种分页对比

| 维度 | 偏移量分页 | 游标分页 |
|---|---|---|
| 实现 | `offset=0&limit=10` | `cursor=last_id&pageSize=10` |
| 性能 | offset 越大越慢 | 稳定 |
| 一致性 | 高速写入下 offset 失效 | 稳定 |
| 推荐 | ❌ | ✓ |

### 5.2 DynamoDB 游标查询示例

```json
{
  "TableName": "comments",
  "KeyConditionExpression": "liveVideoId = :liveVideoId AND commentId < :cursor",
  "ExpressionAttributeValues": {
    ":liveVideoId": "liveVideoId_value",
    ":cursor": "last_comment_id_value"
  },
  "ScanIndexForward": false,
  "Limit": "pageSize_value"
}
```

> **小结**：
> - 高速写入场景必须用游标分页
> - 游标分页与 DynamoDB LastEvaluatedKey 完美契合
> - 偏移量分页会重复或遗漏

## 6. 方案对比表

| 实时方案 | 通信方向 | 连接开销 | 适用场景 | 推荐度 |
|---|---|---|---|---|
| 轮询 | 客户端拉 | 高 | 极小规模 | ❌ |
| WebSocket | 双向 | 高 | 读写均衡聊天 | 中 |
| **SSE** | **单向服务器推** | **低** | **读多写少评论** | **✓** |

| Pub/Sub 方案 | 资源效率 | 复杂度 | 适用 |
|---|---|---|---|
| 简单 Pub/Sub | 低 | 低 | 早期小规模 |
| 分区 Pub/Sub | 中 | 中 | 中等规模 |
| 分区 Pub/Sub + L7 | 高 | 中 | 大规模（推荐） |
| 调度器 | 高 | 高 | 极致场景 |

## 7. 复习清单

1. **直播评论核心非功能指标？** 端到端延迟 ≤ 200ms，水平扩展支持百万观众。
2. **为什么选 SSE 而非 WebSocket？** 读多写少场景，SSE 单向推更轻量；WebSocket 适合双向聊天。
3. **轮询为何不可行？** 100 万观众 3s 间隔 = 33 万 QPS，99% 返空，资源浪费严重。
4. **SSE 基础设施挑战？** 代理不支持流式缓冲、浏览器并发连接数限制、中间设备关闭长连接。
5. **核心实体 3 个？** User、Live Video、Comment。
6. **API 安全细节？** userId 从 JWT/Session 解析，不通过 Body 传，防伪造。
7. **历史评论分页方案？** 游标分页（cursor=last_comment_id）。
8. **偏移量 vs 游标分页？** 偏移量在高速写入下失效；游标稳定且与 LastEvaluatedKey 契合。
9. **SSE 水平扩展核心难题？** 评论落 A 服务器，观众连在 B/C/D，如何让所有观众都收到？
10. **分区 Pub/Sub + L7 负载均衡原理？** `hash(liveVideoId) % N`，同一直播观众路由到同节点。
11. **调度器方案优劣？** 主动路由效率高，但需维护动态映射+心跳，高可用复杂。
12. **Redis Pub/Sub vs Kafka？** Redis 延迟低但"发后即忘"；Kafka 持久化、消息必达。
