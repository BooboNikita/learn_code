# 分布式事务

> 一句话总结：**分布式事务 = 在分库分表 / 微服务拆分后，用"理论协议 + 业务补偿"机制替代数据库本地 ACID**。

---

## 0. 为什么要分布式事务

两种架构演进会带来事务失效：

| 场景 | 起因 | 失效的事务边界 |
|---|---|---|
| **分库分表** | 同一个业务 SQL 路由到多个 DB 实例 | 单库 ACID 失效 |
| **微服务拆分** | 一个业务操作跨多个服务 | 单服务本地事务失效 |

目标：**保证跨多个服务/数据库的数据一致性**。

> **小结**：
> - 分布式事务的根因 = 单体/单库拆分。
> - 面试第一问：先答清楚**是哪一类**（分库分表层 / 业务服务层），再给方案。

---

## 1. 核心理论：2PC / 3PC / XA

### 1.1 两阶段提交（2PC）

**角色**：协调者（Coordinator）+ 参与者（Participant）。

**两阶段**：

| 阶段 | 协调者动作 | 参与者动作 |
|---|---|---|
| **Prepare** | 发"准备"指令 | 执行本地事务（写 Redo/Undo 日志），**不提交**，回复 Yes/No |
| **Commit** | 收齐投票，全 Yes 发 Commit，否则 Rollback | 收到指令后真正 Commit / 回滚 |

**三大致命缺陷**：
1. **同步阻塞**：Prepare 后参与者必须锁定资源等指令 → 资源长期被占，性能差。
2. **协调者单点**：协调者宕机 = 所有参与者卡死。
3. **容错保守**：任何一个节点失败 = 全局回滚，**网络抖动会"误杀"成功事务**。

**两个异常场景的处理**：
- Commit 发出但参与者没收到 → **协调者无限重试**。
- 参与者收到 Commit 但宕机未执行 → 节点恢复后**查本地日志 + Redo 自动提交**。

### 1.2 三阶段提交（3PC）

在 2PC 基础上加一个 **CanCommit** 阶段（只问"能不能做"，不锁资源），缓解"准备阶段白忙活"。

**但**：网络往返更多、更复杂，工程上**几乎没人用**。

### 1.3 XA 规范

> **关键认知**：XA 不是协议，是 **2PC 的工程接口标准**（X/Open DTP 模型）。

- **TM**（事务管理器）= 协调者。
- **RM**（资源管理器）= 数据库（实现 XA 接口）。
- 只要数据库支持 XA，就能被 XA 事务管理器纳管。

> **小结**：
> - **2PC = 理论**；**3PC = 2PC 的改良（不实用）**；**XA = 2PC 的工程标准**。
> - 2PC 三大缺陷：同步阻塞、协调者单点、容错保守。

## 2. 强一致 vs 最终一致

### 2.1 XA 真的满足 ACID 吗？

**不严格满足**。原因是 **C（一致性）有窗口**：

> 协调者发 Commit → 参与者 A 提交成功 → 参与者 B 因网络抖动未收到 → 此刻 A 已提交、B 未提交 → **外部读取会看到不一致**。

**结论**：XA 是**最接近 ACID** 的方案，但**为强一致付出了性能代价（同步阻塞）**。

### 2.2 现代互联网的选择

> **放弃强一致，拥抱最终一致**。

最终一致性 = 允许中间存在不一致态，但**保证最终能一致**（重试 + 补偿机制兜底）。

> **小结**：
> - 现代架构几乎不用 XA（性能差、阻塞严重）。
> - 业界默认选最终一致：TCC / SAGA / AT。

## 3. 主流方案：TCC / SAGA / AT

### 3.1 TCC（Try-Confirm-Cancel）

**思想**：把 2PC 从"数据库资源层"上移到"业务逻辑层"。

| 阶段 | 作用 | 是否立即提交本地事务 |
|---|---|---|
| **Try** | 预留资源（检查 + 锁定） | ✅ 立即提交 |
| **Confirm** | 真正执行业务（使用 Try 预留的资源） | ✅ 立即提交 |
| **Cancel** | 释放 Try 预留的资源 | ✅ 立即提交 |

**下单锁库存示例**：

```
Try:    orders INSERT PENDING  +  inventory available_stock--, locked_stock++
Confirm:orders UPDATE CONFIRMED +  inventory locked_stock--
Cancel: orders UPDATE CANCELLED +  inventory available_stock++, locked_stock--
```

**容错难点**：Confirm 或 Cancel 失败怎么办？
1. **基础**：重试（Confirm/Cancel **必须幂等**） + 重试 N 次失败后**告警 + 人工介入**。
2. **进阶**：T+1 离线 Job 扫描脏数据自动修复（如扫"CONFIRMED 超过 5 分钟但库存仍 LOCKED"的订单）。
3. **高级**：读取时实时发现不一致 → 给用户返回"处理中" + 异步触发修复。

### 3.2 SAGA

**思想**：**立即执行，失败再补偿**。

- 一个 SAGA = 一系列**子事务**（每个立即提交）。
- 任一子事务失败 → **反序执行补偿事务**（也是新事务）。

**跨行转账示例**：

```
正向:  A 银行 -100  → 提交    B 银行 +100  → 失败
补偿:  A 银行 +100  → 提交    (反序补偿已成功的子事务)
```

**核心区别：补偿 ≠ 回滚**：
- **回滚**：数据库层 Undo Log，事务未提交。
- **补偿**：业务层**反向操作**，原事务已提交，补偿是**新事务**。

**SAGA 优势**：支持**并发子事务**（如旅行预订里机票/酒店/租车可同时下单），TCC 不行。

**容错**：和 TCC 一样靠**重试 + 兜底修复**。

### 3.3 AT（Automatic Transaction）

**思想**：**自动化的 SAGA**（Seata 默认模式）。

**原理**：
- 业务 SQL 执行**前** → 中间件解析 SQL，生成 before 快照。
- 业务 SQL 执行 + 写 **undo_log**（同一本地事务提交）。
- 全局回滚 → 中间件读 undo_log，**自动生成反向 SQL**。

**优缺点**：
- ✅ 业务**几乎零侵入**。
- ❌ 依赖数据库（需要 undo_log 表）+ 复杂 SQL 解析受限。

### 3.4 延迟事务（分库分表场景的轻量方案）

**背景**：纯分库分表场景，**第一条 SQL 之前，中间件不知道要连哪些分库**。

**两种解决思路**：
1. **资源消耗大**：Begin 时**在所有分库**都开事务 → 浪费连接。
2. **延迟事务** ✅：Begin 时**什么都不做**，只标记；第一条 SQL 进来时**才在对应分库开事务**；Commit 时**统一提交所有分库**。

**容错升级（亮点方案）**：
- Commit 单库失败 → 中间件**重试 3 次**。
- 仍失败 → **异步发 MQ 消息**（带全局事务 ID / 库 / 表 / 主键 / 失败阶段）。
- **自动修复服务**消费消息 → 业务自定义正向修复 or 逆向补偿。
- 收益：系统可用性 **P80 → P99**，几乎免人工。

> **小结**：
> - TCC 强业务侵入 / 高控制力；SAGA 长流程编排；AT 自动化零侵入；延迟事务 = 分库分表轻量优选。
> - 所有方案的**真正难点 = 容错**：重试 + 幂等 + 告警 + 自动修复 + 必要时人工。

## 4. 4 种方案对比

| 方案 | 适用场景 | 业务侵入 | 一致性强度 | 性能 |
|---|---|---|---|---|
| **XA（2PC）** | 单体跨库 | 无 | 强一致（最接近 ACID） | 差（同步阻塞） |
| **TCC** | 短流程核心交易（订单/支付） | **强**（手写三段） | 最终一致（控制力强） | 好 |
| **SAGA** | 长流程（旅行预订/转账） | 中（写补偿） | 最终一致 | 好（可并发） |
| **AT** | 通用微服务 | **无** | 最终一致 | 好 |
| **延迟事务** | 纯分库分表 | 无 | 最终一致 | **最优** |

---

## 5. 面试回答套路（3 段式）

1. **澄清问题**：先问清是分库分表层还是微服务业务层。
2. **给方案**：根据场景选 TCC / SAGA / AT / 延迟事务。
3. **亮难点**：补**容错机制**（重试 + 幂等 + 告警 + 自动/手动修复）—— 这才是高级工程师的体现。

> **小结**：
> - 答案好坏不在于"会背多少种方案"，在于**能否讲清容错体系**。
> - 面试官喜欢追问的真实问题：跨库事务怎么保证？分片 DELETE 怎么保证原子性？这些都指向"容错"。

## 6. 复习清单

1. **分布式事务根因？** 单体/单库拆分导致本地 ACID 失效。
2. **2PC 三大缺陷？** 同步阻塞、协调者单点、容错保守。
3. **3PC 为什么不流行？** 增加复杂度但收益小，工程性价比低。
4. **XA 和 2PC 的关系？** XA 是 2PC 的工程接口标准（TM + RM）。
5. **为什么放弃 XA 选最终一致？** XA 同步阻塞性能差，互联网架构追求高可用 + 高性能。
6. **TCC 三阶段各自的作用？** Try 预留 / Confirm 真做 / Cancel 释放。
7. **TCC 容错为什么必须幂等？** Confirm/Cancel 会无限重试，不幂等会重复扣款。
8. **SAGA 与 TCC 最大区别？** SAGA 立即提交、失败再补偿；TCC 先预留再确认。
9. **AT 模式的核心机制？** 解析 SQL + undo_log + 自动反向 SQL。
10. **延迟事务解决了什么痛点？** 分库分表 Begin 时不知道要连哪些库，延迟到首条 SQL 时按需开事务。
