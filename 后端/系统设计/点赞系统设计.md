# 点赞系统设计

> 一句话总结：点赞 = Redis 原子操作 + Lua 脚本 + Kafka 异步落库 + MySQL 兜底，本地反馈 + 分级缓存 + 限流熔断扛高并发。

---

## 1. 四大核心需求

| 需求 | 量化指标 | 容忍度 |
|---|---|---|
| 快 | 端到端 ≤ 200ms（点击到反馈） | > 500ms 用户感知卡顿 |
| 准 | 不丢不重 + 最终一致 | 重复计数会引发投诉 |
| 稳 | 突发 1 万 req/s 不崩 | 流量无规律 |
| 省 | 热/温/冷数据分层 | 合理利用资源 |

> **小结**：
> - 3 秒没变色 → 90% 用户划走
> - 重复点赞 → 70% 用户觉得"刷数据"
> - 看似小功能，藏着社交平台生存密码

---

## 2. 三大核心工具选型

### 2.1 Redis：缓存首选

- 数据结构丰富：Hash、Set、Sorted Set
- 原子操作 + Lua 脚本：解决并发状态判断
- 持久化（RDB+AOF）兜底
- 缺点：内存贵，需合理策略

### 2.2 消息队列：Kafka vs RabbitMQ

| 维度 | Kafka | RabbitMQ |
|---|---|---|
| 吞吐量 | 极高（几万 TPS） | 中等 |
| 适用 | 大规模（日活过亿） | 中小规模 |
| 路由策略 | 弱 | 丰富（含死信队列） |
| 持久化 | 强 | 中 |

**注意**：必须开启消息确认 + 定期对账。

### 2.3 MySQL：最后安全锁

```sql
CREATE TABLE `like_records` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT,
  `user_id` bigint(20) NOT NULL,
  `content_id` bigint(20) NOT NULL,
  `content_type` varchar(20) NOT NULL,
  `created_at` datetime DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  `is_canceled` tinyint(1) DEFAULT 0,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_user_content` (`user_id`,`content_id`,`content_type`),
  KEY `idx_content` (`content_id`,`content_type`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**核心索引**：
- `uk_user_content`：联合唯一索引防重复点赞
- `idx_content`：按内容查点赞记录

> **小结**：
> - Redis = 高频读写 + 原子操作
> - 消息队列 = 异步解耦 + 削峰
> - MySQL = 持久化兜底 + 业务支撑

## 3. 四层架构拆解

### 3.1 前端层：本地反馈 + 异步请求

- 按钮立即变色 + "点赞+1" 动画
- 同时异步发请求给后端
- 网络失败：写 `localStorage`，网络恢复后自动重试

### 3.2 服务层：业务中央处理

```
参数校验 → 状态判断（Redis）→ 操作执行（点赞+1/取消-1）
                                       ↓
                              发消息到 Kafka 异步落库
```

**幂等性**：requestId + Redis 分布式锁，防快速重复点击。

### 3.3 缓存层：两类核心数据

**第一类：用户点赞了谁**

| 数据结构 | 适用场景 | 示例 |
|---|---|---|
| Set | 仅判断是否点赞 | `SADD like:status:123 456` |
| Hash | 需要按时间排序 | `HSET like:status:123 456 1629260800000` |

**第二类：内容点赞数（String + INCR/DECR）**

```bash
# 点赞
INCR like:count:{contentId}
# 取消点赞
DECR like:count:{contentId}
```

**过期策略**：
- 热点内容（>10万点赞）：TTL 24h
- 冷内容：TTL 1h

### 3.4 持久层：异步管道

```
服务层更新 Redis → 发 Kafka 消息 → 立即返回前端
                                       ↓
                            后台消费者批量处理：
                            新点赞 → 插记录
                            取消点赞 → is_canceled = 1（逻辑删除）
```

**逻辑删除好处**：
- 反复横跳时直接更新 is_canceled 比删了重插高效
- 保留全量记录，便于分析用户习惯

> **小结**：
> - 前端层：本地反馈让用户"感觉很快"
> - 服务层：Redis 操作 + 消息队列解耦
> - 缓存层：Set/Hash/String 各司其职
> - 持久层：异步落库不阻塞主流程

## 4. 关键细节：两个经典坑

### 4.1 Lua 脚本：并发原子性

**问题场景**：A 和 B 同时取消点赞
1. A 查状态（已点赞）→ 准备 -1
2. B 查状态（已点赞）→ 准备 -1
3. A 改计数 -1
4. B 没察觉状态变，仍 -1 → 结果 -1 → 错误！

**Lua 脚本**（原子执行）：

```lua
local isLiked = redis.call('HEXISTS', 'like:status:123', '456')
if isLiked == 1 then
    redis.call('HDEL', 'like:status:123', '456')
    redis.call('DECR', 'like:count:123')
end
return 1
```

### 4.2 缓存过期策略

| 策略 | 做法 | 目的 |
|---|---|---|
| 主动刷新 | 缓存 miss 时查 MySQL 写回 | 实时性 |
| 定期更新 | 定时任务同步 Redis → MySQL，刷新 TTL | 持久化 |
| 缓存预热 | 新内容发布时初始化 Redis count=0 | 防首次 miss |
| 防穿透 | 布隆过滤器 + 空值缓存（5min） | 拦恶意请求 |

> **小结**：
> - Lua 脚本 = 原子性保障
> - 缓存过期策略：主动刷新 + 定期更新 + 预热
> - 布隆过滤器 + 空值缓存防穿透

## 5. 三大优化技巧

### 5.1 分级缓存：本地 + Redis

- 本地缓存（Caffeine / Go sync.Map）存超热点
- 用户请求：本地 → Redis → MySQL
- 本地缓存 TTL 1min（容忍 1 分钟近似值）
- **效果**：帮 Redis 挡 80% 请求
- **缺点**：多实例数据不一致（社交场景可接受）

### 5.2 MySQL 读写分离 + 分库分表

- 主库写、从库读
- 按 content_id 或 userId 哈希分 16 张表
- 单表 > 1000 万再考虑分表
- 分表后查询示例：

```sql
-- contentId=12345, 12345%16=9, 路由到 like_records_9
SELECT user_id, created_at FROM like_records_9
WHERE content_id = 12345 AND is_canceled = 0
ORDER BY created_at DESC LIMIT 20;
```

### 5.3 限流熔断

- **限流**：Redis + Lua 令牌桶，或 Sentinel/Hystrix
- **熔断**：Redis/MySQL 故障时返回"点赞成功，稍后显示"，缓存请求待恢复同步
- **核心思想**：先保系统，再保体验

> **小结**：
> - 分级缓存：本地 + Redis 减压
> - 读写分离 + 分表：单表 > 1000 万再考虑
> - 限流熔断：宁可部分失败，不可全系统崩溃

## 7. 复习清单

1. **点赞 4 大需求？** 快（≤ 200ms）、准（不丢不重）、稳（不崩）、省（分层存储）。
2. **为什么选 Redis？** 数据结构丰富（Hash/Set/ZSet）、原子操作 + Lua、持久化兜底。
3. **Kafka vs RabbitMQ 何时选 Kafka？** 日活过亿、点赞消息吞吐量极高（几万 TPS）。
4. **点赞记录表核心索引？** 联合唯一索引 `uk_user_content` 防重复，辅助索引 `idx_content` 按内容查。
5. **前端层两大关键动作？** 本地反馈（按钮立即变色）+ 异步请求（同时发后端）。
6. **网络失败如何处理？** 写 `localStorage`，网络恢复后自动重试，提示"已缓存"。
7. **幂等性如何保证？** requestId + Redis 分布式锁。
8. **为什么用 Lua 脚本？** 查状态 + 改计数打包为原子操作，避免并发脏数据。
9. **缓存过期策略？** 主动刷新（miss 查 DB）+ 定期更新（定时任务）+ 预热（初始化）+ 防穿透（布隆+空值）。
10. **逻辑删除的好处？** 反复横跳高效（更新 is_canceled）+ 保留全量记录便于分析。
11. **分级缓存效果？** 本地缓存帮 Redis 挡 80% 请求；容忍 1 分钟近似值。
12. **限流熔断核心思想？** 先保系统，再保体验；宁可部分失败，不可全系统崩溃。
