# 进程切换

> 一句话总结：**进程切换 = 切换 CPU 正在执行的指令流，本质和函数调用一样是"控制权转移"——函数调用改返回地址，进程切换改 PC + 切栈 + 切地址空间 + 保存恢复上下文**。

---

## 0. 一个反直觉的问题

> 进程切换和函数调用是非常类似的。

大多数同学第一反应是"这怎么可能"——函数调用在同一个进程内，进程切换是两个不同程序之间。

但从 CPU 视角看：

| 操作 | 共同点 | 关键差异 |
|---|---|---|
| **函数调用** | 把控制权从一段代码交给另一段代码 | 切换后还在**同一个进程**（同一栈、同一地址空间） |
| **进程切换** | 同上 | 同时**切栈 + 切地址空间** + 保存恢复全部寄存器 |

> **小结**：
> - 进程切换的"内核动作" = 切换指令寄存器 + 切换栈 + 切换地址空间。
> - 这一切**都靠"改返回地址"这个最朴素的操作**完成。

---

## 1. 函数调用的本质

看一个普通 C 函数调用：

```c
void B() { /* ... */ }
void A() {
    /* ... */
    B();
    /* 继续执行下一行 */
}
```

执行流程：

```
A 调用 B
  ↓
CPU 把"返回地址（= A 中 B 之后的指令地址）"压栈
  ↓
CPU 跳到 B 的入口执行
  ↓
B 执行完，ret 指令把栈顶返回地址弹出到 PC
  ↓
CPU 跳回 A 继续执行
```

关键点：**ret 指令弹出栈顶的"返回地址"放回 PC（指令寄存器）**。

> **小结**：
> - 函数调用 = **"压入返回地址 + ret 弹出返回地址"**。
> - 控制权转移 = **修改 PC（程序计数器）**。

---

## 2. 一段"魔鬼"代码

下面这段代码在 `funcB` 返回时**不是跳回 funcA，而是直接跳到 funcC**：

```c
#include <stdio.h>
#include <stdlib.h>

void funcC() {
    printf("jump to funcC !!!\n");
    exit(-1);
}

void funcB() {
    long *p = NULL;
    p = (long*)&p;       // p 指向栈上"自己"的位置
    *(p+2) = (long)funcC; // 修改返回地址为 funcC
}

void funcA() {
    funcB();
}

int main() {
    funcA();
    return 0;
}
```

### 2.1 编译运行结果

```bash
$ ./a.out
jump to funcC !!!
```

代码里没有任何 `funcC()` 的显式调用，但 `funcC` 还是被执行了！

### 2.2 原理

`funcB` 里 `p+2` 正好是栈帧中的**返回地址位置**。通过 `*(p+2) = (long)funcC`：

| 步骤 | 动作 |
|---|---|
| 1 | `funcB` 执行完 |
| 2 | `ret` 指令把栈顶（已被改成 funcC）弹出到 PC |
| 3 | CPU 跳到 `funcC` 执行 |
| 4 | 打印 `jump to funcC !!!` 后 `exit(-1)` 终止进程 |

> **小结**：
> - "改返回地址"是**进程切换的核心**——进程切换也无非是把 PC 改成另一个进程上次离开时的指令地址。
> - 这种"非预期改返回地址"也叫**缓冲区溢出攻击**（buffer overflow attack）。

---

## 3. 函数调用 vs 进程切换 vs 缓冲区溢出

| 维度 | 函数调用 | 进程切换 | 缓冲区溢出 |
|---|---|---|---|
| **触发方** | 程序员显式调用 | 操作系统调度器 | 攻击者构造恶意输入 |
| **修改内容** | 压入"合法"返回地址 | 切换 PC + 栈 + 地址空间 + 寄存器 | 修改栈上返回地址 |
| **栈** | 同一栈 | 切换到目标进程的栈 | 同一栈 |
| **地址空间** | 同一进程 | 切换 | 同一进程 |
| **合法性** | 合法 | 合法 | **非法**（黑客行为） |
| **本质** | 控制权转移 | 控制权转移（含上下文切换） | 控制权转移（被劫持） |

> **小结**：
> - **三者的本质完全相同**：都是修改 PC，把控制权交给另一段代码。
> - **合法 vs 非法** = 修改的返回地址是不是"预期的"。
> - 操作系统做进程切换是"黑客"也是"设计者"——它精确控制着每一次 PC 的改写。

---

## 4. 进程切换的"上下文"

进程切换远比函数调用复杂，因为要把**目标进程上次的运行环境完全恢复**。

进程 = "运行的程序"，CPU 能"看到"的状态叫**上下文**（context）：

| 上下文项 | 含义 | 切换时动作 |
|---|---|---|
| **PC（指令寄存器）** | 当前执行到哪条指令 | 保存旧进程的 PC，加载新进程的 PC |
| **栈指针（SP）** | 栈顶位置 | 切栈 |
| **通用寄存器** | 局部计算用的临时值 | 保存旧进程的寄存器组，加载新进程的 |
| **页表基址（CR3）** | 虚拟地址 → 物理地址的映射根 | 切换地址空间 |
| **内核栈** | 进程在内核态使用的栈 | 切到新进程的内核栈 |
| **进程状态** | 运行 / 就绪 / 阻塞 | 旧 → 就绪，新 → 运行 |

**切换流程（内核态）**：

```
[CPU 正在跑进程 A]
   ↓ 调度器决定切换到进程 B
[1. 保存 A 的上下文]  PC/SP/通用寄存器/页表 等 → 写入 A 的 PCB
[2. 加载 B 的上下文]  从 B 的 PCB 读出 PC/SP/通用寄存器/页表 → 装入 CPU
[3. 切换地址空间]    改 CR3 寄存器
[4. 跳到 B 的 PC]     B 开始执行
```

> **小结**：
> - 进程切换 = **保存旧进程全部状态 + 加载新进程全部状态**。
> - 这个"状态包"就是**进程控制块（PCB）**。
> - 函数调用只需保存返回地址和栈帧，进程切换要保存的是**整个 CPU 可见的状态**。

## 5. 关键汇编示例（funcB 反汇编）

```
000000000040059e <funcB>:
  40059e:  push   %rbp
  40059f:  mov    %rsp, %rbp
  4005a2:  movq   $0x0, -0x8(%rbp)        # p = NULL
  4005aa:  lea    -0x8(%rbp), %rax        # rax = &p
  4005ae:  mov    %rax, -0x8(%rbp)        # p = &p
  4005b2:  mov    -0x8(%rbp), %rax
  4005b6:  add    $0x10, %rax             # rax = p + 2（返回地址位置）
  4005ba:  mov    $0x400586, %edx         # edx = funcC 地址
  4005bf:  mov    %rdx, (%rax)            # *(p+2) = funcC
  4005c2:  nop
  4005c3:  pop    %rbp
  4005c4:  retq                            # 弹出已被改写的"返回地址" = funcC
```

注意 `retq` 之前的 `mov %rdx, (%rax)`：它把"返回地址"改成了 `funcC`，于是 `retq` 弹出的不是 `funcA` 的位置，而是 `funcC` 的入口。

> **小结**：
> - **CPU 视角下没有"高级语言"**：函数调用、进程切换、缓冲区溢出都是"PC 寄存器的改写 + ret 指令"。
> - 这也是为什么"函数调用 = 进程切换的子集"——差别只在**改了 PC 后还做了什么**（是否切栈/切地址空间/恢复寄存器）。

## 6. 复习清单

1. **进程切换的本质？** 修改 PC 寄存器（指令寄存器），把控制权交给另一个进程上次离开时的指令。
2. **函数调用和进程切换的共同点？** 都是"控制权转移"——通过修改 PC 实现。
3. **函数调用和进程切换的关键差异？** 进程切换要切栈、切地址空间（CR3）、恢复全部寄存器。
4. **缓冲区溢出的本质？** 攻击者修改栈上的返回地址，让 `ret` 指令跳到攻击者指定的代码。
5. **什么是进程上下文？** CPU 能"看到"的所有进程状态：PC、SP、通用寄存器、页表基址、内核栈指针等。
6. **PCB 是什么？** Process Control Block，进程控制块，存储进程全部上下文。
7. **进程切换的 4 步？** 保存旧上下文 → 加载新上下文 → 切换地址空间（CR3） → 跳到新进程 PC。
8. **为什么"黑客与操作系统设计者是一伙人"？** 两者都在做"修改 PC + 切上下文"的事，区别仅在"是否被允许"。
