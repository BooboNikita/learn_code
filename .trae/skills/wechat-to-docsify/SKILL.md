# WeChat 文章 → Docsify 复习笔记

把微信公众号（`mp.weixin.qq.com`）里的技术长文**精简成可背诵的复习笔记**，自动落到本仓库的 docsify 文档站里，并完成索引更新与 Git 推送。

---

## 1. 适用场景

用户消息里出现以下任一信号即可触发：
- 出现 `mp.weixin.qq.com/s/...` 链接
- 关键词：`归纳`、`总结`、`精简`、`简化`、`复习`、`笔记` 等

---

## 2. 仓库现状（执行前必读）

工作目录 `/mnt/d/code/learn_code` 已搭建 docsify 文档站，分三大类：
- **前端**：`前端/<子分类>/README.md`
- **后端**：`后端/<子分类>/README.md`
- **人工智能**：`人工智能/<子分类>/README.md`

关键入口文件：
- [_sidebar.md](file:///mnt/d/code/learn_code/_sidebar.md) — 侧边栏导航
- [README.md](file:///mnt/d/code/learn_code/README.md) — 站点首页
- 各分类下 `<分类>/README.md` — 该分类索引页

**Git 远程仓库**：`git@github.com:BooboNikita/learn_code.git`（SSH 已配）
**Git 身份**：当前已配置占位身份 (`[email protected]` / `Trae`)，推送前无需再设。

## 3. 执行流程（5 步必须走完）

### Step 1：抓取原文

```bash
WebFetch(url=<微信文章 URL>)
```

- 输出可能超过 20KB，**会落到 `/tmp/trae/toolcall-output/xxx.txt`**。
- 用 `Read` 完整读完，不要只读 preview。

### Step 2：确定落点

根据文章主题映射到具体路径：

| 文章主题 | 落点路径 | 命名建议 |
|---|---|---|
| 分库分表 / Sharding / 数据迁移 | `后端/分布式/` | `<主题>.md` |
| 分布式事务 / TCC / SAGA / 2PC | `后端/分布式/` | `分布式事务.md` |
| 高可用 / 集群 / 主从 / 选举 | `后端/分布式/` | `系统高可用设计.md` |
| 消息队列 / Kafka / RocketMQ | `后端/分布式/` | `消息队列<主题>.md` |
| 缓存 / Redis / 多级缓存 | `后端/Redis/` | `<主题>.md` |
| MySQL 索引 / 锁 / 性能 | `后端/MySQL/` | `<主题>.md` |
| Java 基础 / JVM / 并发 | `后端/Java/` | `<主题>.md` |
| 微服务 / Spring Cloud / Dubbo | `后端/` 新建 `微服务/` | `<主题>.md` |
| 其他 | **问用户** | — |

### Step 3：精简归纳

**目标**：原文通常 20~30KB，**压到 5~8KB**，保留可背诵的核心。

**保留**：
- 核心概念定义（一句话说清）
- 关键机制 / 协议 / 算法的对比
- 必要的小段代码示例（≤ 20 行）
- 关键数字、参数、阈值

**丢弃**：
- 客套话、自我宣传（"今天我们来聊聊"）
- 重复论述
- 配图说明（Markdown 渲染不了图）
- 文末推荐阅读链接
- 闲聊段落

**结构模板**（直接套用）：

```markdown
# <主题>

> 一句话总结：<核心结论>

---

## 0/1. <背景 / 定义>

> **小结**：
> - 要点 1
> - 要点 2

## 2. <核心机制 A>

| 维度 | A | B | C |
|---|---|---|---|
| ... | ... | ... | ... |

## 3. <核心机制 B>

...（关键代码示例）

## N. 方案对比表（必带）

| 方案 | 适用场景 | 业务侵入 | 一致性强度 | 性能 |
|---|---|---|---|---|
| ... | ... | ... | ... | ... |

## N+1. 复习清单（必带）

1. **问题 1？** 简短答案。
2. **问题 2？** 简短答案。
...（8~12 个）
```

**风格要求**：
- 全文中文
- 每个二级章节末尾带 `> **小结**：` 引用块（3~5 条要点）
- 关键对比必须用 Markdown 表格
- 关键代码用代码块 + 语言标签
- 复习清单放在文末（**8~12 题**）

### Step 4：写入 + 更新索引

1. **写主文章**：`<落点路径>/<文件名>.md`
2. **更新侧边栏** [_sidebar.md](file:///mnt/d/code/learn_code/_sidebar.md)：在对应分类下加 2 空格缩进的子项
   ```markdown
     - [分布式](后端/分布式/README.md)
       - [新文章名](后端/分布式/新文章名.md)   # ← 新增
   ```
3. **更新分类索引** `<分类>/<子分类>/README.md`：在 `## 已收录` 下加一行
   ```markdown
   - [新文章名](新文章名.md)
   ```

**注意**：每次 Edit 前必须 `Read` 对应文件（工具限制）。

### Step 5：提交推送

```bash
git add -A
git commit -m "docs(<子分类>): 新增《<文章名>》精简版文章"
git push origin main
```

推送完成后**给用户清晰回报**：
- 列出本次新增/修改的文件
- 给出 docsify 站点可访问路径
- 提示硬刷新（Ctrl+Shift+R）

---

## 4. 验证清单（推送前自检）

- [ ] 文章已用 `Read` 完整读完（不是只看 preview）
- [ ] 精简后体积 5~8KB
- [ ] 至少有 **1 个对比表**
- [ ] 至少有 **1 段代码示例**（如适用）
- [ ] 文末有 **8~12 题复习清单**
- [ ] 每个二级章节末尾有 `> **小结**：` 引用块
- [ ] 侧边栏已更新（_sidebar.md）
- [ ] 分类索引已更新（对应 README.md）
- [ ] commit 信息格式：`docs(<子分类>): 新增《xxx》精简版文章`

---

## 5. 异常处理

| 异常 | 处理 |
|---|---|
| `WebFetch` 输出被截断 | 用 `Read` 读 `/tmp/trae/toolcall-output/xxx.txt` 完整文件 |
| 文章主题无法归类 | **必须**用 `AskUserQuestion` 问用户放哪 |
| `_sidebar.md` / 分类 README.md 已被其他进程修改 | `Read` 后再 `Edit`，不要盲改 |
| 推送失败（non-fast-forward） | `git pull --rebase origin main` 后再 push |
| 用户没给 URL | **必须**用 `AskUserQuestion` 确认文章来源 |

---

## 6. 核心原则

- **少做事**：用户没要求的功能（封面、动画、SEO、CI）一律不主动加
- **可复习**：每篇结尾必须有复习清单，对比表、代码块、小结引用块缺一不可
- **零构建**：docsify 是纯静态站，所有 .md 文件直接 commit 即生效
- **中文优先**：所有内容用中文（除非用户明确要求英文）
