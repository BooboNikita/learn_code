# 数据迁移方案：平滑迁移三步法

> 一句话总结：不停机迁移=存量复制（mysqldump/XtraBackup）+ 增量同步（非侵入双写+Binlog校验）+ 验证切换（中间过渡+双写反转）；核心是"非侵入双写代理层"+"两阶段校验"。

---

## 0. 背景与意义

"数据迁移"是能精准衡量工程师技术深度与广度的问题，**简历上写了系统重构就必问**。

> "停机迁移"更像是一种技术资源或方案设计不足时的妥协。长时间服务中断在互联网业务中**不可接受**。

> **小结**：
> - "如何设计并实施一套完善的不停机数据迁移方案"是架构师必备能力
> - 核心要求：过程平滑、数据零丢失、最终一致
> - 停机迁移是 fallback，不是首选

---

## 1. 工具选型

| 工具 | 类型 | 优点 | 缺点 |
|---|---|---|---|
| **mysqldump** | 逻辑备份 | 简单易用，SQL 可读 | 数据量大时性能差 |
| **XtraBackup** | 物理备份 | 高性能，支持增量，备份过程对业务影响极小 | 复杂，恢复要求高 |

**选型原则**：
- 业务量不大 → mysqldump 足够
- 业务量大 → XtraBackup
- 考虑业务对性能抖动的容忍度和可用迁移窗口

> **小结**：
> - 工具选择是方案设计的第一步，体现思考问题的全面性
> - mysqldump 可通过参数优化（extended-insert）大幅提升性能
> - XtraBackup 是大数据量的首选

---

## 2. 关键参数：innodb_autoinc_lock_mode

| 模式 | 行为 | 并发性 | 主键连续性 |
|---|---|---|---|
| **0（传统）** | 表级自增锁，插入操作锁整张表 | 最差 | 绝对连续 |
| **1（连续，默认）** | 简单插入分配后释放锁，复杂插入退化为表锁 | 中 | 连续（除特殊场景） |
| **2（交错）** | 所有插入申请主键后立即释放锁 | 最好 | 批量插入内可能不连续 |

**关键警告**：在 `innodb_autoinc_lock_mode=2` 环境下进行双写，需重点测试 `INSERT ... SELECT` 或循环单条 INSERT 场景的主键乱序风险。

> **小结**：
> - 必须确认当前环境的 innodb_autoinc_lock_mode 配置
> - 双写方案中需显式指定主键值以避免冲突
> - 主键乱序可能导致数据"张冠李戴"

## 3. 环境确认清单

| 维度 | 确认内容 |
|---|---|
| Binlog 模式 | 是否为 ROW 模式（STATEMENT 在高并发下可能因执行顺序不确定导致主从不一致） |
| 数据库规范 | 所有表是否包含 update_time 字段？删除是软删除还是物理删除？ |
| ORM 框架 | 是否提供 AOP/Interceptor/Hook 等机制（用于双写实现） |

> **小结**：
> - ROW 模式 Binlog 是数据精确同步的基础
> - 软删除 + update_time 是时间戳轮询校验的前提
> - ORM 框架的 Hook/AOP 是非侵入双写的关键

---

## 4. 三阶段迁移框架

| 阶段 | 目标 | 关键点 |
|---|---|---|
| **第一阶段：存量复制** | 把源表历史数据导入新表 | mysqldump / XtraBackup |
| **第二阶段：增量同步** | 双写 + 校验修复 | 非侵入双写 + 增量校验 |
| **第三阶段：验证切换** | 从源表切到新表 | 中间过渡 + 双写反转 |

> 核心思想：平滑过渡 + 风险可控

---

## 5. 第一阶段：存量数据迁移（mysqldump 优化）

### 5.1 导出端优化

| 优化项 | 作用 |
|---|---|
| extended-insert | 多行合并为单条 INSERT，减少 SQL 体积和网络传输 |

### 5.2 导入端优化

| 优化项 | 作用 |
|---|---|
| SET SQL_LOG_BIN=0 | 临时关闭 binlog 记录 |
| innodb_flush_log_at_trx_commit=2 | 放宽 redo log 刷盘时机 |

> **小结**：
> - mysqldump 简单但慢，针对性优化能显著提升性能
> - 关闭 binlog 和放宽刷盘策略都是以"导入性能"换"安全冗余"
> - 适合数据量不是特别大的场景

---

## 6. 第二阶段：增量同步（核心环节）

### 6.1 非侵入式双写机制

> 最忌讳侵入式修改每一处业务逻辑代码——工作量大且极易引入新 BUG。

**优雅方案**：利用 ORM 框架的 AOP 能力（如 GORM 的 ConnPool 接口、MyBatis 的 Interceptor），在数据访问底层实现**双写代理层**。

**代理层功能**：
- 静默拦截所有数据库写操作（INSERT/UPDATE/DELETE）
- 复制一份分别在源表和新表上执行
- 动态可控（配置中心/API 实时调整工作策略）

### 6.2 动态可切换的模式

| 模式 | 行为 |
|---|---|
| source_first | 优先写源表 |
| target_first | 优先写目标表 |
| off | 关闭双写 |

### 6.3 装饰器模式实现（Go GORM 示例）

```go
// DoubleWritePool 封装源和目标两个数据源
type DoubleWritePool struct {
    source *sql.DB
    target *sql.DB
    mode   string  // "source_first" / "target_first"
}

func (p *DoubleWritePool) ExecContext(ctx context.Context, query string, args ...interface{}) (sql.Result, error) {
    switch p.mode {
    case "source_first":
        res, err := p.source.ExecContext(ctx, query, args...)
        if err != nil {
            return nil, err
        }
        // 源表成功后，尝试写入目标表（错误可降级）
        go p.target.ExecContext(ctx, query, args...)
        return res, nil
    case "target_first":
        // 切换到目标表优先的逻辑
    default:
        return p.source.ExecContext(ctx, query, args...)
    }
}
```

### 6.4 双写核心挑战

**挑战一：数据一致性问题**

> "写源表成功但写目标表失败，怎么办？"

**应对原则**：在双写阶段，**允许短暂不一致**，保证**最终一致性**。后续依赖独立的校验修复工具。

**已评估但放弃的方案**：基于消息队列的异步补偿。
- 原因：无法精确还原 `UPDATE ... WHERE` 类语句的实际影响范围

**挑战二：自增主键冲突问题**

> 线程 A 在源表拿 ID 100，线程 B 在源表拿 ID 101，但写入目标表时 B 抢先完成，导致 B 在目标表拿 ID 100，A 拿 101，**数据张冠李戴**。

**正确方案**：写入目标表时**显式指定主键值**，程序需获取源表生成的自增 ID 并用于目标表插入。

> **小结**：
> - 非侵入式双写是工程优雅性的体现
> - 动态可切换是风险管理的核心
> - 显式主键是双写一致性的关键

## 7. 增量校验与修复机制

### 7.1 基于时间戳轮询

```go
var last_time = get_initial_timestamp()
for {
    // 1. 查源表中 update_time > last_time 的行
    rows := findUpdatedRows(last_time)
    for _, row := range rows {
        // 2. 在目标表查找对应行
        tgtRow := findTgt(row.id)
        // 3. 不一致则用源表数据修复
        if !isEqual(row, tgtRow) {
            fixTargetTable(row)
        }
    }
    // 4. 更新 last_time
    if len(rows) > 0 {
        last_time = getMaxUpdateTime(rows)
    }
    sleep(1 * time.Second)
}
```

**依赖前提**：
- 所有表都有 `update_time` 字段
- 删除操作为**软删除**

**物理删除的解决方案**：增加**低频反向全量校验**，从目标表出发反查源表。

### 7.2 基于 Binlog 实时监听（首选）

> Binlog（ROW 模式）本身就包含变更后的数据。

**核心流程**：
1. 收到 Binlog 事件
2. 直接将 Binlog 中的数据内容与目标表实时数据**比较**
3. 不一致时，**回到源表查询最新数据**修复目标表

**关键细节**：
> 检测到数据不一致时，修复逻辑应**回到源表重新查询最新数据**，而非直接依赖 Binlog 内容。

**原因**：Binlog 中的事件可能滞后于真实状态。直接使用可能把较新的正确数据覆盖掉。

### 7.3 主从同步延迟：两阶段校验

**问题**：主从延迟可能导致拿到"过期"数据，做出错误判断。

**解决方案**：两阶段校验

| 阶段 | 操作 |
|---|---|
| 第一次校验 | 优先从从库读取比对 |
| 第二次校验（仅在不一致时） | 直接访问主库，拿到最新数据再比对 |
| 修复环节 | 始终以主库结果为最终依据 |

> 这样既能保证一致性，又避免对主库的频繁压力。

> **小结**：
> - 时间戳轮询：依赖 update_time + 软删除
> - Binlog 监听：首选方案，更实时
> - 修复时**回源查询**避免 Binlog 滞后问题
> - 两阶段校验平衡一致性与性能

## 8. 第三阶段：验证切换

### 8.1 风险

> 直接从"源表读写"切换到"新表读写"，风险巨大。一旦切换后新库出现问题，源库数据已不是最新，回滚将变得异常困难。

### 8.2 中间过渡阶段

**做法**：
- 调整双写策略改为"新表读写"
- 所有写请求，源表也写一份（"预发布"）
- 新表承担权威数据源角色
- 源表保持同步写入作为备用

**安全保障**：
- 新表出现异常 → 立即切换回源表
- 整个方案具备**可回退的稳妥性**

> **小结**：
> - 中间过渡 = 双写反转 + 源表备用
> - 灰度切换 + 可回滚是高可用迁移的标志
> - 真正稳妥的方案必须能"说撤就撤"

## 9. mysqldump vs XtraBackup 对比表

| 维度 | mysqldump | XtraBackup |
|---|---|---|
| 类型 | 逻辑备份 | 物理备份 |
| 备份产物 | SQL 脚本 | 数据文件 |
| 性能 | 慢（大库耗时长） | 极快 |
| 增量备份 | 不支持 | 支持 |
| 备份期间业务影响 | 中 | 极小 |
| 恢复速度 | 慢（执行 SQL） | 快（拷贝文件） |
| 适用规模 | 中小型 | 大型 |
| 复杂度 | 低 | 中 |
