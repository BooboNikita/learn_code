# 分页查询设计

> 一句话总结：分库分表下「LIMIT offset, size」单库语义失效，方案核心是**让查询稳定可控**：要么禁用跳页（推荐），要么用二次查询、ES / 索引中间表兜底。

---

## 1. 分库分表基础概念

### 1.1 三大分片策略

| 策略 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| 哈希取模 | `user_id % N` | 数据均匀、无热点 | 范围查询差 |
| 范围划分 | 按时间/ID 区间 | 范围查询高效 | 写热点集中在最新分片 |
| 路由中间表 | 主键→分片映射 | 灵活 | 多一次查询、维护复杂 |

> 实际生产**组合使用**：先按 user_id 哈希分库，库内再按时间范围分表。

### 1.2 中间件三种实现形态

| 形态 | 代表 | 优点 | 缺点 |
|---|---|---|---|
| SDK | ShardingSphere-JDBC | 性能最好 | 语言绑定、升级难 |
| Proxy | ShardingSphere-Proxy、MyCat | 透明、跨语言 | 多一跳网络、单点 |
| Sidecar | 与服务网格结合 | 低耦合 | 生态不成熟 |

> **小结**：
> - 哈希取模 = 均匀但难做范围
> - 范围 = 易做范围但易写热点
> - 中间件选型：**性能选 SDK**，**多语言 / 易维护选 Proxy**

---

## 2. 为什么分页变难：单库语义的失效

单库 `LIMIT offset, size` 隐含三个前提：

1. 数据**整体有序**（物理位置已知）
2. offset 是**全局偏移量**
3. 中间件可以**顺序扫描并跳过**

分库分表后：
- 数据分布在 N 个分片，**全局顺序无法直接得到**
- offset 跨分片分布不均
- 中间件必须**拉数据 → 排序 → 截取**

```sql
-- 原始 SQL（单库）
SELECT * FROM order_tab ORDER BY id LIMIT 4 OFFSET 2;

-- 在分片上的「宁可错捞」改写
SELECT * FROM order_tab ORDER BY id LIMIT 6 OFFSET 0;
-- (size + offset = 4 + 2 = 6)
```

中间件会从各分片各取 6 条，**归并排序**后跳过 6 条返回 4 条。

> **小结**：
> - 分页难的本质 = 全局有序 + 跨节点聚合
> - 全局查询法「能用」，但**深分页有灾难性开销**

---

## 3. 全局查询法的性能账

假设 10 个分片，每条记录 1KB，`LIMIT 10 OFFSET 10000`：

| 维度 | 数值 | 说明 |
|---|---|---|
| 单分片传输 | 10010 条 ≈ 9.77 MB | offset + size |
| 全网传输 | 10 × 9.77 MB ≈ **97.7 MB** | 各分片独立 |
| 有效数据 | 10 KB | 真正需要的 |
| **有效率** | **~0.01%** | 100MB 换 10KB |

**三大瓶颈**：
- 网络开销剧增
- 内存消耗巨大（深度分页 → OOM）
- CPU 负载过高（归并排序）

**优化点**：多路归并（Min-Heap 维护 N 个分片游标），流式读取降低内存。

> **小结**：
> - 深度分页下网络和内存是**双重灾难**
> - 优化只能缓解，**无法根治**
> - 根本出路 = **不要深分页** 或 **改设计**

## 4. 方案一：禁用跳页查询法（移动端首选）

**核心思想**：产品上**放弃跳页**，只允许「下一页」，传上一页最后一条的排序键。

```sql
-- 第一页（无游标）
SELECT * FROM order_tab ORDER BY id ASC LIMIT 50;
-- 客户端记录最后一条 id = 1050

-- 第二页
SELECT * FROM order_tab WHERE id > 1050 ORDER BY id ASC LIMIT 50;

-- 复合排序（避免排序谬误）
SELECT * FROM order_tab
WHERE (create_time < '2025-06-23 12:00:00')
   OR (create_time = '2025-06-23 12:00:00' AND id < 1050)
ORDER BY create_time DESC, id DESC
LIMIT 50;
```

| 优点 | 缺点 |
|---|---|
| 性能**稳定且极高**（与页深无关） | 无法跳到任意页 |
| 实现简单，天然适合无限滚动 | 对后台管理类需求不友好 |
| 完美契合移动端交互 | 需客户端配合 |

> **小结**：
> - **首选方案**，性能与页深解耦
> - 关键 = 复合排序时 OR 条件要写全
> - 牺牲 UX 换性能 = 移动端无限滚动的标准答案

---

## 5. 方案二：二次查询法（精确跳页）

适用场景：**后台管理系统**必须精确跳页。

**核心步骤**（以 3 分库、LIMIT 5 OFFSET 10 为例）：

| 步骤 | SQL | 目的 |
|---|---|---|
| 1. 首次改写 | `LIMIT 5 OFFSET 3`（3 = 10/3 向下取整） | 错捞足够数据 |
| 2. 求全局最小值 | 比较各分库首条 → `age_min = 6` | 锚定起点 |
| 3. 二次改写 | `WHERE age BETWEEN 6 AND max_i` | 放宽条件补数据 |
| 4. 计算 age_min 的全局 offset | 各分库中 age_min 之前的记录数之和 | 定位真实偏移 |
| 5. 调整偏移 + LIMIT | 跳过 `offset - min_offset`，取 size 条 | 截取目标 |

```sql
-- 第一次：各分库执行（改写后）
SELECT * FROM T ORDER BY age LIMIT 5 OFFSET 3;

-- 第二次：用 BETWEEN 拿到完整上下文
SELECT * FROM T WHERE age BETWEEN 6 AND <分库最大值>;
```

**致命限制**：

| 场景 | 是否适用 | 原因 |
|---|---|---|
| 数据均匀分散 | ✅ | 设计假设 |
| 某分页数据集中在一个分片 | ❌ | 第一次改写会丢数据 |
| 分段法（范围分表） | ❌ | 数据不在同一分片就找不到 |

> **小结**：
> - 用两次查询换「精确跳页 + 低数据传输」
> - **强假设**：数据必须**均匀**分布到各分片
> - 面试亮点：能演示清楚为什么 OFFSET 要除以 N + 找 min

## 6. 方案三：引入中间表（空间换时间）

**思路**：建一张**未分片**的索引表，存「排序键 + 主键 + 分片位置」。

```sql
-- 索引表（单表，分页性能高）
CREATE TABLE index_table (
  primary_key    BIGINT,
  update_time    DATETIME,
  target_db_shard VARCHAR(16)
);

-- 1. 在索引表上分页
SELECT primary_key, target_db_shard
FROM index_table
ORDER BY update_time DESC
LIMIT 10 OFFSET 100;

-- 2. 拿到主键 + 分片位置后，回分片查详情
SELECT * FROM order_tab
WHERE id IN (?, ?, ...);
```

**两大挑战**：

| 挑战 | 方案 |
|---|---|
| 一致性维护 | 同步双写（强一致）/ 异步 binlog（最终一致，Canal） |
| 异步失败 | 重试 + 死信队列（DLQ）+ 告警 + 人工补偿 |
| 查询能力限制 | 索引表必须包含所有筛选列，否则方案失效 |

> **小结**：
> - 空间换时间的经典代表
> - binlog 同步 + 死信 + 告警是标配
> - 适合「按某固定列排序 / 筛」的高频分页场景

## 7. 方案四：引入外部存储

| 方案 | 适用 | 原理 |
|---|---|---|
| Elasticsearch | 复杂排序 / 多条件筛选 / 全文 | 倒排索引 + 分布式聚合 |
| NewSQL（TiDB / OceanBase） | 关系型 + 全局有序 | 底层 KV 全局有序，对应用透明 |

```sql
-- ES 方案：分页用 search_after（与禁用跳页同思路）
POST /orders/_search
{
  "sort": [{"update_time": "desc"}, {"_id": "desc"}],
  "search_after": [<last_update_time>, <last_id>],
  "size": 10
}
```

> **小结**：
> - 关系型搞不定的复杂查询 → ES
> - 想保持 SQL 又要全局有序 → NewSQL
> - ES 方案同样推荐用 `search_after` 替代深分页

## 8. 方案总对比

| 方案 | 跳页支持 | 性能 | 复杂度 | 适用场景 |
|---|---|---|---|---|
| 全局查询 | ✅ | 浅分页 OK，深分页灾难 | 低 | 小数据量 / 浅分页 |
| 禁用跳页 | ❌ | **极高** | 极低 | 移动端无限滚动（**首选**） |
| 二次查询 | ✅ | 中 | 高 | 后台管理 + 数据均匀 |
| 索引中间表 | ✅ | 高 | 中 | 固定列排序 + 高频分页 |
| Elasticsearch | ✅ | 高 | 中 | 复杂排序 / 全文搜索 |
| NewSQL | ✅ | 中高 | 低 | 强 SQL 依赖 + 数据量大 |

> **小结**：
> - **移动端 → 禁用跳页**
> - **后台管理 → 二次查询 或 索引中间表**
> - **复杂查询 → ES**
> - 没有银弹，**按业务场景选型**

## 9. 复习清单

1. **分库分表下深分页为什么慢？** 各分片都要传 `offset + size` 条，中间件再做归并排序，开销随分片数线性放大。
2. **三种分片策略各有什么优缺点？** 哈希（均匀但范围差）、范围（范围好但写热点）、路由表（灵活但多一跳）。
3. **SDK 和 Proxy 模式的区别？** SDK 在业务进程内，性能好但绑语言；Proxy 独立部署，跨语言但多一跳。
4. **禁用跳页查询的 SQL 怎么写？** 客户端传上一页最后一条的排序键，SQL 改为 `WHERE sort_key > last_key ORDER BY ... LIMIT size`。
5. **复合排序的游标条件怎么写？** `WHERE (col1 < v1) OR (col1 = v1 AND col2 < v2)`，避免排序谬误。
6. **二次查询法的核心假设？** 数据必须**均匀**分布到各分片，否则首查会丢数据。
7. **二次查询为什么是「两次」？** 第一次错捞取最小值，第二次 BETWEEN 放宽拿上下文，定位真实全局 offset。
8. **索引中间表的两大挑战？** 一致性维护（同步双写 / binlog）和查询能力限制（索引表必须包含筛选列）。
9. **异步 binlog 同步失败怎么办？** 重试 + 死信队列 + 告警 + 人工补偿。
10. **ES 分页用什么替代深分页？** `search_after`（游标分页）。
11. **全局查询法的优化点？** 多路归并（Min-Heap）流式排序，降低内存。
12. **深分页优化口诀？** 浅分页用全局查询，移动端禁跳页，精确跳页用二次查询或索引中间表，复杂查询上 ES。
