# 系统高可用设计

> 一句话总结：**高可用 = 冗余 + 自动故障转移**，本质是"不让任何组件成为单点"。

---

## 1. 什么是高可用

**目的**：保障业务连续性，让用户感知不到"故障"。

**两个核心手段**：
1. **冗余**：单点是大敌，组件至少 2 台起步（2 台 90% × 90% → 99%）。
2. **自动故障转移**：靠"第三方仲裁者"（Keepalived / 注册中心 / 哨兵 / ZAB、Raft）近实时切换，**不要靠人工**。

**业界衡量标准 ——"几个九"**：

| 级别 | 可用性 | 年宕机 |
|---|---|---|
| 基本可用 | 99% | 87.6 小时 |
| 较高可用 | 99.9% | 8.76 小时 |
| **高可用**（大厂基线） | **99.99%** | **52.56 分钟** |
| 极高可用（金融/轨交） | 99.999% | 5.26 分钟 |

> **小结**：
> - 高可用的核心 = 冗余 + 自动故障转移。
> - 业界用"几个九"衡量，4 个 9 是大厂标配。
> - 自动转移比冗余更重要（人工切换不可靠）。

## 2. 互联网典型架构分层

```
接入层（LVS/F5）
  → 反向代理层（Nginx）
    → 网关（流控/风控/协议转换）
      → 站点层
        → 基础 Service
          → 存储层（DB）
            → 中间件（ZK/Redis/ES/MQ）
```

**做好高可用的前提**：每一层都要做高可用，**任一层单点 = 全局单点**。

> **小结**：
> - 互联网架构分 7 层，每层都要冗余。
> - "高可用"是分层治理的，不存在"全局开关"。

---

## 3. 接入层 & 反向代理层（LVS + Nginx + Keepalived）

**架构**：两台 LVS 一主一备，通过 **Keepalived** 互发心跳。

- Master 存活 → VIP 在 Master 上生效（对外 IP）。
- Master 挂掉 → Keepalived 检测到 → Backup 接管 → **VIP 漂移**到 Backup。

Keepalived 的检测机制：ICMP / TCP 端口扫描 / 自定义脚本检测 Nginx。

Nginx 的高可用做法：多机 Nginx + Keepalived，Keepalived 也可探测 Nginx 端口，剔除故障节点。

> **小结**：
> - **VIP 漂移**是接入层高可用的核心。
> - Keepalived 既是心跳探测工具，也是故障转移的"仲裁者"。

---

## 4. 微服务层（Dubbo + 注册中心）

**三要素**：Provider（服务提供方） / Consumer（消费方） / Registry（注册中心，如 ZK / Nacos）。

**流程**：
1. Provider 启动 → 向 Registry 注册服务。
2. Consumer 启动 → 向 Registry 订阅并拉取 Provider 列表。
3. Consumer 按负载均衡策略选一个 Provider 调用。
4. Provider 故障 → Registry 通过**心跳**发现 → 推送更新 → Consumer 剔除该节点。

**结论**：注册中心 = 微服务里的"Keepalived"，承担自动故障转移。

> **小结**：
> - 微服务高可用 = 注册中心 + 心跳 + 客户端负载均衡。
> - 故障节点的剔除由 Registry 推送完成，无需人工。

## 5. 中间件高可用

### 5.1 ZK（ZooKeeper）

**角色**：
- **Leader**（唯一）：处理所有事务请求，并向 Follower 广播；过半 Follower 写入成功才认为提交。
- **Follower**：处理读请求、参与投票、参与 Leader 选举。

**Leader 单点怎么办？**
- Follower 收不到 Leader 心跳 → 发起投票 → 用 **ZAB 协议**（崩溃恢复协议）选出新 Leader。
- 类似协议还有 **Paxos / Raft**——这些协议就是分布式系统的"仲裁者"。

> **小结**：
> - ZK 高可用 = 多 Follower + ZAB 协议选举。
> - 选主协议（ZAB/Raft/Paxos）是分布式高可用的通用解药。

### 5.2 Redis

**两种模式**：

| 模式 | 适用 | 做法 |
|---|---|---|
| **主从**（一主多从） | 数据量小、写压力低 | 写主、读从；引入**哨兵集群**（sentinel），哨兵用 Raft 选新主 |
| **Cluster 分片**（多主多从） | 数据量大、写压力高 | 数据按 **16384 槽**分片；每个分片独立做主从；故障时该分片内部选主 |

**主从模式痛点**：
- 单主写压力高。
- 单机存储有上限。
- 多从同步造成"同步风暴"。

**Cluster 解决**：多个 master 分担写 + 每个 master 各自做主从，**每个 master 仍要单独高可用**。

> **小结**：
> - 数据量小 → 主从 + 哨兵；数据量大 → Cluster 分片。
> - Cluster 内部每个分片都是独立的主从集群。

### 5.3 ES（Elasticsearch）

- 数据按**分片**（Shard）分布式存储，**主分片 P** + **副本分片 R**。
- 任意节点都可接收读写请求（负载均衡），但写请求会被路由到主分片所在节点，主分片再同步到副本。
- 集群通过 **Bully 算法**选出 Master 节点，Master 挂掉则其他节点重新选。

> **小结**：
> - ES 高可用 = 分片 + 副本 + Master 选举。
> - ES 的写不是直接打 Master，而是先路由到主分片所在节点。

### 5.4 Kafka（MQ）

- Topic 的 **Partition** 分布式存储在多台 Broker 上。
- 每个 Partition 有 1 个 Leader + N 个 Follower，**Follower 冷备**（正常不服务）。
- Leader 挂掉 → 从 Follower 中选新 Leader。

> **小结**：
> - Kafka 高可用 = Partition 多副本 + Leader 选举。
> - Follower 是冷备，与 ES 的热副本不同。

## 6. 存储层（MySQL）

**两种架构**：

| 架构 | 适用 | 高可用做法 |
|---|---|---|
| **主从** | 数据量小 | Keepalived + VIP 漂移（同 LVS），写主读从 |
| **分库分表（多主）** | 数据量大 | 每个分片独立做主从（解决高可用，不解决读写性能） |

**误区澄清**：分库分表后做主从，**不是为了读写性能**，**纯粹是为了高可用**——每个分片都可能有单点风险。

> **小结**：
> - MySQL 主从高可用 = Keepalived + VIP 漂移。
> - 分库分表 + 主从的本质目的 = 高可用，不是性能。

## 7. 总结：高可用的"套路"

观察所有组件，会发现**高度一致的设计模板**：

```
        冗余（多副本）
            ↓
      第三方仲裁者（选举协议）
            ↓
   故障自动转移（心跳 + 选主）
```

| 组件 | 仲裁者 | 选主协议 |
|---|---|---|
| LVS / Nginx | Keepalived | VRRP |
| MySQL | Keepalived | VRRP |
| Dubbo | 注册中心 | 服务列表推送 |
| ZK | ZAB 协议 | ZAB |
| Redis 哨兵 | 哨兵集群 | Raft |
| Redis Cluster | 集群内部 | Raft |
| ES | 集群内部 | Bully |
| Kafka | 集群内部 | Raft |

### 高可用 ≠ 银弹

架构层高可用只是**第一步**，生产环境还要应对：

- **瞬时流量**：秒杀场景 → 独立部署、隔离。
- **安全**：DDoS、爬虫、删库跑路 → 风控 + 权限分级。
- **代码 bug**：FullGC、内存泄漏 → 监控告警 + 单元测试。
- **发布**：要**优雅停机、平滑发布**。
- **第三方**：依赖降级、兜底文案。
- **不可抗力**：机房断电 → **异地多活**。

### 三个"额外手段"

1. **降级**（保证核心可用）：
   - 资金方挂掉 → 提示"系统升级中"避免客诉。
   - 直播卡顿 → 自动降码率。
   - 双 11 高峰 → 停非核心功能（注册/登录）保下单。
2. **事前防御**：单元测试、全链路压测、监控告警、双 11 封网。
3. **事后定位**：发布记录（时间/人/内容）支持一键回滚。

> **小结**：
> - 所有组件高可用 = 冗余 + 仲裁者 + 选主协议，是同一套模板。
> - 架构层高可用只是冰山一角，限流/熔断/降级/容灾/监控/发布管理缺一不可。

## 8. 复习清单

1. **高可用的两个核心？** 冗余 + 自动故障转移。
2. **为什么单点不行？** 90% × 90% = 99%，2 台机器故障率已经降到 1%。
3. **VIP 漂移是什么？** Keepalived 把对外 IP 从故障 Master 切到 Backup，无需改客户端配置。
4. **微服务高可用靠什么？** 注册中心 + 心跳 + 客户端负载均衡。
5. **Redis Cluster 怎么分片？** 16384 槽，key 算 hash 落槽，槽落哪个 master 由 master 负责。
6. **Kafka Follower 是热备还是冷备？** 冷备，正常不服务。
7. **MySQL 分库分表 + 主从的目的？** 解决高可用，不是性能。
8. **降级 vs 熔断 vs 限流的区别？** 降级 = 牺牲非核心保核心；熔断 = 快速失败防止雪崩；限流 = 控制入口流量。
