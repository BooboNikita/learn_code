# 分库分表主键生成策略

> 场景：数据量大、单库单表扛不住时采用分库分表。  
> 核心问题：**如何生成全局唯一、最好还递增的主键？**

---

## 1. 为什么自增主键失效

分库分表后多表各自 `AUTO_INCREMENT`，第一张表插 ID=1，第二张表插的也可能是 ID=1 → **主键冲突**。

```sql
-- 单库单表 OK，分库分表会冲突
CREATE TABLE order_info (
  id           BIGINT PRIMARY KEY AUTO_INCREMENT,
  buyer_id     BIGINT NOT NULL,
  order_amount DECIMAL(10,2)
);
```

主键方案需同时满足：**全局唯一**、**尽量递增**、**扛高并发**。

> **小结**：
> - 自增主键只在单库单表内有效。
> - 核心诉求：唯一 + 递增 + 高并发。

---

## 2. 常见方案对比

| 方案 | 唯一性 | 递增性 | 复杂度 | 适用场景 |
|---|---|---|---|---|
| UUID | ✅ | ❌ 随机 | 极低 | 对写入性能不敏感的小项目 |
| 数据库步长自增 | ✅ 单库内 | ⚠️ 单表内 | 低 | 拆分粒度固定的传统架构 |
| 雪花算法 | ✅ | ✅ 大致递增 | 中 | 绝大多数分布式系统 |
| 主键内嵌分库分表键 | ✅（DB 兜底） | ⚠️ 大致 | 中高 | 高频按 ID 查询的场景 |

---

## 3. UUID：能用但有坑

**两个核心缺点**：

1. **长度 36 字符** → 索引占用大。
2. **随机 → 频繁页分裂**（致命）：
   - InnoDB 的 B+ 树按主键有序存储；自增主键让数据总追加到尾部。
   - UUID 是随机的 → 频繁插到页中间 → **页分裂**，最坏会从叶子一直裂到根。
3. **无法利用顺序读**：相近 ID 在磁盘上相距远，范围查询退化为随机 IO。

> **小结**：
> - UUID 最大问题是**随机**带来的页分裂。
> - 自增主键 = 数据追加写，UUID = 数据随机写。

---

## 4. 数据库步长自增

**做法**：N 张表，每张表起始值不同、步长 = N。

举例（16 张表）：

- 表 0：1, 17, 33, 49, ...
- 表 1：2, 18, 34, 50, ...
- ...

**优点**：应用层零改造，DB 性能即方案性能。  
**缺点**：扩容要重新分配步长，迁移成本高。

> **小结**：
> - 用起始值 + 步长规避冲突，扩缩容麻烦。

---

## 5. 雪花算法（Snowflake）

**64 位分段**：

```
| 1 bit 符号 | 41 bit 时间戳 | 10 bit 机器ID | 12 bit 序列号 |
```

- 1 bit：符号位，固定 0。
- 41 bit：毫秒级时间戳，约 69 年。
- 10 bit：机器/实例 ID，1024 个节点。
- 12 bit：单节点单毫秒内自增，4096 个 ID。

唯一性来源：**时间戳不同 / 机器不同 / 序列号不同**。

### 5.1 分段可灵活调整
字段含义与长度可自定义：
- 时间戳 39 位也够十几年。
- 机器 ID 不必是物理机，可前 5 位机房 + 后 5 位进程。
- 序列号长度可按并发量调整。

### 5.2 序列号耗尽
12 位 → 单毫秒最多 4096 个。
- 升级：扩到 128 位。
- 兜底：阻塞到下一毫秒（**注意线程池耗尽风险**，必要时按业务切分多个 ID 生成器）。

### 5.3 数据倾斜
如果按 ID % 64 分表，**低频业务**可能每时刻只生成同一尾号 → 数据全落一张表。  
**解法**：序列号不从 0 开始，而是从**上一时刻的尾号**继续（必要时回退到 0）。比"每时刻随机起点"更可控。

> **小结**：
> - 64 位分段、时间戳 + 机器 + 序列号三要素保证唯一。
> - 字段长度按业务灵活切割。
> - 序列号耗尽 → 扩位 / 阻塞 / 拆业务。
> - 数据倾斜 → 序列号接上一时刻尾号继续。

## 6. 主键内嵌分库分表键（创新方案）

**设计**：将分库分表键直接编进主键里，举例：

```
| 时间戳 | 业务分片键（buyer_id 后 6 位）| 随机数 |
```

**核心收益**：单凭主键就能反推"这条数据在哪个库、哪张表" → **点查无需再走一次分片计算 / 索引**，可直达目标节点。

**代价**：
- **递增性**：大致递增，不严格。同毫秒两个用户，因随机段大小不同，ID 大小可能反序。但跨毫秒仍递增，写入性能介于"完全自增"与"完全随机"之间。
- **唯一性**：靠 **DB 主键唯一约束兜底**。同一用户、同一毫秒、随机数也撞上的概率极低（10 万随机数 → 十万分之一）。程序捕获 `Duplicate entry` 错误后**重新生成**即可，对用户透明。

> **小结**：
> - 主键 = 时间戳 + 分片键 + 随机数，主键即路由。
> - 大致递增：写性能优于 UUID，弱于纯自增。
> - 唯一性最终靠 DB 约束 + 重试兜底。

---

## 7. 发号性能优化（工程化思路）

算法之外，发号器本身可以再提速：

1. **段粒度批量取号**：一次取一段（如 1000 个），本地按需消费，DB 压力降低 2~3 个数量级。
2. **客户端预热**：本地余量低于阈值时异步续约，业务线程几乎零等待。
3. **singleflight 合并**：同一时刻多个取号请求合并为 1 个。
4. **线程级本地缓冲**：减少进程级锁竞争，形成"远端 → 进程 → 线程"三级缓存。

核心思想：**把"集中式同步"打散成"分层协作"**。

> **小结**：
> - 段粒度 + 预热 + 合并 + 线程局部，是发号器高性能的通用配方。
> - 适用于所有生成算法，是工程化补充。

---

## 8. 复习清单（5 问 5 答）

1. **UUID 最大问题？** 随机 → 频繁页分裂，破坏 B+ 树顺序写入。
2. **雪花 64 位如何分配？** 1 符号 + 41 时间戳 + 10 机器 + 12 序列号。
3. **主键内嵌分库分表键的核心思想？** 把分片键直接编进主键，主键即路由，单键可直达库表。
4. **序列号不够怎么办？** 扩位 / 升 128 位 / 阻塞下一毫秒 / 拆业务多生成器。
5. **唯一性最终怎么保证？** 随机数 + DB 主键唯一约束，冲突时程序捕获异常重试。

---

> 没有完美方案，只有**最匹配业务**的方案：UUID 简单但慢、步长自增传统实用、雪花算法经典但要会调参、主键内嵌分片键则是把"分库分表思维"推到了极致。
