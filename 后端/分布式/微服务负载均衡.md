# 微服务场景下的负载均衡

> 一句话总结：**负载均衡 = 把请求"按规则/按实时状态"分配到集群中"最合适"的节点**，静态算法靠统计学均衡、动态算法靠实时探测；面试要拿亮点，必须聊平滑加权、一致性哈希、动态权重调整、本地缓存一致性。

---

## 0. 为什么微服务离不开负载均衡

微服务架构下，**任何一次远程调用都要先面对"调哪台"的问题**：

| 部署形态 | 调用方式 | 负载均衡位置 |
|---|---|---|
| 单体应用 | 进程内方法调用 | 不需要 |
| 集群部署 | RPC / HTTP 跨机器调用 | **客户端** 或 **服务端 LB** |
| 微服务 | 多实例 + 动态扩缩容 | **客户端服务发现 + LB**（主流） |

> **小结**：
> - 微服务 ≠ 必须用复杂算法，业界大规模仍是**简单静态算法**打底。
> - 面试加分项 = 把"为什么这个算法不够好 / 如何改造"讲清楚。

---

## 1. 两大类算法总览

| 类别 | 是否感知真实负载 | 典型算法 | 适用场景 |
|---|---|---|---|
| **静态** | ❌ 不感知 | 轮询、加权轮询、平滑加权轮询、随机、加权随机、哈希、一致性哈希 | 绝大多数生产环境（性能好、零依赖） |
| **动态** | ✅ 实时探测 | 最少连接数、最少活跃数、最快响应时间 | 节点性能差异大、对延迟敏感 |

> **小结**：
> - 静态算法 = "统计学意义上的最优"，简单可控。
> - 动态算法 = "实时探测出的最优"，实现复杂、需采集指标。

## 2. 静态负载均衡算法

### 2.1 轮询（Round Robin）

**最简单**：A → B → C → A → B → C ……

- 优点：实现简单，绝对均衡。
- 缺点：忽视服务器性能差异，**法拉利和拖拉机同等待遇**。

**加权轮询（Weighted RR）**：按 weight 比例分配请求，能者多劳。

### 2.2 平滑加权轮询（Nginx 算法）

**问题**：普通加权轮询可能出现 `A, A, A, B, C` 这样的"流量突刺"，高权重节点瞬时压力大。

**平滑加权轮询**为每个节点维护两个权重：

```text
weight        = 固定初始权重
currentWeight = 动态权重
```

**每次选节点的 3 步**：

```text
1. 遍历所有节点：currentWeight += weight
2. 选 currentWeight 最大的节点作为本次目标
3. 选中节点的 currentWeight -= sum(weight)
```

效果：节点被选中后"当前权重"立刻大幅下降，**保证请求在时间维度上均匀分散**，避免流量集中爆发。

> **小结**：
> - 轮询类算法可控性强，序列确定。
> - Nginx 的平滑加权轮询 = 工业级生产首选。

### 2.3 随机 / 加权随机

- 随机：完全随机抽一个节点。
- 加权随机：按权重比例设置被抽中概率。

> **小结**：宏观效果与轮询相似，**轮询可控性更强**，实际可互替。

### 2.4 哈希 / 一致性哈希

**普通哈希**：`hash(key) % N` 把同类请求固定打到同一节点（适合本地缓存、Session）。

| 问题 | 说明 |
|---|---|
| 哈希函数差 | 分布不均 → 数据倾斜 |
| 集群扩缩容 | `N` 变化 → `mod N` 几乎全部失效 → **缓存雪崩** |

**一致性哈希（面试高频）**：

```text
- 哈希环：0 ~ 2^32-1 的闭环
- 节点映射：对节点 IP/主机名做 hash，落环
- 请求映射：对请求 key 做 hash，落环
- 节点查找：从请求位置顺时针找，第一个节点就是目标
```

> 钟表类比：现在 3:45，朋友说"下一个整点见" → 4 点。

**最大价值**：节点上下线只影响环上**逆时针相邻一小段请求**，不会全量映射失效，**扩缩容稳定性极佳**。

> **小结**：
> - 哈希算法 = 牺牲分布均匀性换"稳定路由"。
> - 一致性哈希 = 哈希算法的"扩缩容友好版"，是分布式存储必考点。

## 3. 动态负载均衡算法

### 3.1 最少连接数（Least Connections）

**假设**：连接数少 = 负载低。

**盲点**：在 HTTP/2、连接多路复用下，**1 个连接可承载 10 个请求**，连接数已不能真实反映负载。

### 3.2 最少活跃数（Least Active Requests）

**改进**：看"未返回响应的请求数"而非"连接数"。

**盲点**：分不清请求重量——一个大商家报表 vs 10 个普通查询，活跃数相同但 CPU 消耗天差地别。

### 3.3 最快响应时间（Fastest Response Time）

**思路**：响应时间 = 网络延迟 + 服务器负载 + 处理时长 的综合体现。

**实现要点**：

- 用**近期窗口**的响应时间（滑动窗口/衰减）。
- 越近的响应权重越高（**指标随时间衰减**思想）。
- 可选 P99 / P999 分位线，效果差异不大。

> **小结**：
> - 动态算法本质 = "选一个**单维指标**评估节点负载"。
> - 同一思路可推广：CPU 密集型应用 → 自己造一个"CPU 负载最低"算法。

### 3.4 共同挑战：信息孤岛

**问题**：客户端各自采集、各自决策，不知道其他客户端向各节点发了多少请求 → **基于不完整信息的判断可能错误**。

**解决方案（建立全局上帝视角）**：

| 方案 | 实现 | 代价 |
|---|---|---|
| **响应时携带** | 服务端通过 HTTP Header / RPC Metadata 返回 CPU、内存等负载 | 需框架支持 |
| **统一监控平台** | 所有节点向 Prometheus 上报，客户端查询后再决策 | 引入额外依赖和延迟 |

> **小结**：动态算法虽好，**生产环境仍以简单静态算法为主**——这套"信息孤岛"的深度思考本身就是面试亮点。

## 4. 面试三大实战亮点

### 4.1 亮点一：巧妙应对"大请求"难题

**场景**：用轮询的线上服务，RT 偶发毛刺，间隔不固定。

**根因**：少数大客户查询消耗大量 CPU/内存，轮询到该节点时拖慢其他普通用户。

**静态算法通病**：只关心"请求数量均衡"，忽略"请求重量均衡"。

**两层解决方案**：

| 层面 | 方案 | 性质 |
|---|---|---|
| **业务（治本）** | 与业务方协作，限制单次最多 100 条 | 产品层面优化 |
| **架构（治标）** | 用户画像识别大客户 → 专用节点 → **流量隔离** | 服务治理思想 |

> **小结**：隔离方案技术含量更高，可顺势引出"服务治理"。

### 4.2 亮点二：动态权重调整的"成加败减"

**原则**：

| 调用结果 | 权重动作 |
|---|---|
| **成功**（含业务错误码） | 适当**加**权 |
| **失败（网络错误/超时）** | 适当**降**权；网络类错误（如 EOF）降权幅度**更大**；超时降权**更小** |

**⚠️ 工程必踩的坑：必须设安全边界**

| 边界 | 原因 |
|---|---|
| **下限不能为 0** | 否则权重归零的节点"死"了，永远拿不到流量 |
| **上限不超过初始权重的 2~3 倍** | 防止表现好的节点被全部流量压垮 |

> **小结**：权重的"绝对值"不重要，**节点间的比例**才重要；上下限是工程红线。

### 4.3 亮点三：一致性哈希 × 本地缓存

**问题**：本地缓存 + 轮询 → 同一 Key 分散到不同节点 → 缓存命中率低 + 内存浪费 + 数据一致性问题。

**方案**：用 **一致性哈希 + 用户 ID 作 Key** → 同一用户请求稳定路由到同一节点。

**效果**：

- ✅ 本地缓存命中率大幅提升
- ✅ 整体内存消耗下降
- ⚠️ 数据一致性问题**只能缓解，不能根治**

**为什么不能根治？——关键词：应用发布**

```text
时序（一致性失效窗口）：
1. 读请求 A：hash → 老节点 N1，N1 缓存未命中，准备查 DB
2. 此时集群变更（应用发布 / 扩缩容），哈希环结构改变
3. 写请求 B：hash → 新节点 N2，更新 DB + 刷新 N2 缓存为"小刚"
4. 读请求 A：在 N1 查到旧数据"小明"，写入 N1 缓存
→ 数据不一致，但窗口期很短
```

> **小结**：方案能引出"缓存一致性"等更深入话题，**主动暴露局限性 = 严谨**。

## 5. 算法对比速查表

| 算法 | 是否感知负载 | 实现复杂度 | 适用场景 | 主要缺点 |
|---|---|---|---|---|
| 轮询 / 加权轮询 | ❌ | 极低 | 节点性能相近 | 不够平滑、忽略重量 |
| 平滑加权轮询 | ❌ | 中 | Nginx、生产首选 | 不感知实时负载 |
| 随机 / 加权随机 | ❌ | 极低 | 可替代轮询 | 序列不可控 |
| 哈希 | ❌ | 低 | 固定路由、本地缓存 | 扩缩容雪崩 |
| 一致性哈希 | ❌ | 中 | 分布式缓存、存储 | 实现复杂 |
| 最少连接数 | ✅ | 中 | 长连接服务 | 多路复用下失真 |
| 最少活跃数 | ✅ | 中 | RPC 调用 | 区分不了请求重量 |
| 最快响应时间 | ✅ | 中高 | 延迟敏感业务 | 需采集、计算衰减 |

## 6. 复习清单

1. **负载均衡要解决的核心问题是什么？** 请求到来时交给集群中哪台服务器处理。
2. **静态算法和动态算法的本质区别？** 是否实时感知后端真实负载。
3. **普通加权轮询的缺陷？** 短时间连续打高权重节点，导致流量突刺。
4. **平滑加权轮询的 3 步流程？** currentWeight += weight → 选最大 → 最大者 -= sum(weight)。
5. **普通哈希的最大问题？** 集群扩缩容时 mod N 结果剧变，引发缓存雪崩。
6. **一致性哈希的核心思想？** 节点上下线只影响环上逆时针相邻一小段请求。
7. **最少连接数的盲点？** HTTP/2 多路复用下连接数不能反映真实负载。
8. **动态算法的共同挑战？** 信息孤岛——客户端各自决策看不到全局。
9. **成加败减的"败"如何细化？** 网络错误降权大，超时降权小。
10. **动态权重的安全边界？** 下限 > 0（防死），上限 ≤ 初始 2~3 倍（防压垮）。
11. **一致性哈希 + 本地缓存能根治一致性问题吗？** 不能，应用发布时仍会出现窗口期不一致。
12. **大请求问题的两层解决方案？** 业务层治本（限条数）+ 架构层治标（流量隔离）。
